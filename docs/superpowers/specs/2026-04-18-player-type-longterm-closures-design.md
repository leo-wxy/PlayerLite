# 2026-04-18 播放器型长期版本第一阶段设计

## 背景

当前仓库已经具备在线播放、搜索、详情页、登录、歌词、在线音质切换、最近播放、自建歌单、网页歌单导入和后台播放等基础能力。

长期版本先不继续扩产品面，第一阶段只围绕三个核心闭环收敛实现：

1. 播放闭环
2. 在线资源闭环
3. 队列与歌单闭环

目标不是一次性补齐所有音乐平台能力，而是先把“点了能播、播了稳定、失败可恢复、队列行为一致”做稳。

## 范围

本阶段只处理以下问题：

- 断点续播
- 切歌稳定
- 缓冲失败与失败重试
- 弱网恢复
- 搜索到可播上下文的稳定映射
- 在线音质切换
- 版权失效、下架、URL 过期等在线资源失效兜底
- 下一首播放
- 最近播放
- 收藏 / 歌单与播放队列的边界
- 单曲来源切换

本阶段不处理以下方向：

- 重发现流、排行榜和运营专区扩张
- 社区、评论、动态、MV、播客、电台
- 多端同步和账号云资产建模
- 全量多音源聚合搜索
- 重型本地媒体库管理

## 产品定位

产品继续按“播放器型”推进，发现层保持轻量，只承担找歌和进入播放链路的职责。

当前阶段采用以下约束：

- 离线能力目标为完整离线库，但本阶段只先补齐与三个闭环直接相关的稳定性基础，不展开完整下载系统设计。
- 数据主语以设备为主，不要求账号承担跨设备恢复责任。
- 在线音源采用“主音源 + 备用源”策略，但兜底范围先限制在单曲级别。
- 本地、在线、下载内容采用中融合策略，共享音乐库、最近播放与播放队列，但保留来源差异。

## 闭环一：播放闭环

### 目标

确保当前歌曲能够稳定进入播放、继续播放、切换下一首，并在短时网络异常后优先恢复当前播放上下文。

### 责任边界

- `playback-orchestrator` 负责应用侧播放编排和状态协同。
- `playback-client` / `playback-service` 负责跨进程播放控制与后台宿主恢复。
- `player` / native runtime 负责底层 prepare、解码、渲染与失败回传。
- 页面层只消费状态和触发动作，不自行拼接恢复逻辑或重试逻辑。

### 约束

- 恢复时优先保留队列、当前项和播放位置，不因临时无权威快照而把 UI 或播放上下文打空。
- 切歌链路保持单向，不允许 app 侧、service 侧和 native 侧重复触发同一轮 prepare。
- 短时可恢复故障和不可恢复故障必须区分，前者可自动重试，后者给出明确失败态。
- 弱网恢复时优先继续当前项，不因为一次 URL 抖动或状态抖动误切空闲态或误切下一首。

## 闭环二：在线资源闭环

### 目标

确保用户从搜索或详情页点到一首在线歌曲后，能够稳定落到同一份可播上下文；遇到版权失效、下架、URL 过期或主音源失效时，有统一的资源兜底链路。

### 责任边界

- 搜索、详情页和推荐入口只产出统一歌曲引用对象，不各自维护独立播放语义。
- 在线资源解析、音质目录读取、URL 刷新和主备音源兜底统一收口到播放前准备层。
- UI 只展示“当前可播结果”和失败反馈，不在页面层写音源分支逻辑。

### 约束

- 搜索结果点播必须落到稳定的统一歌曲对象，再进入播放链路。
- 音质切换必须基于同一首歌的可用音质目录，切换后按当前位置重准备。
- 版权失效、下架、无可播链接和 URL 过期统一视为资源失效，由同一套兜底逻辑处理。
- 备用音源兜底范围先限制在当前单曲，不自动改写整个歌单、历史或收藏来源。

## 闭环三：队列与歌单闭环

### 目标

确保“当前在播什么、接下来播什么、这首歌从哪来”在所有入口上保持一致，避免页面各自演化临时规则。

### 责任边界

- `playlist-core` 负责播放队列、active index、插队、重排和持久化语义。
- 收藏、歌单、最近播放属于长期记录，不直接等同于当前播放队列。
- 搜索页、详情页、播放器页和首页只发出统一的队列动作，不自行维护一份页面私有队列规则。

### 约束

- “下一首播放”是独立插队语义，不等同于加入队列末尾。
- 最近播放以实际成功开播为记录时机，不以点击行为为准。
- 收藏 / 歌单和当前播放队列分离：前者是长期资产，后者是临时上下文。
- 单曲来源切换只影响当前歌曲的资源解析和后续恢复，不悄悄污染整条歌单或整个历史记录。

## 实施顺序

### 第一优先级：播放闭环

先稳定恢复、切歌、缓冲失败和弱网恢复。原因是另外两个闭环都依赖播放主链路稳定。

### 第二优先级：在线资源闭环

在播放主链稳定后，再收口在线资源失效、URL 过期和音质切换。这样可以减少把播放层问题误判成音源问题。

### 第三优先级：队列与歌单闭环

最后统一“下一首”“最近播放”“收藏 / 歌单”“单曲来源切换”的行为边界，避免前两轮实现后再返工共享语义。

## 第一阶段验收标准

### 播放闭环

- 冷启动后能够恢复上次播放项、播放位置和播放队列。
- 连续切歌不会出现卡死、串歌、回跳或重复 prepare。
- 缓冲失败具备自动重试和最终失败态。
- 短时弱网恢复后优先继续当前项。

### 在线资源闭环

- 搜索结果点播能够进入稳定的一致可播上下文。
- 音质切换后在当前进度附近继续播放。
- URL 过期时能够自动刷新并续播。
- 主音源失效时，当前单曲能够进入备用源兜底。

### 队列与歌单闭环

- 下一首播放、最近播放、收藏 / 歌单和单曲来源切换四类行为互不打架。
- 页面入口不同，但对同一首歌产生的队列语义一致。

## 非目标

以下事项即使后续会做，也不纳入本阶段验收：

- 下载中心、批量下载、断点续传和 Wi-Fi 下载策略的完整实现
- 跨设备同步
- 更重的首页推荐运营
- 社区能力
- 多源聚合搜索与大范围批量来源替换
