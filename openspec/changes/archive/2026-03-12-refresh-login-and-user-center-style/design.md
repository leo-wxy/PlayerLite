## Context

当前登录页与用户中心页已经具备可用的基础能力，但页面实现仍停留在“功能先到位”的阶段：`LoginActivity` 使用暖色渐变背景、顶部品牌图和表单卡片完成登录流程承载，`UserCenterScreen` 则以头像、摘要和按钮为主，满足一级 Tab 的基本展示。这些实现保证了登录、跳过、退出登录和会话同步都能正常工作，但在视觉语义上仍然偏向通用占位样式，与首页、搜索页逐步形成的“在线音乐产品”感受还没有完全拉齐。

本次变更的约束比较明确：

- 不能改变现有登录流程、`LoginViewModel` 状态流、跳过逻辑、退出登录能力和用户会话恢复行为。
- `user-account-session` 与 `user-center-tab-shell` 已经定义了登录页和用户中心的功能边界，因此本次设计主要调整 UI 投影与视觉 requirement，而不是新增业务能力。
- 仓库当前全局 `PlayerLiteTheme` 服务于整个应用，登录页和个人页如果直接改全局主题，容易把这次局部风格升级扩散成全局视觉回归。
- 用户要求“类似网易云”的样式方向，但实现上仍需保留 `PlayerLite` 自有品牌资产与页面结构，避免直接复制第三方图形或文案。

因此，这份设计的重点不是重做账户架构，而是定义一套可在登录页和个人页复用的局部视觉方案，并明确它如何在不动现有状态链路的前提下落地。

## Goals / Non-Goals

**Goals:**

- 为登录页和用户中心页建立统一、鲜明的音乐产品视觉语言，包括强调色、背景氛围、头部层级、卡片质感和核心按钮样式。
- 让登录页具备更强的“欢迎页/品牌入口”感，让用户中心页具备更强的“个人主页/账户中心”感，同时两者保持同源设计语义。
- 复用现有会话状态模型与交互回调，只替换页面结构和展示方式，避免引入新的登录态分支或额外状态源。
- 把与视觉风格直接相关的颜色、渐变、圆角、阴影和组件样式收敛为可复用的页面级 token 或共享 Composable，减少后续调样时的重复修改。
- 为后续个人页继续补充歌单、会员信息或更多在线能力预留稳定的头部区域和内容承载骨架。

**Non-Goals:**

- 不修改手机号/邮箱登录、退出登录、跳过登录、会话恢复和登录成功返回等现有业务流程。
- 不引入新的导航层级，不把用户中心拆成新的二级详情路由。
- 不要求本轮补齐收藏歌单、最近播放、会员权益等新的账户功能。
- 不直接改造全局 `MaterialTheme` 配色，也不要求首页、播放器页同步切换到相同视觉风格。
- 不复制第三方 App 的图标、插画、文案或专有资源，只借鉴“强调品牌、突出头部、重视情绪氛围”的产品风格方向。

## Decisions

### 1. 使用“页面级品牌视觉 token + 共享展示组件”，而不是直接改全局主题

登录页和用户中心页都需要明显更强的品牌气质，但这种风格升级目前只作用于账户域页面。如果直接修改 `PlayerLiteTheme` 的主色、surface 和 typography，会让播放器页、首页发现页等现有界面一起发生连带变化，增加回归风险。

本次设计选择在账户域页面内部定义一套共享的局部视觉 token，例如：

- 账户域强调色、浅色/深色渐变背景、弱对比信息色
- 头部卡片圆角、头像容器圆角、操作按钮样式
- 页面段落间距、头部留白、浮层阴影强度

这些 token 可以先以 `private` 常量、数据类或小型 `@Composable` 辅助函数的形式定义在账户相关文件中，再由登录页和用户中心页共同使用。这样既能保持两页风格一致，也能避免不必要地影响全局主题。

**备选方案：**

- 直接调整全局 `PlayerLiteTheme`：统一性最强，但波及面过大，容易让首页、播放器和搜索页一起发生非目标变更。
- 两个页面各自写一套内联颜色和样式：实现最快，但后续细调成本高，也不利于保证风格统一。

### 2. “类似网易云”按产品气质借鉴处理，不复制第三方品牌资产

用户想要的是接近网易云的视觉方向，本质上是更强的音乐产品感、品牌聚焦的头部、清晰的主 CTA、统一的红色系强调以及带情绪的卡片层级，而不是一比一复刻第三方界面。设计上应保留现有 `ic_playerlite_brand` 作为主视觉来源，并围绕 PlayerLite 现有品牌图形去重构页面。

具体取舍是：

- 保留自有品牌图形和应用命名，不引入第三方 logo、唱片插画或复制式文案。
- 借鉴红色系强调色、居中的主视觉焦点、信息分层和沉浸式头部背景氛围。
- 在用户中心页强化个人头部区域、信息层次和主要操作按钮，让页面“像音乐产品个人中心”，但保持当前业务信息和文案语义。

这样可以满足风格方向要求，同时降低视觉抄袭和后续品牌不一致的风险。

**备选方案：**

- 完全照搬目标产品的布局和插画风格：短期更像，但风险高，也会削弱 PlayerLite 自身辨识度。
- 完全不参考目标产品，仅做轻量美化：风险低，但很可能无法达到用户对风格跃迁的预期。

### 3. 保持现有状态模型与交互回调不变，只重构 UI 投影

当前登录页由 `LoginUiState` 驱动，用户中心页由 `UserSessionUiState` 驱动，业务链路已经覆盖未登录、登录中、已登录、退出登录等状态。此次变更不需要新增状态，只需要重新组织不同状态在页面上的视觉呈现。

实现上维持以下边界：

- `LoginViewModel` 继续负责登录方式切换、输入更新、提交登录、跳过登录和退出登录。
- `UserSessionUiState` 继续作为用户中心页唯一展示状态来源，不新增第二套“个人页专用状态”。
- UI 层只根据 `isLoggedIn`、`isBusy`、`title`、`summary`、`avatarUrl` 等现有字段切换头部文案、按钮和信息块展示。

这样做能把这次改动严格限定在表现层，也让后续 specs 更容易准确表达“视觉 requirement 变化，但行为 requirement 不变”。

**备选方案：**

- 为个人页新增额外 UI 状态模型：可表达更多视觉细节，但会让本次需求从 UI 升级演变成状态架构扩展。
- 在 ViewModel 中加入更多“样式专用字段”：会增加状态噪音，而这些内容更适合由 Composable 本地派生。

### 4. 登录页采用“品牌头部 + 浮层表单卡 + 轻量跳过入口”的欢迎页结构

当前登录页已经是欢迎页式布局，但主视觉、文字、表单和跳过入口之间的层级还不够鲜明。本次设计将其进一步收敛为更完整的音乐产品登录入口：

- 顶部和背景层提供更明确的品牌氛围，可通过大面积渐变、柔和光晕或装饰圆形塑造视觉焦点。
- 中上部保留品牌图形与标题副标题，但提升尺寸、留白和聚焦感，让主视觉更像“入口 hero”而不是普通图标。
- 中下部使用更明确的浮层卡片承载登录方式切换、输入框和提交按钮，让操作区与背景层拉开层次。
- “跳过”入口保持轻量，不抢主 CTA，但应与新风格匹配，例如使用更轻的 pill 样式或半透明容器。
- 小屏幕与输入法弹出场景下，需要允许内容可滚动或至少保证表单卡不会被遮挡得无法提交。

这个结构既保留现有流程，也能更自然地承接“登录后解锁在线播放”的产品叙事。

**备选方案：**

- 继续保留当前结构只微调颜色：改动小，但整体气质提升有限。
- 把登录页改成强营销落地页：视觉冲击更大，但容易过度设计，且和现有功能型登录页目标不完全匹配。

### 5. 用户中心页采用“头部资料区 + 状态感知操作区 + 后续能力占位区”的个人主页骨架

用户中心页当前更像一张信息卡片加按钮，满足功能但不具备“个人页”感。设计上将其升级为更明确的个人主页结构：

- 页面上半部作为头部资料区，使用更有层次的背景和头像容器承载用户身份信息。
- 已登录时突出头像、昵称和摘要；未登录时突出品牌化引导文案和“去登录”主按钮。
- 中部放置基础账户信息或当前阶段说明，使用统一卡片样式承载，便于后续拓展歌单、会员或统计模块。
- 退出登录按钮在已登录态中保留，但降级为次级操作，不和主身份信息争抢注意力。
- 页面整体与登录页共享强调色、留白节奏和背景语言，让用户在“登录入口”和“个人中心”之间感到同属一个账户域。

这样既能提升当前页面完成度，也为后续继续叠加个人页内容保留了稳定骨架。

**备选方案：**

- 只在当前卡片基础上加大头像和按钮：工作量小，但整体结构仍像占位页。
- 直接做成长列表式个人中心：更接近完整产品，但会把本轮需求扩展到新功能设计。

### 6. 共享基础组件优先放在账户域局部，而不是提前抽成全局设计系统

这次登录页和用户中心页很可能会共享背景层、头部展示卡、主按钮、次级信息卡等组件，但这些组件目前还高度依赖账户域内容。如果一开始就把它们抽成全局 UI kit，容易过早抽象，反而增加命名和复用成本。

因此实现策略是：

- 先在 `feature/user` 或 `feature/main` 相邻位置抽出账户域共享组件。
- 只抽离明确被两页共同使用的展示能力，例如品牌背景容器、强调按钮、头像头部卡片。
- 等后续首页、搜索页也需要复用同一风格语言时，再考虑是否提升到更通用的 theme extension 或 design-system 层。

这能平衡代码整洁和当前需求边界，避免为了“看起来可复用”而做过度工程化设计。

**备选方案：**

- 所有样式先都写在页面里：最直接，但两页同步调样会重复劳动。
- 立刻建立完整的全局设计系统：结构最整齐，但明显超出本轮范围。

## Risks / Trade-offs

- **[风格过于接近第三方产品]** → 保留 `PlayerLite` 自有品牌图形、文案和页面语义，只借鉴头部层级、强调色使用方式和音乐产品氛围，不复制第三方资产。
- **[局部风格与全局主题割裂]** → 通过共享局部 token 保证登录页和个人页自洽，同时让强调色与现有主题中的暖色系保持可衔接关系，避免完全跳脱。
- **[小屏或输入法场景下布局拥挤]** → 登录页采用可滚动或自适应布局，优先保证输入区域、提交按钮和跳过入口始终可达。
- **[已登录/未登录两态视觉差异过大导致维护成本上升]** → 复用同一页面骨架，只替换头部内容和主按钮语义，不分裂成两套完全不同的布局。
- **[局部组件抽象过度]** → 仅抽取明确复用的账户域组件，避免提前把局部页面样式上升为全局设计系统。

## Migration Plan

1. 在账户域页面中整理共享视觉 token，明确背景、强调色、卡片圆角、按钮和头部区域的统一样式。
2. 重构 `LoginActivity.kt` 中 `LoginScreen` 的页面结构，形成更清晰的品牌头部、登录卡片和跳过入口层级，并补齐小屏可达性处理。
3. 重构 `MainShellScreen.kt` 中 `UserCenterScreen` 的页面结构，形成头部资料区、状态感知操作区和基础信息卡片区。
4. 复用或抽离账户域共享展示组件，减少登录页和用户中心页之间的重复样式定义。
5. 通过现有 UI 状态与页面测试补充或更新验证，确认登录、跳过、已登录展示、退出登录展示仍与当前会话逻辑一致。

## Open Questions

- 首版“类似网易云”的程度是否只需要收敛到红色系品牌氛围和头部结构，还是希望进一步加入更强的装饰元素和动态感？
- 用户中心页是否需要在本轮就加入诸如“我的歌单/会员入口/最近播放”等静态占位模块，还是保持当前基础信息承载即可？
- 登录页是否需要在本轮同步细调应用品牌文案，例如标题、副标题的措辞，让它更贴近完整音乐产品口吻？
