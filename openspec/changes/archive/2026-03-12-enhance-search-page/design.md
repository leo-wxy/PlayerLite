## Context

当前首页已经通过 `HomeOverviewScreen` 提供悬浮搜索框，并由 `HomeViewModel` 结合 `HomeDiscoveryRepository` 完成默认关键词轮播；但点击行为仍停留在 `Toast` 提示，尚未承接真实搜索流程。与此同时，仓库约束要求 `MainActivity` 保持轻量、Compose UI 尽量无状态、网络访问通过 `AppContainer` + `JsonHttpClient` 统一提供，因此搜索能力不能简单继续堆在首页 ViewModel 里。

这次变更还引入了三个职责不同的接口：`/search/hot/detail` 负责首屏推荐热搜内容，`/search/suggest` 负责输入过程中的建议，`/cloudsearch` 负责提交后的结果检索。它们的触发时机、错误影响范围和 UI 展示都不同，如果没有清晰的状态边界，搜索页很容易退化为一个同时承载“输入态、历史态、建议态、结果态、异常态”的大而乱页面。

另一个上下文是：仓库已经使用 `LoginActivity` 这类独立 Activity 承载单一业务场景，首页搜索入口也明确属于 Home 域的一部分，但用户已经明确要求“新增搜索 Activity”。因此本次设计既要兑现独立 Activity 的需求，又要避免打破当前 Home 页与主壳层的职责划分。

## Goals / Non-Goals

**Goals:**
- 新增独立 `SearchActivity`，作为首页搜索入口进入后的唯一首版承载页面。
- 用户进入搜索页后以 `/search/hot/detail` 的更丰富热搜内容作为首屏默认推荐，并通过本地缓存避免频繁重复请求热搜接口。
- 用户输入关键词时请求 `/search/suggest` 展示联想建议；用户提交或点击建议后请求 `/cloudsearch` 展示结果。
- 搜索页提供本地持久化的历史搜索，并在空查询时以固定区域优先展示历史搜索，再向下展示可滚动的推荐热搜。
- 搜索页改成更紧凑的顶部搜索栏与单一滚动内容区，减少顶部留白并提升首屏信息密度。
- 搜索页视觉收敛到中度网易云风格：更柔和的页面底色、更轻的历史标签、更有榜单感的热搜区，以及更紧凑的音乐结果列表层次。
- 搜索页适度放大关键文本层级，保证输入框、历史搜索、热搜榜单、建议和结果列表在不同设备上的可读性。
- 首页概览态视觉与搜索页统一收敛到中度网易云风格，避免出现“搜索页像音乐 App、首页仍是通用卡片风”的割裂感。
- 将首页悬浮搜索框点击行为从临时提示升级为真实跳转，并保持首页现有默认关键词轮播逻辑不受影响。
- 为热搜、建议、搜索结果分别建模状态与错误处理，避免一个接口失败拖垮整个页面。

**Non-Goals:**
- 本次不实现服务端同步历史、结果筛选、多 tab 分类或分页加载。
- 本次不改造首页默认关键词的数据来源，首页仍沿用现有 `HomeDiscoveryRepository` 的关键词轮播策略。
- 本次不引入新的全局导航框架，也不把搜索页塞回 `MainActivity` 的 Home Tab 视图层级中。
- 本次不扩展在线播放、播放队列联动或搜索结果项的复杂二级详情，仅先承接搜索页首版闭环。

## Decisions

### 1. 使用独立 `SearchActivity` + `SearchViewModel` 承载搜索页，而不是继续塞进 `MainActivity`

搜索页将参照 `LoginActivity` 的模式，新增独立 `SearchActivity` 作为单场景 Activity。Activity 只负责 `setContent`、系统返回和 `Intent` 参数承载，实际页面状态与交互由新的 `SearchViewModel` 管理，搜索页 Compose 组件只消费状态并回调事件。

这样做的原因有三点：
- 用户已经明确要求新增 Activity，直接满足需求而不是绕回首页内部导航。
- `MainActivity` 当前已经承载 Home / Player / User Center 三个主域，继续塞入搜索页会让主壳层状态和返回逻辑继续膨胀。
- 仓库规范要求 `MainActivity` 保持轻量，独立 Activity 可以让搜索生命周期、返回栈和输入法体验与主壳层解耦。

备选方案：
- **方案 A：把搜索页做成 `MainActivity` 内部一个新的 Composable 路由。** 优点是少一个 Activity；缺点是当前仓库没有现成导航框架，主壳层状态会明显变重，不符合“MainActivity 保持轻量”。
- **方案 B：把搜索页做成 Dialog / BottomSheet。** 实现更快，但搜索建议、结果列表和软键盘交互都更难稳定，不适合作为正式搜索页。

因此首版选择独立 `SearchActivity`，并提供 `SearchActivity.createIntent(context)` 这类统一入口。

### 2. 新增独立搜索数据层，按接口职责拆成 `hot / suggest / result` 三条读取路径

搜索能力会新增独立的 `SearchRepository` 与 `SearchRemoteDataSource`，通过 `AppContainer` 注入，与首页发现仓库并列，而不是把 `/search/hot`、`/search/suggest`、`/cloudsearch` 混进 `HomeDiscoveryRepository`。Remote data source 只负责发起请求，Repository 负责把原始 JSON 转成稳定的 UI model 和页面状态输入。

建议的数据接口形态如下：
- `fetchHotKeywords()`: 对应 `/search/hot/detail`，返回更丰富的热搜词列表，并在存在登录态时自动携带当前 cookie / csrf 头信息。
- `fetchSuggestions(keyword: String)`: 对应 `/search/suggest`，返回建议列表，并在存在登录态时自动携带当前 cookie / csrf 头信息。
- `search(keyword: String)`: 对应 `/cloudsearch`，返回首版搜索结果列表，并在存在登录态时自动携带当前 cookie / csrf 头信息。

这样拆分的原因是三类接口的生命周期不同：
- 热搜在进入页时自动触发；
- 建议随输入变化而变化，且需要取消旧请求；
- 搜索结果由明确提交动作驱动，应该有独立的加载 / 失败 / 重试策略。

备选方案：
- **方案 A：继续扩展 `HomeDiscoveryRepository`。** 这会把首页发现和搜索页状态耦合到一起，后续首页和搜索的演进速度不同步时会越来越难维护。
- **方案 B：UI 直接拿 `JsonHttpClient` 发请求。** 实现短平快，但会让 Compose / ViewModel 直接依赖接口细节和 JSON 结构，不利于测试与后续扩展。

因此搜索页采用独立仓库，复用现有 `JsonHttpClient` 和 `AppContainer`，但不复用首页发现仓库；同时搜索 remote data source 默认开启 `requiresAuth`，让已登录用户的 cookie 能自动透传到搜索请求中。

### 2.1 `/search/hot/detail` 采用 repository 内缓存，优先复用可用热搜数据

推荐热搜列表是搜索页的默认首屏内容，但它的更新频率通常低于搜索建议和搜索结果，没有必要在每次进入搜索页时都强制重新请求。首版将把热搜缓存放在 `SearchRepository` 层：当存在仍可接受的缓存结果时，搜索页优先展示缓存热搜；只有在首次进入、缓存缺失，或缓存被显式判定失效时，才重新请求 `/search/hot/detail`。

把缓存放在 repository 而不是 ViewModel 有几个好处：
- 缓存策略与页面生命周期解耦，Activity 重建或再次进入时也能复用结果；
- ViewModel 只关心“当前能拿到什么热搜状态”，不负责管理缓存有效期和刷新判定；
- 后续如果要增加内存缓存失效时间、手动刷新或磁盘持久化，不需要改动页面层协议。

备选方案：
- **方案 A：每次进入搜索页都重新请求热搜。** 实现最简单，但会制造不必要的网络请求，也会让页面首屏更容易反复闪到加载态。
- **方案 B：把热搜缓存仅保存在 ViewModel 内。** 对单次页面打开足够，但用户退出再进入、Activity 重建或进程恢复后无法稳定复用。

因此首版选择 repository 内的轻量接口缓存，先以“减少频繁请求、优先秒开首屏”为目标，不把缓存设计扩展成复杂持久化系统。

### 2.2 搜索历史采用本地持久化，并在空查询态优先展示

用户多次进入搜索页时，最近搜索记录往往比纯热搜更贴近当前个人意图，因此搜索历史会作为一个独立数据源存在，并在空查询状态下优先于推荐热搜展示。首版将历史搜索落在本地 `SharedPreferences` 中，由搜索仓库负责读写和去重排序：每次用户提交搜索、点击建议词或点击热搜词时，都将该关键词写入历史；展示时按“最近使用在前、去重、限制数量”的规则输出。

把历史搜索放进 repository 而不是 ViewModel 的好处是：
- 历史可以跨 Activity 重建和应用重进保留，不依赖单次页面内存态；
- 持久化格式、数量上限、清空逻辑集中在一处，页面层不关心存储细节；
- 后续如果要升级到更复杂的本地数据库或服务端同步，也不需要改动页面层协议。

备选方案：
- **方案 A：只保存在 ViewModel 内存里。** 实现最轻，但退出页面再进后历史就消失，不符合用户对“搜索过的记录”的直觉。
- **方案 B：首版直接做服务端同步历史。** 长期能力更强，但这轮明显超出当前搜索页闭环范围。

因此首版使用 repository 内的本地持久化历史，并让空查询态优先展示历史。

### 3. 用“页面模式 + 分区子状态”建模搜索页，避免一个大布尔集合失控

搜索页状态会采用单一 `SearchUiState`，但内部显式区分当前页面模式：
- `Hot`: 查询为空或页面初次进入时展示“历史搜索 + 推荐热搜”；
- `Suggest`: 用户正在输入且有非空查询时展示建议；
- `Result`: 用户提交关键词或点击建议后展示搜索结果。

在这个主模式之下，再分别维护历史、热搜、建议、结果的子状态，例如：
- `historyKeywords`: 最近搜索关键词列表
- `hotState`: `Loading / Content / Error`
- `suggestState`: `Idle / Loading / Content / Error`
- `resultState`: `Idle / Loading / Content / Empty / Error`

这样设计的核心是让三类数据各自失败时影响范围可控：
- 热搜加载失败时，页面首屏展示完整错误态和重试；
- 建议接口失败时，不应该清空输入框或把整页打回错误态，只需要隐藏建议并允许继续提交搜索；
- 搜索结果失败时，只影响结果区域，并保留上一次提交关键词用于重试。

备选方案：
- **方案 A：只用一组 `isLoading / items / errorMessage`。** 代码表面更简单，但很快会分不清当前错误来自热搜、建议还是搜索结果。
- **方案 B：热搜、建议、结果分别使用三个独立 ViewModel。** 边界很纯，但对首版单页来说过重，输入联动也会更复杂。

因此首版使用一个 `SearchViewModel` + 一个 `SearchUiState`，但通过显式页面模式和子状态来保持职责清晰。

### 4. 搜索输入采用“输入改动触发建议、明确提交触发结果”的交互分层

为了避免 `/cloudsearch` 在输入过程中被频繁误触发，搜索页首版采用两级触发策略：
- 输入框文本变化后，仅在非空时请求 `/search/suggest`，并对连续输入做短延迟防抖与旧请求取消；
- 只有用户点击键盘搜索、点击建议项或明确点击搜索按钮时，才请求 `/cloudsearch`；
- 当输入框被清空时，页面回退到热搜模式，而不是保留过期建议或结果。

这个策略更符合常见搜索体验，也能显著减少搜索结果接口的无效流量。与此同时，搜索页首屏仍然遵守用户要求：进入页先调用 `/search/hot/detail` 展示更丰富的推荐热搜，而不是直接拿首页轮播词自动发起结果搜索。

备选方案：
- **方案 A：输入每变化一次就直接调 `/cloudsearch`。** 用户可更快看到结果，但接口开销更高，列表抖动更明显，也会弱化 `/search/suggest` 的意义。
- **方案 B：只做结果搜索，不做建议。** 首版更省事，但与用户明确给出的 `/search/suggest` 范围不匹配。

因此首版保留建议层，结果接口只由明确提交动作触发。

### 5. 采用“远端 JSON → 稳定 UI Model”的手工映射层，而不是让 UI 直接消费接口 JSON

搜索相关接口的响应结构很可能并不完全一致：`/search/hot/detail` 返回的热搜条目包含 `searchWord`、`score`、`iconType`、`iconUrl` 等附加信息，建议可能包含更轻量的联想条目，`/cloudsearch` 结果则可能包含歌曲、歌手、专辑等复杂字段。延续首页发现的做法，本次设计会在 Repository 内引入手工 mapper，把不同接口映射为稳定的 UI model，例如：
- `SearchHotKeywordUiModel`
- `SearchSuggestionUiModel`
- `SearchResultItemUiModel`

首版 `SearchResultItemUiModel` 只覆盖当前页面确实要展示的字段，例如标题、副标题、唯一 id、封面或辅助信息，而不是提前把所有服务端字段原样透出。

备选方案：
- **方案 A：UI 直接消费 `JsonObject`。** 开发快，但 UI 会直接依赖服务端字段命名，稍有变动就会连锁破坏页面与测试。
- **方案 B：为所有接口建立严格 `@Serializable` DTO。** 类型更完整，但当前接口字段仍有不确定性时，维护成本较高；首版更适合先用手工 mapper 只抽取必要字段。

因此继续沿用“手工 mapper + 稳定 UI model”的策略，配合 focused unit test 保证解析稳定。

### 5.1 搜索页采用紧凑头部 + 空查询态固定历史区 + 下方滚动内容区

当前搜索页顶部留白偏大，主要是状态栏安全区、默认输入框高度和分散内容块叠加造成的。用户进一步明确要求“历史搜索那一块也是固定的，不随着推荐滑动而移动”，因此本轮样式会调整为：
- 顶部采用更贴近状态栏的单行紧凑搜索栏，减少额外垂直留白；
- 空查询态在搜索栏下方放置固定高度的历史搜索区域，内部允许横向滚动，但该区域本身不随着下方内容滚动；
- 推荐热搜放在历史搜索区域下方的独立滚动内容区；
- 建议态与结果态隐藏固定历史搜索区，把主要空间让给滚动列表。

备选方案：
- **方案 A：只调小 padding，不调整内容组织。** 代码改动更少，但无法解决“历史区应该固定”和“接口内容少时页面显空”的问题。
- **方案 B：做复杂的沉浸式折叠搜索头。** 视觉更强，但这轮实现成本明显过高。

因此本轮选择“紧凑头部 + 空查询态固定历史区 + 下方滚动内容区”，在保持实现可控的前提下显著改善页面观感与交互连贯性。

### 5.2 搜索页视觉采用“中度网易云风”，但不做 1:1 拟真复刻

用户已经明确提出“整体都偏网易云一点”，但当前仓库整体主题仍是基于现有 Material 体系的播放器应用，因此搜索页更适合收敛到“中度网易云风”：保留现有布局结构与主题基底，只在视觉层次、卡片形态、榜单信息密度与标签质感上贴近音乐 App，而不做完全复刻。

这一决策拆成四个具体方向：
- 顶部搜索栏保留较高触控面积，但弱化返回按钮的存在感，搜索框改成更圆、更轻、更统一的胶囊输入样式；
- 固定历史搜索区继续保留，但历史词改成更轻的标签样式，减少厚重卡片感；
- 推荐热搜区从“通用卡片列表”收敛成更像榜单的列表：排名数字更突出，前几名有更强层级，副信息更克制；
- 搜索结果列表保留功能结构，但压缩空白、弱化卡片边界，让它更像音乐搜索结果而不是通用内容卡片。

备选方案：
- **方案 A：继续沿用通用 Material 风格，只做 spacing 微调。** 实现成本最低，但无法回应用户对“偏网易云一点”的明确诉求。
- **方案 B：高度拟真复刻网易云搜索页。** 风格最强，但容易与现有应用主题割裂，也会把这轮工作从样式收敛扩大成一次完整视觉重构。

因此本次选择中度网易云风：重点改视觉层次和组件质感，不改现有搜索行为、页面模式与数据结构。

### 6. 首页搜索入口只负责导航，不再由 `HomeViewModel` 直接处理页面外行为

当前 `HomeViewModel.onSearchClick()` 直接弹 Toast，这在占位期可以接受，但一旦改为真实跳转，继续让 ViewModel 直接承担 Activity 启动职责会让 UI 导航和业务状态混在一起。首版将把首页搜索点击改成由 `MainActivity` 或其所在 UI 层直接处理导航，`HomeOverviewScreen` 继续保持无状态，只接收 `onSearchClick` 回调。

这意味着首页搜索入口的职责调整为：
- 首页 ViewModel 继续维护关键词轮播与首页发现状态；
- 主壳层负责把搜索点击映射为 `startActivity(SearchActivity.createIntent(...))`；
- 搜索页自己的加载与输入状态完全由 `SearchActivity` / `SearchViewModel` 接管。

备选方案：
- **方案 A：保留 `HomeViewModel` 作为点击处理者，并让它发 Activity 启动事件。** 也能实现，但对一个简单导航动作来说会引入额外一次性事件管理。
- **方案 B：让 Compose 直接读取 `LocalContext` 并在 `HomeOverviewScreen` 内部启动 Activity。** 会让纯展示组件承担导航实现细节，不符合“UI mostly stateless”。

因此选择在主壳层注入导航回调，让首页展示层只负责发出点击事件。

### 7. 首页概览态整体视觉统一为中度网易云风，而不是只改单个搜索入口

在搜索页已经完成中度网易云风收敛之后，如果首页仍保持当前这套“暖色背景 + 通用白卡片”的视觉语言，用户从首页进入搜索页时会明显感知到风格断层。用户进一步明确要求“整个首页风格”都一起改，并确认底部播放入口卡也要纳入统一，因此这轮首页设计会以 `HomeOverviewScreen` 为中心，对首页顶部搜索入口、发现区模块标题、banner、横滑内容卡、快捷入口卡与底部播放入口卡做一轮统一的视觉收敛。

这次不会重写首页结构或数据来源，而是保留当前 `HomeOverviewScreen` 的信息架构：
- 顶部仍然是悬浮搜索入口；
- 中间仍然是发现区列表，包含 banner、横滑推荐和快捷入口等区块；
- 底部仍然是 Home 域内的播放入口卡。

在此基础上，视觉调整聚焦四个方向：
- **搜索入口**：从“浮空白卡片”收敛成更像网易云首页搜索条的轻量胶囊入口；
- **发现区**：banner、横滑推荐卡与快捷入口卡统一成更偏音乐 App 的卡片层次与文字密度；
- **模块标题**：让分区标题更像推荐模块标题，而不是通用列表标题；
- **底部播放入口卡**：保持 Home 壳层语义不变，但外观改成与首页和搜索页同一套设计语言。
- **快捷入口卡片细修**：去掉重复的“推荐”标签，并适度压低高度，让快捷入口更像轻量功能入口而不是层级过重的宣传卡。

备选方案：
- **方案 A：只改首页搜索入口。** 与搜索页连接最直接，但首页其余部分仍会显得像另一套视觉系统。
- **方案 B：整页中度收敛（当前选择）。** 改动集中在 `MainShellScreen.kt` 样式层，能明显提升整体一致性，又不会演变为首页重构。
- **方案 C：高度拟真复刻网易云首页。** 视觉最强，但会放大和现有主题体系的冲突，也超出当前 change 的可控范围。

因此本轮选择方案 B：保持首页结构与行为，统一视觉层次，重点是“更像音乐 App”，而不是“完全照搬网易云”。

### 7.1 搜索页统一放大字体层级，而不是继续做大小屏分流

用户进一步明确“搜索的字体整体调大，不用管大小屏”，因此这一轮不再维持大小屏分流策略，而是直接把搜索页中的关键文本统一做适度放大。这样做的重点不是追求复杂的响应式规则，而是优先解决当前搜索页在实际使用中的“小字感”，同时避免过度放大带来的压迫感。

这次统一放大的范围包括：
- 搜索输入框与占位文案；
- 历史搜索标签与分组标题；
- 推荐热搜榜单的排名、标题和副信息；
- 搜索建议列表与结果列表；
- 加载态、空态和错误态的标题与说明文案。

因此本轮实现会把搜索页字体整体收口为一套更大的默认层级，不再让字体大小依赖屏幕宽度判断。

## Risks / Trade-offs

- **[接口返回结构与预期不一致]** → 使用手工 mapper 只抽取页面必需字段，并为 `/search/hot`、`/search/suggest`、`/cloudsearch` 分别补充解析测试，避免 UI 直接依赖远端字段。
- **[本地历史过多导致首屏拥挤]** → 通过数量上限、去重和紧凑展示控制首屏密度，必要时提供清空入口。
- **[建议请求过于频繁导致闪烁或浪费流量]** → 在 `SearchViewModel` 中对输入变化做防抖，并取消过期请求；建议失败时不清空当前输入与热搜/结果承载。
- **[热搜缓存过旧导致内容不够新]** → 首版只要求“避免频繁请求”而不是永久缓存；缓存策略放在 repository，后续可补充失效时间或显式刷新而不破坏页面层协议。
- **[Activity 增多后入口与返回行为变复杂]** → 通过 `SearchActivity.createIntent(...)` 统一入口，保持搜索页是单向进入、系统返回关闭的简单模型，不引入额外 task 或 deep link 复杂度。
- **[搜索结果需求后续快速膨胀]** → 首版只稳定热搜、建议、结果三段核心链路，并通过独立 repository / UI model 预留后续增加历史、筛选、分页的扩展位。
- **[首页搜索逻辑与搜索页逻辑边界重新变糊]** → 明确首页只负责入口与默认关键词展示，搜索页独立负责数据加载与交互，避免再次把搜索实现回灌到 `HomeViewModel`。
- **[视觉过度贴近网易云导致与现有主题割裂]** → 保持当前主题色和排版系统，只调整组件形态、层级与信息密度，不做 1:1 拟真颜色和布局复刻。
- **[首页整页重塑导致搜索 change 目标漂移]** → 明确本轮只改 `HomeOverviewScreen` 的展示样式，不改首页数据协议、搜索入口行为、播放入口语义与 Home 壳层交互结构。

## Migration Plan

1. 在 `app` 模块新增搜索 feature 目录，补充 `SearchActivity`、`SearchViewModel`、搜索页 Compose UI 与状态模型。
2. 在 `AndroidManifest.xml` 中注册 `SearchActivity`，并在 `AppContainer` 中补充 `SearchRepository` 的构建入口。
3. 将首页悬浮搜索框点击回调改为由主壳层启动 `SearchActivity`，移除原有 Toast 占位行为。
4. 接入 `/search/hot/detail`、`/search/suggest`、`/cloudsearch` 的 remote data source、热搜缓存、历史持久化与 mapper，并为关键解析和 ViewModel 状态迁移补充测试。
5. 若首版实现出现严重问题，回滚路径是撤回 `SearchActivity` 注册与首页导航改动，首页可临时恢复为仅展示默认关键词的占位入口。

## Open Questions

- `/cloudsearch` 首版是否只承接单一结果类型（如歌曲），还是需要在第一页同时展示多种类型摘要？当前设计先按“单列表结果”收敛。
- 首页当前轮播关键词是否需要作为搜索页输入框 hint 透传；这不影响首版架构，可在实现阶段按交互细节决定。
