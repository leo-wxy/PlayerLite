## Context

当前播放器能力主要由 `MainActivity`/`PlayerViewModel` 驱动，播放生命周期与 UI 生命周期耦合，后台控制链路不完整。变更目标是将播放宿主下沉到独立后台服务，并通过系统标准媒体会话暴露状态与控制入口。项目当前 `app` 模块尚未接入 Media3，会新增会话/控制相关依赖，且保留现有 FFmpeg/JNI 播放内核不替换。

约束与前提：
- 最低系统版本 `minSdk 24`，需要兼容前台服务通知与媒体按键行为。
- 现有播放状态与队列逻辑已在应用层存在，需要与后台服务保持单一事实源。
- 本次实现明确采用 `MediaSessionService` 作为后台播放基础设施。

## Goals / Non-Goals

**Goals:**
- 以 `MediaSessionService` 作为唯一后台播放入口，承载播放生命周期。
- 建立 `MediaSession` 控制面，支持系统媒体中心、通知栏、耳机/蓝牙按键控制。
- 让 UI 通过会话控制器与服务通信，前后台切换时状态与控制保持一致。
- 定义服务重建与状态恢复路径，避免后台场景下播放状态丢失。

**Non-Goals:**
- 不重写或替换现有 FFmpeg/JNI 解码与底层播放器实现。
- 不在本次引入媒体库浏览（MediaLibrary）或复杂内容目录服务能力。
- 不扩展到跨设备投播（Cast）等非本次范围能力。

## Decisions

1) 选择 `androidx.media3.session.MediaSessionService` 作为服务基座
- 原因：与系统媒体控制协议天然对齐，能统一通知、媒体按键、会话生命周期管理。
- 备选方案：
  - 普通 `ForegroundService` + 平台 `MediaSession`：可行，但控制器生态与后续扩展性较弱。
  - `MediaBrowserServiceCompat`：偏旧方案，与当前目标（控制优先）不匹配。

2) 通过 `MediaSession` + `MediaController` 建立应用内控制通道
- 原因：避免自定义 Binder 命令协议，复用标准传输控制动作与回调语义。
- 备选方案：
  - 本地 Binder/AIDL：开发成本高，且系统级控制链路需额外桥接。
  - 仅保留 UI 直连播放器：无法满足后台与系统控制一致性要求。

3) 引入播放器适配层，将现有播放内核映射到会话可消费状态
- 原因：当前播放内核不是 Media3 `Player` 语义，需一个适配层将播放状态、进度、媒体信息、可用操作映射到 `MediaSession`。
- 备选方案：
  - 直接替换为 ExoPlayer：变更面过大，不符合本次“增强后台能力”目标。
  - 仅做部分映射：会导致系统端可见状态与实际状态不一致。

4) 前台通知由 `MediaSessionService` 生命周期驱动
- 原因：播放开始时进入前台服务，暂停/停止后按策略降级，满足后台稳定性与系统限制。
- 备选方案：
  - 始终常驻前台：资源占用与用户体验成本高。
  - 完全依赖 Activity 生命周期：与后台播放目标冲突。

5) 保持队列/当前项数据由应用既有队列模型管理，服务负责消费与同步
- 原因：复用既有播放列表模型，降低迁移风险；服务聚焦“执行与对外会话”。
- 备选方案：
  - 在服务内新建一套队列模型：会造成双写与状态漂移风险。

## Risks / Trade-offs

- [播放内核与会话状态映射不完整] → 先定义状态映射矩阵（Idle/Buffering/Ready/Ended、可用操作、错误码），并以单元测试覆盖关键转换。
- [服务被系统回收后恢复不一致] → 在服务启动路径增加最近播放上下文恢复与幂等初始化，恢复失败时降级到可重试空闲态。
- [UI 与后台状态竞争更新] → 采用单向状态流（服务为源头），UI 仅通过控制器发命令并订阅状态。
- [前台服务策略不当导致体验问题] → 明确进入/退出前台阈值与时机，补充后台切换回归用例。

## Migration Plan

1. 引入 Media3 session/controller 依赖，新增 `MediaSessionService` 骨架与 Manifest 声明。
2. 实现播放内核适配层，打通会话状态、元数据与传输控制动作。
3. 在 UI 层切换为 `MediaController` 调用路径，保留旧路径作为短期兜底（受开关控制）。
4. 完成通知、音频焦点、媒体按键与前后台切换测试后，默认启用新链路。
5. 回滚策略：关闭新链路开关并恢复旧控制路径；保留服务代码但不参与启动流程。

## Open Questions

- 服务重建后队列恢复是否必须精确到播放进度，还是允许恢复到当前曲目起点？
- 是否需要在首版同步锁屏封面大图加载策略，避免大图加载造成前台服务启动抖动？
