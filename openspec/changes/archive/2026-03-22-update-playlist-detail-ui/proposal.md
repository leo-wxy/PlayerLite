## Why

当前歌单详情页已经打通“歌单头部 + 动态信息 + 分页歌曲列表 + 播放入口”的主链路，但页面视觉层级与歌手/专辑详情页存在割裂，且歌曲条目承载偏卡片流，整体不够紧凑，容易偏离既有规格中“更偏云音乐风格/音乐列表密度”的体验预期。为了让歌单详情页与现有 detail chrome 体系一致、并使 UI 表现更贴合规格要求，需要对歌单详情页 UI 做一次聚焦的视觉与交互刷新。

## What Changes

- 调整歌单详情页头部布局与信息层级：封面、标题、作者、曲目数与播放入口更聚合，歌单简介从头部移出并通过独立“简介”tab 展示，同时与现有 detail 页的氛围背景/取色策略保持一致。
- 更新动态信息区域的视觉承载（评论/收藏/播放等），更偏“统计信息条/胶囊”而非独立大卡片，同时保持失败/空态不影响主体内容。
- 将歌曲列表条目从“卡片流”收敛为更接近音乐 App 的紧凑行样式（密度、对齐、信息层级），并保持分页加载、加载更多 footer 与错误重试体验一致。
- 保持现有“播放全部/点歌播放”的语义不变：仍以歌单上下文替换播放列表并打开播放器页；播放成功后才上报播放量并刷新动态信息。
- 明确本次变更仅聚焦 UI/交互层刷新，不新增或调整歌单相关网络接口，不引入收藏/取消收藏等写操作能力。

## Capabilities

### New Capabilities
<!-- None. -->

### Modified Capabilities
- `playlist-detail-page`: 更新歌单详情页在头部信息、动态信息与歌曲列表区域的 UI 承载要求，使其与现有 detail 页面视觉语言对齐，并进一步明确歌曲列表条目应采用更紧凑的行样式（而非卡片流），同时保持加载、失败与分页追加的分区状态承载语义不变。

## Impact

- 影响代码范围：`PlaylistDetailActivity`（Compose UI 结构与组件）、与之相关的 `feature/detail` 共享组件（如需要抽取/复用更紧凑的条目样式）、以及对应的 UI 测试 tag（若结构调整导致 tag 需要迁移）。
- 影响规格文档：需要为 `playlist-detail-page` 提供 delta spec，补充或收敛“云音乐风格/紧凑行样式/头部与动态区层级”的要求表述，使后续实现与验收标准一致。
- 影响网络：不新增接口；继续使用现有 `/playlist/detail`、`/playlist/track/all`、`/playlist/detail/dynamic`、`/playlist/update/playcount` 等请求链路。
- 影响验证：需要覆盖歌单详情页的加载态、错误态、动态信息失败降级、分页追加与播放上报后的动态刷新，确保 UI 刷新不引入交互回归。
