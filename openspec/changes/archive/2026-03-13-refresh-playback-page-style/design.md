## Context

当前 `PlayerScreen` 更像一块“播放信息仪表板”：顶部卡片同时承载文件名、状态文案、唱盘百分比、播放状态 badge，外围还叠着选音频、登录、UI test、清缓存等浮动按钮；中间又有一整块 `Audio Info` 统计卡，底部另有进度区和播控区。对现在已经偏向云音乐风格的首页与搜索页来说，这个播放页显得过重、过杂，也把用户注意力从歌曲本身拉到了大量次级信息上。

与此同时，播放页并不是完全独立的全局页面，而是包在 `PlayerExpandedScreen` 这个 Home 域展开壳层里。外层已经提供了返回首页的按钮，因此本次设计不需要再在 `PlayerScreen` 内重复造一层返回导航，重点应该放在内容区本体：哪些内容是“必须的播放 UI”，哪些内容应当从主视觉里移除或下沉。

当前还存在两个约束：
- 播放业务真相仍由现有播放状态、播放列表、播放模式和后台播放链路负责，本次不能因为视觉重做而改坏这些行为。
- `playlist-management` 与 `playback-speed-control` 已经约束了主播控区的基本结构：上一首 / 播放暂停 / 下一首是主操作，播放模式和播放列表入口仍需要在主控区保持可达。
- 当前播放链路并不是所有内容都具备真实在线 `songId`：本地音频通常只有本地生成的队列 id 与文件名，因此任何“歌曲百科”能力都必须先判断当前播放项是否具备真实在线歌曲标识，不能对所有播放内容一视同仁。

## Goals / Non-Goals

**Goals:**
- 将播放页收敛为更接近云音乐风格的简洁播放页，弱化工具面板感。
- 只保留必要播放信息：歌曲主信息、封面 / 主视觉、进度、基础播控，以及必要的模式 / 列表入口。
- 移除或下沉开发调试型按钮、音频链路统计卡、重复状态信息和冗余百分比读数。
- 保持 Home 域播放展开态、播放列表 bottom sheet、播放模式切换、进度拖动和后台播放状态全部可用。
- 为“无选中音频”和“已有播放内容”两种状态提供清晰但简洁的承载。

**Non-Goals:**
- 不在这次设计里重做播放业务逻辑、播放状态来源或后台服务职责。
- 不新增歌词页、评论页、分享页、倍速弹窗重设计等二级能力。
- 不在本次内重构 `PlaybackControls` 以外的整个播放器组件体系为新 module。
- 不要求首版就完全 1:1 还原云音乐，只要求风格方向、信息密度和层级明显收敛。

## Decisions

### 1. 播放页收敛为“三段式内容骨架”：主视觉区、进度区、播控区

`PlayerScreen` 将改成更单纯的三段式：
- **主视觉区**：以歌曲封面 / 唱片主视觉和轻量状态承载为主，避免再叠过多文字块；
- **进度区**：在进度条上方先展示歌曲名，再展示歌手名，然后才进入进度条与当前时长 / 总时长；
- **播控区**：保留上一首、播放 / 暂停、下一首，并维持播放模式与播放列表入口可达。

这样做的原因是：
- 云音乐风格的核心不是“信息很多”，而是“主视觉聚焦 + 主播控突出 + 次级信息克制”；
- 当前页面最影响观感的问题，不是缺功能，而是层级过多、块太碎；
- 三段式骨架能最大程度复用现有状态和回调，只重排承载方式。

备选方案：
- **方案 A：继续保留现有多卡片结构，只改颜色与圆角。** 风格变化会很有限，工具感依然很重。
- **方案 B：进一步做全屏沉浸歌词页式布局。** 改动会更大，也超出本次“只保留最基本播控”的范围。

因此本次选择“三段式骨架 + 极简层级”。

### 2. 移除开发 / 调试型浮动按钮，保留必要入口但不再抢主视觉

当前 `onRunUiTestEntry`、`onClearCache`、`onOpenLogin` 这些浮动按钮都不应继续出现在标准播放页主界面。`onPickAudio` 也不再以悬浮工具按钮形式叠在主视觉上。

这次的取舍是：
- **开发 / 调试入口**：从播放页主视觉移除；
- **登录入口**：不再在播放页单独占一个浮动位，统一回到用户中心或其他宿主入口；
- **选音频入口**：只在“当前没有任何选中音频”的空状态下，以单一主按钮保留，避免用户无从开始播放。

这样做的原因是：
- 用户明确要求“只保留最基本的播控以及必须的 UI”，这些工具入口不属于标准播放页的核心任务；
- 如果在有播放内容时仍保留这些浮动按钮，主视觉会继续被打碎；
- 但完全移除 `pick audio` 会让空状态失去进入能力，因此空状态仍需要一个清晰入口。

备选方案：
- **方案 A：全部保留，只缩小样式。** 仍然会破坏极简播放页的完整感。
- **方案 B：把所有入口收进播放页顶部更多菜单。** 首版成本更高，也超出本次需求边界。

因此本次选择“标准播放态移除工具入口，空状态保留单一主 CTA”。

### 3. 保留现有播放列表与播放模式入口，但将其视为“次级主控位”

虽然用户希望页面更极简，但现有主 spec 已经要求：
- 上一首 / 播放暂停 / 下一首是核心操作；
- 播放模式与播放列表入口仍需要保持可达。

因此本次不会砍掉这些入口，而是把它们收成更统一的音乐 App 播控带：
- 中间仍是最强的播放 / 暂停；
- 左右仍保留上一首 / 下一首；
- 播放模式与播放列表入口作为两侧次级控制位保留；
- `PlaylistBottomSheet` 保持既有交互和能力，不在本次重做。

这样做的原因是：
- 这能满足“保留最基本播控”的用户预期，同时不破坏既有 spec；
- 若直接移除播放模式或播放列表入口，会额外牵动 `playlist-management` / `playback-speed-control` 的能力边界；
- 视觉上只要统一样式并降低次级控件权重，就已经足够“极简”。

备选方案：
- **方案 A：只保留上一首 / 播放暂停 / 下一首。** 最简，但会直接损伤已有播放模式和列表入口能力。
- **方案 B：把播放模式或播放列表移到独立二级面板。** 可以更纯粹，但会引入更多交互迁移。

因此本次保留入口，但视觉上明显弱化为次级主控。

### 4. 音频信息卡、重复状态文案与百分比读数从主视觉中移除

以下内容将不再继续作为播放页主界面主块展示：
- `Audio Info` 统计卡（codec、bitrate、output mode、route input / output）
- 顶部 `progressPercent` 文本与底部重复的百分比文案
- 冗长的 route active / cue loaded 状态描述

播放页会只保留一条更轻的状态承载：
- 例如“准备中”“播放中”“已暂停”这类轻量状态；
- 但不会再把链路级输出信息继续放到主界面。

这样做的原因是：
- 这些信息对开发调试有用，但对日常听歌几乎都是噪音；
- 顶部和底部重复展示百分比，会进一步挤压歌曲名、歌手名与主视觉；
- 若未来确实还需要技术信息，更适合放在调试面板或开发构建入口，而不是默认播放页。

备选方案：
- **方案 A：保留信息卡但默认折叠。** 比完全展示更好，但首版仍会残留过重的结构。
- **方案 B：把信息卡搬到播放列表 bottom sheet。** 语义不合适，也会污染列表面板。

因此本次直接从主视觉中移除。

### 5. 播放页区分“空状态”和“已选中播放状态”

当前 `PlayerScreen` 即使没有有效选中内容，也仍围绕完整播放器结构布局。本次将明确拆成两种界面：
- **空状态**：展示轻量占位文案 + 单一“选择音频”CTA；
- **已选中播放状态**：展示简洁播放页主视觉、进度和播控。

这样做的原因是：
- 空状态没有必要承载完整播放器骨架，否则页面会显得空且怪；
- 有内容时再进入简洁播放器，页面会更纯粹；
- 这能保留 `onPickAudio` 的必要性，同时不让它在正常播放态打断视觉。

备选方案：
- **方案 A：无论有无选中内容都维持一套布局。** 开发成本低，但空状态观感最差。
- **方案 B：空状态直接跳回首页处理。** 流程会变绕，也会让展开播放器失去自洽性。

因此本次选择“双状态页面壳层”。

### 6. 右上角歌曲百科入口仅对在线歌曲显示，并按需请求摘要

播放页会在右上角补一个轻量的百科信息 icon，但它不是对所有播放内容都默认展示：
- **当当前播放项具备真实在线 `songId` 时**：显示百科 icon；
- **当当前播放项只是本地音频或缺少真实在线 `songId` 时**：不显示该入口。

点击百科 icon 后：
- 系统按需调用歌曲简要百科信息接口；
- 以不打断主播放页的轻量信息面板展示歌曲简介摘要；
- 面板需要覆盖加载中、请求失败和空摘要三种状态；
- 再次打开同一首歌时，可以复用当前页内已拿到的摘要状态，但不要求做跨会话持久缓存。

这样做的原因是：
- 用户希望在播放页就能看到歌曲简要百科，但本地音频没有稳定的在线歌曲身份，强行对所有内容展示入口会产生大量“点开不可用”或“查错歌”的风险；
- 右上角 icon 是比新建整块信息区更轻的入口，不会破坏已经收敛出的简洁播放页；
- 按需请求比进入播放页就预取更节制，也更符合“百科是补充信息”的定位。

备选方案：
- **方案 A：所有播放内容都显示 icon，本地音频置灰。** 入口统一，但会长期暴露“能看见却不能用”的噪音。
- **方案 B：本地音频也按文件名 / 标题自动搜索百科。** 命中错误内容的风险高，不适合首版。

因此本次选择“仅在线歌曲显示 icon + 按需请求摘要面板”。

## Risks / Trade-offs

- **[过度精简导致必要能力入口丢失]** → 保留上一首 / 播放暂停 / 下一首 / 播放模式 / 播放列表 / 进度拖动这些明确必须能力，不在首版直接删除已有 spec 约束入口。
- **[空状态失去开始播放的入口]** → 在空状态保留单一“选择音频”主按钮，而不是把 `pick audio` 完全移除。
- **[与现有 Home 展开壳层重复导航]** → 依赖 `PlayerExpandedScreen` 现有返回入口，`PlayerScreen` 本体不再重复造顶栏导航。
- **[移除调试信息后开发定位问题变难]** → 仅从默认播放页主视觉移除，不等于彻底删除对应回调或底层状态；必要时后续再补独立调试面板。
- **[只改样式但不改结构，效果不够明显]** → 本次明确改成三段式内容骨架，而不是仅调颜色 / 圆角。
- **[本地音频误展示百科入口]** → 将百科入口展示条件严格收敛到“存在真实在线 `songId`”，而不是仅凭文件名或当前标题推断。
- **[百科请求干扰主播放流程]** → 百科采用独立轻量面板与独立加载状态，不挤占主进度区和核心播控区。

## Migration Plan

1. 先在 `playback-page-shell` spec 中明确“必须保留 / 可以移除 / 需要弱化”的播放页元素清单，并在 `playlist-management` delta spec 中补充播放列表入口在新布局中的承载要求。
2. 改造 `PlayerScreen`：拆出空状态与播放状态，移除浮动工具按钮和音频统计卡，将内容收敛为主视觉区、进度区、播控区，并将歌曲名 / 歌手名固定放到进度条上方。
3. 在播放页右上角加入歌曲百科 icon 的承载位，并梳理“当前播放项是否具备真实在线 `songId`”的展示条件与面板状态模型。
4. 调整现有 `PlaybackControls`、封面 / 唱片主视觉与进度承载样式，使其更贴近云音乐风格，但保持既有交互回调不变。
5. 补充播放页 UI 验证，确认简洁布局下仍能完成播放 / 暂停、上下首、进度拖动、打开播放列表，以及在线歌曲百科入口 / 面板状态。
6. 若落地后发现某些必须入口确实被过度弱化，可优先通过样式回调与布局权重回退，而不先回退播放逻辑。

## Open Questions

- 播放页主视觉首版是继续保留当前简化唱盘动效，还是改成更直接的封面主图承载；这会影响视觉方向，但不影响本次骨架设计。
- 播放模式入口是否继续保留文字 / 图标复合表现，还是进一步收敛成更轻的图标按钮；这属于交互细节，后续可在 spec 阶段再定。
- 空状态除了“选择音频”外，是否还需要保留一句引导文案或最近播放提示；当前设计先以极简空状态为前提。
- 歌曲简要百科面板首版采用 bottom sheet 还是 dialog/card overlay；当前先假定为不打断主页面的轻量 bottom sheet，更符合当前播放页结构。
