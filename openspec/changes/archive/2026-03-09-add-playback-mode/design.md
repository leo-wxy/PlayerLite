## Context

当前播放器的主控区左侧仍是倍速入口，右侧是播放列表入口；播放列表状态只维护一份 `items + activeIndex` 的顺序化数据，远端播放快照当前只同步倍速、Seek 能力和输出信息。这样的建模还不足以同时表达“列表循环默认态”“随机后的实际播放顺序”“当前激活项稳定不变”和“随机模式下的列表展示偏好”。

这次变更要同时解决三件事：一是把播放模式切换 UI 顶替掉主控区原有的倍速入口；二是让随机播放不只是影响下一首决策，还能影响播放列表 UI 展示；三是把播放模式纳入 `playback-service` / `MediaSession` 的权威状态，使前台 UI、后台运行时和系统媒体控制对“当前模式”有同一份理解。

用户已经确认以下产品约束：
- 随机模式下，播放列表默认按随机后的播放顺序展示，而不是原始顺序。
- 播放列表中增加 `Checkbox` 文案“显示原始顺序”，且只在随机模式下生效。
- `Checkbox` 勾选状态需要持久化，应用重启后继续保留。
- 点击播放列表项只负责切到该项播放，不会重新洗牌。
- 只有切换到随机模式时才会生成或刷新随机顺序。
- 从随机模式切回顺序/列表循环/单曲循环时，当前正在播放的歌曲保持不变。

## Goals / Non-Goals

**Goals:**
- 支持 `列表循环`、`单曲循环`、`随机播放` 三种播放模式，并按固定顺序循环切换。
- 用播放模式入口替换主控区当前的倍速入口，并让当前模式通过图标可见、可直接切换。
- 在播放列表状态中同时保存原始顺序、随机顺序、当前激活项和“显示原始顺序”偏好。
- 让随机模式下的播放列表默认按随机顺序展示，并允许用户通过 `Checkbox` 切回原始顺序查看。
- 将播放模式同步到后台播放服务和 `MediaSession`，使系统控制、前台 UI 和后台自然播完决策保持一致。
- 把上一首 / 下一首 / 自然播完后的推进逻辑统一收口为模式驱动规则，而不是散落在多个层级。
- 保证模式切换时当前曲目不被强制切换；切入随机模式才重建随机顺序。

**Non-Goals:**
- 不在本次实现“智能随机”“避免短时间重复”或可配置随机算法。
- 不把 `显示原始顺序` 暴露给 `MediaSession` 或系统外部控制界面；它是 app 内的 UI 展示偏好。
- 不在本次为通知栏、耳机按键、车机界面额外设计独立的播放模式按钮布局；首版先保证状态同步与命令通路。
- 不在本次重新设计倍速能力的新入口；主控区左侧位置由播放模式占用后，倍速入口迁移作为后续独立问题处理。

## Decisions

1. 使用“双顺序 + 稳定激活项 ID”模型替代当前单一 `activeIndex` 语义。
   - 方案：将播放列表持久化模型升级为同时包含 `originalItems`、`shuffledOrderIds`、`activeItemId`、`playbackMode`、`showOriginalOrderInShuffle`。UI 和后台需要的“当前索引”都由这组状态按场景派生，而不是继续把单一 `activeIndex` 当作所有视图和模式下的唯一真相。
   - 原因：随机模式要求“展示顺序”和“实际播放顺序”可切换，但当前曲目要稳定不变；如果继续只保存 `items + activeIndex`，一旦切换显示顺序或模式，当前项会因为 index 漂移而失真。以 `activeItemId` 作为稳定锚点，才能在原始顺序、随机顺序和模式切换之间安全映射。
   - 备选方案：
     - 直接把 `items` 改写为随机顺序：实现表面简单，但无法同时保留原始顺序，也会把拖拽、删除、恢复逻辑搅在一起。
     - 只保存原始顺序 + 随机种子：存储更少，但列表内容变化或恢复过程会导致随机顺序漂移，难以保证重启后一致。

2. 切换到随机模式时生成新的随机顺序，并以“当前播放项保持不变”为硬约束。
   - 方案：进入随机模式时，根据当前 `originalItems` 生成一份新的 `shuffledOrderIds`；当前激活的媒体项继续保持播放，不因为模式切换而跳歌。新随机序列只在“切入随机模式”这一时刻生成，之后点击列表项、切到上一首/下一首都只消费当前这份随机序列，不重新洗牌。
   - 原因：用户已经明确“随机触发的行为只有切换模式”，因此随机模式需要像一次会话级队列快照，而不是每次点击或切歌都重新生成。这样也更容易让上一首 / 下一首在随机模式下保持可预期。
   - 备选方案：
     - 每次点列表项都重洗随机队列：会让用户失去对当前随机会话的预期，不符合确认过的行为。
     - 每次自然播完后按需抽下一首：能实现“随机”，但上一首 / 下一首无法形成稳定队列，调试与恢复都更复杂。

3. 随机模式下的播放列表 UI 默认展示随机顺序，并提供持久化的“显示原始顺序”复选框。
   - 方案：`PlaylistBottomSheet` 在随机模式下显示一个 `Checkbox("显示原始顺序")`。默认未勾选时，列表按 `shuffledOrderIds` 对应的顺序展示；勾选后仅切换展示到 `originalItems`，不会改变当前播放项或后台实际播放顺序。该偏好持久化到本地，重启 app 后恢复。
   - 原因：用户既要“随机播放时看到当前真实播放顺序”，又要“随时能切回原始顺序查看”，因此 UI 必须显式区分“播放顺序”和“展示偏好”。把该偏好保存在播放列表状态附近，能和原始顺序 / 随机顺序一起恢复。
   - 备选方案：
     - 始终展示原始顺序，只在当前项上标记随机结果：用户无法直接看到随机会话的实际队列，不符合需求。
     - 用菜单而不是复选框切换：交互更隐蔽，不如 `Checkbox` 直观。

4. 在随机模式下，“展示原始顺序”是只影响视图的开关，不影响播放状态和系统会话状态。
   - 方案：`showOriginalOrderInShuffle` 只在 app 内部控制列表展示，后台 `PlaybackProcessRuntime` 和 `MediaSession` 继续基于实际生效的播放模式与随机队列做决策。点击列表项时，通过 item ID 找到目标媒体并切换播放；不论当前展示的是原始顺序还是随机顺序，都不会触发重洗。
   - 原因：用户确认 `Checkbox` 是“播放列表 UI 展示设置”，而不是新的播放模式。把它混进 `MediaSession` 会让外部控制界面承担不必要的 UI 语义，也会污染后台运行时职责。
   - 备选方案：
     - 把 `Checkbox` 一并同步给 `MediaSession`：对通知栏、耳机按键和车机控制没有明确价值，反而扩大状态面。

5. 播放模式由业务层维护，`MediaSession` / 后台服务只消费投影后的播放队列并同步执行态。
   - 方案：在业务层的播放列表状态中维护 `playbackMode`、`originalItems`、`shuffledOrderIds` 与 `activeItemId`；`PlayerViewModel` / `PlayerRuntime` 根据这些状态生成当前生效的播放队列和活动项，再同步给 `playback-service`。`PlaybackMetadataExtras` / `RemotePlaybackSnapshot` 仍可透出当前模式，作为会话状态投影，而不是让 service 成为原始播放模式和列表顺序的真相源。
   - 原因：播放模式和随机顺序本质上属于播放列表业务语义，和 UI 展示、原始顺序视图切换、随机顺序持久化强绑定；把它们留在业务层更容易保持一致，也能让 service 聚焦在“按给定队列播放”。
   - 备选方案：
     - 让 service 直接拥有原始列表、随机顺序与模式：会把业务语义压到播放执行层，增加状态分叉风险。
     - 仅在 `PlayerUiState` 存模式而不同步给 service：会导致后台播放和会话快照无法正确投影当前模式。

6. 主控区播放模式入口采用图标直切，而不是弹出选择弹窗。
   - 方案：左侧主控位只显示当前模式图标；用户每次点击时按 `列表 -> 单曲 -> 随机 -> 列表` 的固定顺序循环切换，并使用短 Toast 提示当前模式。
   - 原因：用户已经明确不需要额外文案和弹窗，而且既然“顺序播放”和“列表循环”无法通过图标清晰区分，就直接移除顺序模式，以列表循环作为默认模式，避免增加理解成本。
   - 备选方案：
     - 弹出对话框选择模式：可见性更强，但交互更重，不符合本次确认的产品方向。
     - 图标长按展开模式菜单：实现更复杂，也不是当前需求。

7. 空的远端快照不得覆盖本地持久化的播放模式。
   - 方案：当 `MediaSession` 快照没有当前媒体项时，前台运行时继续保留业务层已恢复的 `playbackMode`，只在远端确实存在当前媒体项时才把 service 投影的模式回写到本地。
   - 原因：播放模式的真相源在业务层，service 只是消费投影后的队列；应用重启后，如果 service 仍处于空闲默认态，不能用它的默认 `sequential` 去冲掉本地恢复出的 `shuffle/list-loop/...`。
   - 备选方案：
     - 无条件接受远端快照模式：会导致“已持久化但启动后 UI 又回到顺序播放”的假恢复问题。

8. 将“上一首 / 下一首 / 选中列表项”的主导航决策放在业务层，而让 service 处理当前生效队列内的播放执行。
   - 方案：业务层统一计算列表循环、单曲循环和随机播放下的有效播放队列与当前激活项；UI 触发上一首 / 下一首或列表点击时，先更新业务层状态，再驱动 service 播放新的目标项。service 内部只需要对当前已同步队列执行播放、切项和必要的完成态衔接，不直接维护原始播放列表的业务规则。
   - 原因：这样可以让播放模式、随机顺序、原始顺序视图和激活项始终围绕同一份业务状态演进，同时避免让 service 变成第二套播放列表状态机。
   - 备选方案：
     - 让 service 自己决定所有 `next / previous / onCompleted` 规则：会把播放列表业务层和执行层耦合得更深。
     - 让 UI 和 service 各自维护一套导航逻辑：会形成双重真相源。

9. 对列表结构变化采用“修补当前顺序，而不是隐式重洗”的策略。
   - 方案：删除项目时从 `originalItems` 和 `shuffledOrderIds` 同步移除对应 ID；新增项目时追加到 `originalItems`，并以稳定规则追加到当前 `shuffledOrderIds` 末尾；随机模式下若用户切到“显示原始顺序”，则拖拽重排只修改 `originalItems`，当前随机序列保持不变，直到下次重新切入随机模式才整体刷新。若当前展示的是随机顺序，则禁用拖拽重排以避免语义混乱。
   - 原因：用户已经要求“只有切模式才触发随机行为”，因此结构变化不能偷偷把整条随机序列重建。与此同时，拖拽重排在随机视图里没有自然语义，禁用它比“猜用户想改哪条顺序”更安全。
   - 备选方案：
     - 任意列表改动都立刻重洗：违背“只有切模式触发随机”的约束。
     - 允许在随机顺序视图里直接拖拽：最终到底改的是原始顺序还是随机顺序会变得含糊。

10. 通过播放列表状态版本升级完成平滑迁移。
   - 方案：把 `PlaylistState.VERSION` 从当前版本升级，在解码旧版本时将现有 `items` 视为 `originalItems`，把旧的 `activeIndex` 映射为 `activeItemId`，并初始化 `playbackMode = list_loop`、`shuffledOrderIds = originalItems.ids`、`showOriginalOrderInShuffle = false`。迁移后立即按新格式回写；若读到旧的 `sequential` 持久化值，则映射为 `list_loop`。
   - 原因：现有用户已经有本地播放列表存量数据，如果不做迁移，升级后会丢失当前播放列表。把旧顺序视为原始顺序是最自然、风险最低的兼容方式。
   - 备选方案：
     - 直接废弃旧状态：实现最简单，但会导致升级后播放列表清空。

## Risks / Trade-offs

- [双顺序状态让状态映射复杂度上升] → 使用 `activeItemId` 作为唯一稳定锚点，所有索引都按视图/模式派生，避免 index 直接跨模式复用。
- [随机视图与原始视图切换时，用户可能误以为会改变实际播放顺序] → 在随机模式下把 `Checkbox` 文案限定为“显示原始顺序”，并保持切换后当前播放项与后台行为不变。
- [模式状态在业务层、MediaSession、后台服务之间不同步] → 以业务层播放列表状态为真相源；service 只消费投影结果，并通过远端快照回传执行态和当前模式投影。
- [随机模式下的上一首 / 下一首与自然播完逻辑不一致] → 统一通过模式导航层处理三个入口，而不是分别写分支。
- [从旧版播放列表状态迁移时当前项丢失] → 迁移时先将旧 `activeIndex` 解析成 `activeItemId`，再基于 ID 派生新索引。
- [主控区左侧被播放模式占用后，倍速能力入口消失形成功能回退] → 将其明确记录为本次设计边界，若产品仍要求保留倍速入口，需在后续变更中单独规划新的入口位置。
- [随机视图下仍允许拖拽会导致“改原始顺序还是改随机顺序”不清晰] → 在随机顺序展示态禁用拖拽；只有查看原始顺序或处于非随机模式时允许重排。

## Migration Plan

1. 在 `app` / `playback-contract` / `playback-client` / `playback-service` 中补齐 `PlaybackMode` 状态模型、自定义 session command 和 extras 读写。
2. 升级播放列表持久化模型，增加 `originalItems`、`shuffledOrderIds`、`activeItemId`、`playbackMode`、`showOriginalOrderInShuffle`，并实现对旧版本 `PlaylistState` 的迁移。
3. 调整前台 `PlayerRuntime` / `PlaylistSessionCoordinator` 的状态派生逻辑，让 UI 可以区分“实际播放顺序”和“当前展示顺序”。
4. 将 `PlaybackControls` 左侧入口从倍速按钮替换为播放模式按钮，并用当前模式图标做可见反馈；点击后按“列表 -> 单曲 -> 随机”固定顺序直接循环切换并显示 Toast。
5. 改造 `PlaylistBottomSheet`：在随机模式下增加 `Checkbox("显示原始顺序")`，默认展示随机顺序；随机顺序展示态禁用拖拽重排。
6. 在业务层接入模式切换、随机序列生成和模式驱动的 `next / previous / 列表选择` 决策；切出随机模式时保持当前歌曲不变。
7. 让 `MediaSession` / `playback-service` 消费业务层投影后的当前队列，并通过 extras 持续暴露当前 `playbackMode` 作为执行态投影。
8. 为状态迁移、模式切换、随机顺序恢复、列表展示切换、自然播完、上一首 / 下一首和 session 状态同步补充测试；回滚时可先恢复旧 `PlaylistState` 读写和主控区左侧倍速入口。

## Open Questions
