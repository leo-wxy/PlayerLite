## Context

当前搜索链路已经能在 `feature-search` 中产出 `SearchRouteTarget.Artist`、`SearchRouteTarget.Playlist` 和 `SearchRouteTarget.Album`，`SearchActivity` 也已经把结果点击统一转交给 `SearchRouteHandler`。但首页发现流和个人中心内容列表仍只有展示字段，没有稳定的点击目标；这导致首页 banner、推荐卡、快捷入口，以及个人中心里的收藏歌手、用户歌单、收藏专栏都容易落回“看起来像入口、点了却没有明确结果”的半成品状态。同时，歌手页和歌单页虽然已经有了基础实现，但歌手/歌单/专辑简介浮层仍需支持滑动查看全文，才能真正承载长文本。

应用当前的数据装配模式仍然是 `app` 作为组合根，通过 `AppContainer` 统一创建 `JsonHttpClient`、用户态透传和 feature repository。首页发现与搜索已经都走这条链路，因此歌手详情页、歌单详情页和专辑详情页如果要新增接口访问，仍应沿用 `repository -> remote data source -> mapper` 模式，而不是在 Activity 或 Compose 中直接碰原始 JSON。

这次需求已经扩展为七类接口：获取歌手详情、歌手简要百科信息、歌手热门 50 首歌曲、获取歌单详情、获取歌单所有歌曲、获取专辑内容、专辑动态信息。UI 方向要求“偏云音乐”，并进一步明确希望歌手详情头图支持“头像主色取色 + 渐变氛围层”。与此同时，仓库仍要求 `MainActivity` 保持轻量、Compose UI 尽量无状态，所以视觉升级需要通过共享页面骨架、状态模型和局部可控的视觉提取逻辑实现，而不是把交互和请求逻辑散落到界面代码里。

## Goals / Non-Goals

**Goals:**
- 打通搜索结果到歌手详情页、歌单详情页、专辑详情页的真实跳转闭环，让 `SearchRouteTarget.Artist`、`SearchRouteTarget.Playlist` 与 `SearchRouteTarget.Album` 都有明确宿主承接。
- 打通首页发现内容 item 与个人中心内容 item 的点击闭环，让所有首页/个人中心内容入口都具备明确反馈，并在可识别时进入歌手、歌单、专辑详情页。
- 为首页 banner、快捷入口和暂未站内承载的资源类型提供统一的外链/兜底处理，而不是让不同卡片在 UI 层各自硬编码分支。
- 在 `app` 内新增专辑详情页，并完成歌手详情页、歌单详情页剩余 UI 能力收口，继续复用 `JsonHttpClient` 与 `requiresAuth` 的统一约定。
- 让歌手详情页承载基础信息、简要百科信息、热门 50 首歌曲、头像点击放大预览，以及基于头像主色提取的渐变头图。
- 让歌单详情页同时承载歌单头部信息与“歌单所有歌曲”列表，并将作者与“XX 首歌曲”做出清晰层级，同时支持简介浮层展开全文。
- 让歌单详情页与专辑详情页都采用分页歌曲列表，首屏优先出头部和首批曲目，并在续页阶段保持页面可继续浏览。
- 让专辑详情页同时承载专辑基础信息、简要百科信息、歌曲列表与动态元信息，并在动态接口较慢或失败时仍保持专辑主体内容可读。
- 让歌手详情页、歌单详情页与专辑详情页共享一套偏云音乐风格的详情页视觉骨架，统一头图氛围、标题信息层级、返回栏和内容滚动节奏。
- 让歌单详情页和专辑详情页像歌手页一样，优先基于封面主色提取渐变氛围层，封面缺失或取色失败时退回默认渐变。
- 让歌手/歌单/专辑简介浮层都支持滑动查看全文，并保持关闭入口稳定可见。
- 把详情页的加载态、失败态与重试入口定义清楚，避免接口波动时页面彻底不可用。

**Non-Goals:**
- 本次不扩展到歌曲详情页、用户详情页，也不要求把搜索里所有结果类型都接出真实详情。
- 本次不引入新的全局导航框架；详情跳转继续由宿主 `app` 通过 Activity/Intent 接线。
- 本次不引入新的 WebView 或完整兼容所有 `orpheus://` 协议；站外资源继续优先交给系统宿主处理。
- 本次不补歌手专辑分页、歌单评论流、专辑评论流、收藏、关注、分享等二级能力；歌手页在本次仅额外补充“热门 50 首歌曲”列表，专辑页只补动态元信息而不继续扩展评论列表。
- 本次不做复杂折叠动效或沉浸式大规模动画；视觉升级聚焦于取色渐变、主视觉、信息层级与浮层承载。

## Decisions

### 1. 详情导航继续留在 `app` 宿主层，`feature-search` 只输出稳定 route target

现有 `feature-search` 已经通过 `SearchRouteTarget` 把“点击的是歌手、歌单还是专辑”表达清楚，因此这次不需要把详情页逻辑反向塞回搜索 module。具体做法是：
- 继续保留 `SearchRouteTarget` 作为搜索模块对外的稳定导航语义；
- 在 `PlayerLiteApplication.searchHostDependencies()` 中补齐 `routeHandler`；
- 由 `routeHandler` 根据目标类型打开 `ArtistDetailActivity`、`PlaylistDetailActivity` 或 `AlbumDetailActivity`，并通过明确的 `Intent` extra 只传递稳定 ID。

这样做的原因是：
- 搜索模块已经完成了“识别目标”的职责，再让它直接依赖 `app` 详情页会破坏刚建立起来的 module 边界；
- 仓库当前没有统一导航框架，Activity 级宿主接线是最贴合现状、最容易回退的方案；
- 详情页未来可能继续演化，但搜索模块只需要稳定输出 `type/id`，不应感知详情实现细节。

因此本次继续选择“搜索 module 只抛 route target，详情页导航由 `app` 宿主承接”。

### 2. 三类详情页先留在 `app`，各自拥有独立 repository，而不是继续塞进 `SearchRepository`

这次详情页虽然和搜索入口有关，但它们的生命周期、状态模型和远端数据完全不同，不适合继续堆进 `SearchRepository`。设计上会在 `app` 中新增三组能力：
- `ArtistDetailRepository`：负责获取歌手详情、歌手简要百科信息并映射为页面模型；
- `PlaylistDetailRepository`：负责获取歌单详情与歌单所有歌曲，并向页面暴露组合后的详情内容；
- `AlbumDetailRepository`：负责获取专辑内容与专辑动态信息，并向页面暴露组合后的详情内容。

它们都通过 `AppContainer` 注入，继续复用已有的 `JsonHttpClient` 与会话头透传能力。页面层只消费本地页面模型，不直接读取原始 `JsonObject`。

这样做的原因是：
- 搜索 repository 的职责是“搜索”，不是“承载详情页完整数据生命周期”；
- 歌单详情页与专辑详情页都依赖多接口或多段状态，继续塞进搜索 repository 会让搜索 feature 的边界变得模糊；
- 维持和 `HomeDiscoveryRepository` 类似的装配模式，后续测试、缓存和错误处理都更清晰。

因此本次选择“详情实现暂留 `app`，但数据层按独立 repository 收敛”。

### 3. 歌单详情页继续采用“两段式承载”，并补强作者/歌曲数元信息与简介浮层

歌单详情页同时依赖“歌单详情”和“歌单所有歌曲”两个接口。如果把它们绑定成一次性全有或全无的页面状态，歌曲列表较慢或大歌单返回较重时，用户会长时间停留在整页白屏。因此页面状态继续拆成两部分：
- `headerState`：承载歌单封面、标题、创建者、描述、歌曲数等头部信息；
- `tracksState`：承载全部歌曲列表的加载、内容、空态和错误态。

在 UI 上，歌单头部再额外补两点：
- 将歌单作者和“XX 首歌曲”作为可区分的元信息块展示，而不是仅并排落成普通正文；
- 歌单简介直接并入头部信息区，只保留最多两行摘要，点击后以浮层展示完整文案，避免再额外引入一张独立简介卡把首屏主信息挤散。

因此本次明确采用“头部先可见、歌曲列表独立加载”的两段式承载，并把歌单元信息做成更接近云音乐层级的视觉块。

### 4. 歌手详情页首版聚焦基础信息、简要百科信息与热门 50 首歌曲，并允许头像取色驱动头图渐变

歌手详情页的首版定位仍然是“一个可以真实进入、可展示基础信息与代表性热门歌曲列表的详情页”，而不是完整的歌手作品中心。页面模型优先覆盖：
- 歌手头像/封面、名称、别名等头部信息；
- 作品数量、头部两行简介摘要与 `artist/desc` 返回的简要百科信息；
- 热门 50 首歌曲列表的标题、副标题、封面和时长等紧凑行信息；
- 头像点击后以前景浮层提供更大的主视觉预览；
- 加载失败、无可展示信息时的兜底状态。

在视觉上，歌手头图会优先使用“从头像主色提取出的主色 + 纵向渐变/遮罩”构成氛围层；当图片取色失败、头像缺失或加载较慢时，退回稳定的默认渐变，不让页面出现闪烁或白屏。

这样做的原因是：
- 用户明确希望歌手详情头图更接近云音乐的视觉质感；
- `artist/detail.briefDesc` 偏基础摘要，而 `artist/desc` 会返回更接近百科的分段内容，融合后能明显提升详情页信息密度；
- 仅在歌手页启用头像取色，能把体验收益集中在最需要情绪化头图的页面，同时控制复杂度；
- 有默认渐变兜底后，取色异步失败不会影响页面主体承载。

因此首版选择“歌手基础信息 + 简要百科信息 + 热门 50 首歌曲 + 头像放大预览 + 头像取色渐变头图”的收敛方案。

### 5. 专辑详情页采用“专辑内容主状态 + 动态信息副状态”承载

专辑接口和歌单页不同：`/album` 一次返回专辑基础信息、描述性百科信息和歌曲列表，而 `/album/detail/dynamic` 返回评论数、分享数、收藏数等动态元信息。因此页面状态会拆成两层：
- `contentState`：承载专辑封面、标题、歌手、公司、发行时间、简介/百科摘要与歌曲列表，是页面的主体可用性来源；
- `dynamicState`：承载评论数、分享数、收藏数等动态元信息，可在主体内容成功后独立加载、失败或重试。

这样做的原因是：
- 专辑主内容和动态信息并非同等关键，动态接口失败不应该拖垮整个专辑页；
- `/album` 已经返回歌曲列表和长描述，页面可以用一条主请求完成可读首屏与百科信息，再补齐动态数字；
- 这种双层状态与歌单页“头部/列表分段”保持了相同设计思路，降低心智负担。

因此本次明确采用“专辑内容主状态 + 动态信息副状态”的承载方式。

### 6. 三个详情页共享一套云音乐风格 Detail Scaffold，但允许歌手页头图使用动态取色

为了让歌手详情页、歌单详情页和专辑详情页在视觉上更像同一产品体验，本次会抽一套共享的详情页骨架，至少统一以下元素：
- 顶部 edge-to-edge 返回栏与半透明叠层；
- 大面积氛围头部区域；
- 头部主视觉组件插槽：歌手使用圆形头像，歌单与专辑使用圆角方形封面；
- 标题、辅助信息、描述文字和主内容列表之间的统一间距、字号和层级；
- 内容区默认使用 `LazyColumn` 的纵向滚动结构，列表条目保持更接近音乐 App 的紧凑密度。

其中歌单页和专辑页继续使用可控的品牌化默认渐变，而歌手页允许在默认渐变之上用头像主色修正氛围色。这样既保持三个详情页是一套产品语言，又把动态取色的复杂度限制在用户明确最在意的页面。

### 7. 错误处理采用“页级失败 + 区块级重试”组合，而不是一律 Toast

详情页承载的是用户主动进入的内容页，失败时不能像搜索入口初期那样只做临时提示。设计上会把状态分成：
- 首次进入且关键内容不可用时的页级 loading / error / retry；
- 歌单详情页中歌曲列表独立失败时的区块级重试入口；
- 专辑详情页中动态信息独立失败时的区块级重试入口；
- 已有内容时再次重试不打断当前可见信息，尽量避免整页闪回骨架。

这样做的原因是：
- 详情页失败比搜索建议失败更影响用户判断，必须给出稳定重试通道；
- 分段状态能让“头部已成功、列表失败”或“专辑内容已成功、动态信息失败”的场景仍然保持页面可用；
- 这也更符合 Compose 无状态 UI 的组织方式，ViewModel 负责状态切换，界面只负责渲染。

### 8. 首页发现流与个人中心列表统一产出内容动作，宿主层负责执行

首页与个人中心都已经是 `app` 内的宿主 UI，因此这次不再为每种卡片单独发明点击处理，而是统一抽象一个轻量内容动作模型，例如：
- `OpenDetail(SearchRouteTarget)`：进入站内歌手/歌单/专辑详情；
- `OpenUri(uri: String)`：交给宿主层打开可解析的站外链接或系统深链；
- `Unsupported(message: String)`：当前版本暂未承载，但点击后给出稳定提示。

首页发现流 `HomeSectionItemUiModel` 和个人中心 `UserCenterCollectionItemUiModel` 都会新增该动作字段，mapper 负责尽可能把原始资源类型映射成上述三类动作。执行动作的职责留在 `MainActivity` 或其宿主辅助层，`MainShellScreen` 只负责把 item click 回调向外抛出，不直接拼装 `Intent`、`Uri` 或 Toast 文案。

这样做的原因是：
- 搜索、首页、个人中心都能复用已有的 `SearchRouteTarget` 和详情页 `Intent` factory，避免三套路由语言并存；
- `MainActivity` 仍保持“薄宿主”角色，只做动作分发与系统 `startActivity`，不侵入业务解析；
- UI 层不需要知道 `playlist`、`album`、`artist`、`banner url`、`专栏未支持` 这些细节，组合和测试都更稳定。

具体映射边界如下：
- 首页里能稳定识别为 `playlist / album / artist` 的资源，一律映射为 `OpenDetail`；
- banner、快捷入口、活动卡若无法站内承载但有 `url/action`，映射为 `OpenUri`；
- 既无稳定详情页又无可解析链接的首页 item，映射为 `Unsupported`；
- 个人中心中收藏歌手映射为 `OpenDetail(Artist)`，用户歌单映射为 `OpenDetail(Playlist)`，收藏专栏先映射为 `Unsupported("当前版本暂不支持打开专栏详情")`。

### 9. 详情页全文浮层统一改为可滚动承载，而不是静态全文卡

歌手、歌单、专辑三个详情页都已经引入“摘要 + 点击浮层展开全文”的交互，但长文本如果只放进静态 Column，很容易在小屏设备上超出视窗，导致用户无法查看全部内容。因此共享的 `DetailTextDialog` 需要改成可滚动承载：
- 文本主体放入可滚动容器，允许用户在浮层内上下滑动查看全文；
- 关闭按钮固定在浮层底部，避免长文时退出入口被推到屏幕外；
- 浮层高度保留最大值限制，避免全文直接撑满整屏导致层级感消失。

这样做的原因是：
- 用户明确要求“滑动查看全文”，这是长文简介真实可用性的底线；
- 改在共享 detail scaffold/工具层收口，一次调整即可覆盖歌手、歌单、专辑三个详情页；
- 浮层的滚动和关闭入口都稳定后，简介入口就可以收敛在详情页顶部信息区，只显示最多两行摘要，首屏信息层级不会被长文挤散。

### 10. 歌单页与专辑页复用封面主色取色逻辑，详情头图不再只依赖默认渐变

歌手页已经有“头像主色取色 + 渐变氛围层”，用户希望歌单和专辑详情页也拥有同样的头图质感。因此这次会把取色逻辑继续收敛到共享层，让歌单封面和专辑封面也能驱动头图：
- 歌单页优先从歌单封面提取主色并生成头部渐变；
- 专辑页优先从专辑封面提取主色并生成头部渐变；
- 取色失败、图片缺失或加载较慢时，继续退回稳定默认渐变，避免首屏闪烁。

这样做的原因是：
- 三类详情页会形成一致的“内容主视觉驱动头图氛围”的产品语言，而不是只有歌手页更精致；
- 取色逻辑已经在歌手页验证过，实现上适合抽成共享 helper，而不是再写三套分叉；
- 有默认渐变兜底后，视觉增强不会影响页面稳定性。

### 11. 搜索结果类型筛选条采用固定 chip 高度，避免切换类型时视觉抖动

搜索结果页顶部的类型筛选条目前依赖文案和选中态 padding，自适应高度在不同字体/状态下容易出现轻微跳动。为了让结果切换更稳，这次会把筛选 chip 收敛到固定高度：
- 所有类型 chip 使用统一固定高度；
- 选中与未选中状态通过颜色、描边、字重和轻微缩放区分，而不是改变垂直内边距；
- 横向筛选条自身保持稳定高度，分页切换时不会因为 chip 重测量而抖动。

这样做的原因是：
- 搜索页已经是高频切换场景，筛选条抖动会放大“页面不稳”的感知；
- 固定高度不影响现有类型滚动和居中逻辑，只是把视觉约束收紧；
- 继续保留轻微缩放和颜色变化，交互反馈不会因为高度固定而变得僵硬；
- chip 文案在固定高度内使用真正的中心对齐，避免高度固定后文字仍然出现上偏或下偏。

### 13. 详情页返回键固定在宿主外层，不再随 hero 正文一起滚动

共享 detail scaffold 之前把返回键放在 `LazyColumn` 的 hero item 里，这会导致用户向上滚动内容时返回键跟着滑走。既然三类详情页都是内容详情，不是长表单页面，返回入口应该始终稳定可见。因此这次会把返回键移到 scaffold 外层的固定悬浮层：
- 返回键位于详情页顶层 Box 中，保持在状态栏下方固定位置；
- hero 区在布局上为该固定返回键预留顶部空间，避免主视觉与返回键重叠；
- 歌手、歌单、专辑三个详情页共享这条固定返回入口行为。

### 12. 歌单和专辑歌曲列表采用分页加载，避免长列表阻塞首屏

歌单和专辑详情都可能承载很长的曲目列表。如果继续一次性拉全量并立即渲染，网络响应和 Compose 首次布局都会放大首屏卡顿。因此这次会把两类详情页的歌曲列表改成分页承载：
- 头部信息继续优先加载并尽快可见；
- 歌曲列表首屏只拉取或展示第一页；
- 用户滚动接近列表尾部时自动触发下一页加载；
- 续页失败时仅在列表尾部展示重试，不回退已显示内容。

接口层策略：
- 歌单详情的 `/playlist/track/all` 追加 `limit/offset` 参数，按页拉取；
- 专辑详情的 `/album` 也追加 `limit/offset` 参数；如果服务端仍返回全量数据，则页面层继续按本地分页节奏逐批展示，保证交互和渲染层面仍然分段。

这样做的原因是：
- 用户明确反馈“卡太久”，问题不只是布局多，还包括长列表首包和首次渲染；
- 歌单和专辑的头部价值远高于一次性看到全部曲目，分页能显著降低首屏等待感；
- 即使专辑接口最终不严格按页返回，页面层分批展示仍能降低 Compose 初次渲染成本。

## Risks / Trade-offs

- **[歌单歌曲列表很长，首屏可能被拉慢]** → 通过“header 与 tracks 分段状态”降低等待感，并把列表渲染保持在 `LazyColumn` 范式内，避免一次性重布局。
- **[歌手热门歌曲列表较长，可能压缩头部首屏空间]** → 让头部信息保持在歌曲列表之前，并使用紧凑歌曲行样式，避免详情页变成纯列表页。
- **[头像主色提取带来异步波动]** → 只在歌手页启用取色，并提供稳定默认渐变兜底；取色失败不会阻塞页面渲染。
- **[Activity 数量增加，页面模板可能出现重复]** → 通过共享 Detail Scaffold、统一 Intent factory 和通用状态组件压缩重复成本。
- **[专辑动态信息接口失败，页面层级可能显得不完整]** → 将动态信息做成独立区块和重试入口，不让主内容回退成白屏。
- **[搜索只接入歌手/歌单/专辑详情，其他结果类型行为不一致]** → 在宿主路由层显式处理已支持目标，未支持类型继续维持原状，避免错误跳转；后续 capability 可以逐步补全。

## Migration Plan

1. 在 `app` 的 `AppContainer` 中新增专辑详情 repository 的创建和缓存，并扩展 `Services` 装配。
2. 在 `PlayerLiteApplication` 中补齐 `SearchHostDependencies.routeHandler`，将 `SearchRouteTarget.Artist` / `Playlist` / `Album` 分发到新的详情 Activity。
3. 新增专辑详情页的 Activity、ViewModel、页面状态模型、remote data source 与 mapper；同时补齐歌手页与歌单页剩余浮层和头部能力。
4. 扩展首页发现流与个人中心列表的数据模型和 mapper，为 item 补齐统一内容动作，并在 `MainActivity` 宿主层新增执行逻辑。
5. 扩展共享的 Detail Scaffold 与通用状态区块，再分别接入歌手页、歌单页和专辑页的头部/列表内容，并补齐头像预览、简介浮层与动态元信息区块。
6. 补充搜索结果点击、首页/个人中心内容动作映射、详情页数据映射和状态切换相关测试；如果中途发现路由或页面稳定性问题，可先回退为保留动作字段但暂时禁用宿主执行，数据层实现不必推翻。

## Open Questions
