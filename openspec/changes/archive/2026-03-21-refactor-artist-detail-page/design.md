## Context

当前歌手详情页已经具备从搜索、首页、个人中心与播放器歌手入口进入的完整路由闭环，也已经实现了基础信息、简介浮层、头像预览、热门歌曲列表与“全部播放”能力。但现状仍有三个明显约束：

1. 数据层仍把百科信息建立在 `/artist/desc` 上，而你已经明确新的目标接口是需登录的 `/ugc/artist/get`。
2. 页面内容仍停留在“基础信息 + 百科 + 热门歌曲”的首版承载，尚未支持歌手专辑列表，而 `/artist/album` 已经成为本次范围的一部分。
3. 当前 `ArtistDetailActivity.kt` 同时承载 Activity、页面骨架、分区 UI 和局部交互，`ArtistDetailRepository.kt` 也把接口定义、远端数据源和 mapper 放在同一文件里；如果后续要按你要求启用 multiagent 并行实现，这种边界不利于拆分。
4. 头部信息目前仍缺少关注状态、视频数、粉丝数等增强字段，而新的 `/artist/detail/dynamic` 与 `/artist/follow/count` 已经进入范围；与此同时，这些增强信息只应作为头部软依赖，不应反向阻塞基础详情页的可用性。
5. 这次确认的交互约束不仅是“有 sticky tab”，还包括 collapsed top bar 与 sticky tabs 必须职责分离，不能通过双重状态栏占位去硬撑布局；顶部渐变、标题淡入与头部信息淡出需要随滚动连续过渡，而不是硬切；切换 tab 时需要保持外层滚动位置稳定，并让各 tab 保留各自独立的内容滚动位置，而不是把上一 tab 的内容偏移错误带到当前 tab。
6. 这轮交互收口还明确了测试与验收口径：优先验证用户可见结果，例如当前 tab 面板首项是否可见、旧 tab 内容是否消失、简介卡片在未吸顶前是否仍在，而不是把内部 spacer、高度占位或临时 test tag 当成主要行为契约。

仓库现有约束仍然成立：详情页实现继续留在 `app` 宿主内，通过 `AppContainer` 注入 repository，网络访问继续复用 `JsonHttpClient` 和现有鉴权链路；UI 需要尽量保持无状态，编排逻辑留在 ViewModel。现有主 spec 还要求首屏加载时保留返回能力与基本详情页结构，因此本次重构不能退回成全屏转圈。

## Goals / Non-Goals

**Goals:**
- 将歌手百科的数据源从 `/artist/desc` 调整为 `/ugc/artist/get`，并明确登录失败、接口失败与无百科内容时的稳定降级行为。
- 在歌手详情头部接入 `/artist/detail/dynamic` 与 `/artist/follow/count`，增强是否关注、视频数、粉丝数等信息，同时保证这些增强字段失败时不拖垮基础头部。
- 在歌手详情页中加入歌手专辑列表区块，使用 `/artist/album` 的 `limit/offset` 语义支持分页加载与续页重试。
- 重构歌手详情页的页面结构，让头部、百科、热门歌曲、专辑列表都能对齐新的 HTML UI 参考，同时保留现有播放与路由语义。
- 明确粉丝数展示规则：`fansCount >= 10000` 时按 `10000` 换算并展示 `w` 单位，低于 `10000` 时展示具体数值。
- 明确 UI 硬约束：头图全宽正方形且无圆角、简介预览两行、颜色走 theme roles、列表 item 为小间距卡片、百科 tab 视觉尽量填满以避免回弹、collapsed top bar 与 sticky tabs 分离职责并与状态栏安全区正确避让、顶部渐变/标题淡入随滚动连续过渡，并且切换 tab 时保持外层滚动位置稳定、各 tab 保留各自独立内容位置。
- 补齐主 spec 中“加载态仍保留详情页壳层和返回能力”的差距，而不是继续使用全屏 loading。
- 为后续 multiagent 并行实现预留清晰的写入边界，让数据层、状态层、UI 层和接线层可以尽量独立推进。

**Non-Goals:**
- 本次不改动搜索、首页、个人中心到歌手详情页的导航协议；入口仍然继续传递稳定 `artistId`。
- 本次不把歌手详情页扩展成完整作品中心，不补 MV、全部歌曲、收藏、评论、分享等二级能力；头部里的“是否关注”仅限展示和状态承载，不在本 change 中扩展关注操作闭环。
- 本次不引入新的全局导航框架、缓存层或数据库持久化。
- 本次不改变“热门歌曲点击/全部播放会以当前歌手上下文替换现有队列”的既有业务语义。

## Decisions

### 1. 保留“Activity + ViewModel + Repository”总体结构，但把 `feature/artist` 内部边界拆细

现有导航与宿主接线已经稳定，因此本次不把歌手详情页迁出 `app`，也不改成新的导航宿主。重构重点放在 `feature/artist` 包内，把目前偏大的单文件拆成更清晰的内部边界：

- `ArtistDetailActivity` 只保留 Activity 宿主、`artistId` 读取和 `setContent` 装配；
- 页面渲染拆到独立的 screen/section 级文件，避免继续把所有 UI 都堆在一个 Activity 文件中；
- 数据层至少拆出“模型/远端数据源/mapper/repository”几个稳定边界，避免一个文件同时承载所有职责。

这样做的原因是：
- 现有路由、DI 和播放跳转并不是本次问题核心，改动它们只会扩大风险；
- 用户已经明确后续希望通过 multiagent 加快实现，内部边界不拆细就很难做到真正不重叠写入；
- 这与仓库里“UI 无状态、逻辑收敛到 ViewModel / repository”的已有模式一致。

替代方案是继续在现有两个大文件上增量堆功能，但那会让专辑分页、百科源迁移和 UI 重构全都挤在同一处，后续并行成本更高，因此放弃。

### 2. 头部摘要继续依赖 `/artist/detail`，百科正文改为 `/ugc/artist/get`，不再以 `/artist/desc` 作为主来源

本次把百科能力拆成两层：

- 头部摘要仍优先使用 `/artist/detail` 返回的基础字段，例如名称、头像、作品数和可直接展示的简要说明；
- 百科分段正文改由 `/ugc/artist/get` 提供，作为“歌手百科区块”和“全文浮层”的主要数据来源；
- 当 `/ugc/artist/get` 因登录态缺失、接口失败或返回空内容不可用时，页面继续保留头部主体和热门歌曲/专辑区块，百科区域降级为稳定空态或提示态，而不是拖垮整页。

这样做的原因是：
- `/ugc/artist/get` 是用户给出的新目标接口，并且它天然携带“需登录”的边界；
- `/artist/detail` 仍然是头部主体最稳定的数据源，不适合被百科接口替代；
- 把“头部摘要”和“百科正文”拆开后，可以更自然地满足 spec 对局部失败不阻塞整页的要求。

替代方案是对 `/ugc/artist/get` 失败时自动回退到 `/artist/desc`。这会降低短期风险，但会让数据语义重新混回旧接口，也会让测试边界更模糊，因此本次不把 `/artist/desc` 作为正式兜底源，只保留 `/artist/detail` 的摘要作为主体信息。

### 3. ViewModel 从“三段状态”扩展为“四段状态”，并把页级失败与区块级失败明确区分

当前 `ArtistDetailViewModel` 已经有 `headerState / encyclopediaState / hotSongsState`。本次新增专辑区块后，状态模型扩展为：

- `headerState`：歌手头部主体信息，是页面可用性的主判定来源；
- `encyclopediaState`：百科正文区块，支持 `loading / content / empty / error`；
- `hotSongsState`：热门歌曲区块，继续保留现有 `loading / content / empty / error`；
- `albumsState`：专辑区块，除 `loading / content / empty / error` 外，还要覆盖“正在续页”和“尾部续页失败”的分页状态。

重试语义也随之明确：

- 当 `headerState` 失败时，页面进入页级失败态，提供统一重试入口；
- 当 `encyclopediaState`、`hotSongsState` 或 `albumsState` 单独失败时，不回退头部主体，而是在各自区块内提供重试；
- `retry()` 需要区分“整页重试”和“分区重试”，避免已经成功的内容被无意义重置。

这样做的原因是：
- 现有 spec 已经要求百科和热门歌曲失败时保留主体内容，专辑区块也应该沿用相同模式；
- 专辑分页天然是分区状态，不能继续套用“一次性全量成功/失败”的模型；
- 明确区分页级状态和区块状态后，Robolectric 与 ViewModel 单测都更容易写。

替代方案是把所有内容折成一个总状态对象统一刷新，但这会让分页、局部失败和多接口并发加载互相牵制，因此不采用。

### 4. `/artist/detail/dynamic` 与 `/artist/follow/count` 作为头部增强软依赖接入，不改变基础头部主来源

头部增强信息与基础头部信息分层处理：

- `/artist/detail` 继续作为头部主来源，负责名称、头像、基础描述等必需字段；
- `/artist/detail/dynamic` 提供是否关注、视频数等增强字段；
- `/artist/follow/count` 提供粉丝数；
- 上述两个增强接口都属于软依赖，任何一个失败、超时或返回空值时，页面都继续展示基础头部，而不是把 header 打成失败态。

同时，头部展示规则要明确：

- `fansCount >= 10000` 时展示按 `10000` 换算后的 `w` 单位结果；
- `fansCount < 10000` 时展示原始具体数值；
- 这个格式化规则属于展示层契约，需要在明确的 formatter 边界中集中处理，避免散落在多个 Composable 中；是否放在 ViewModel state 或头部展示层 helper，不改变该规则本身。

这样做的原因是：
- 用户已经把这些字段加入当前范围，但又明确它们不影响基础详情承载；
- 软依赖建模能避免新增两个接口后，header 可用性反而变差；
- 粉丝数字格式规则一旦散落在 UI 组件中，后续很难统一维护和测试。

替代方案是把这些接口并入 header 的单一成功条件。这会让“增强信息缺失”错误升级成“整页头部失败”，与用户给出的软依赖边界冲突，因此不采用。

### 5. 专辑列表作为独立区块承载，按 `/artist/album` 的分页语义前进，不与热门歌曲混成单一列表

歌手专辑不是热门歌曲的附属信息，而是新的详情区块。因此页面结构保持“头部 -> 百科 -> 热门歌曲 -> 专辑列表”或以 HTML 参考要求调整顺序，但专辑数据模型必须独立于热门歌曲：

- repository 新增 `fetchArtistAlbums(artistId, limit, offset)`；
- mapper 输出面向 UI 的 `ArtistAlbumRow` 与分页结果对象，而不是把专辑 JSON 直接丢给 UI；
- ViewModel 维护首屏加载和续页加载，两者对 UI 是不同状态；
- UI 以 `LazyColumn` 中的独立 section 呈现专辑卡片/行，并在接近尾部时触发 `loadMoreAlbums()`。

这样做的原因是：
- `/artist/album` 自带 `limit/offset`，本身就是分页接口，不应被当作一次性附加字段；
- 把专辑区块独立后，后续若用户继续要求“查看全部专辑”或“专辑详情联动”，也更容易扩展；
- 专辑和热门歌曲如果共用一个混合列表，播放语义、测试语义和 section UI 都会被污染。

替代方案是首版只拉第一页专辑且不做续页，但这会浪费接口本身的分页能力，也很难支撑“详情页继续浏览作品”的目标，因此本次直接把分页能力设计进去。

### 6. 歌手详情页加载态也走详情骨架，顶部 compact header 与 sticky tab 分离职责，tab 切换按吸顶状态决定是否复位

当前实现中，`headerState = Loading` 直接渲染全屏转圈，和主 spec 要求不一致。本次统一为：

- 无论是首次加载、局部失败还是内容成功，都保留 `MusicDetailScaffold` 或等价详情壳层；
- 返回按钮位于外层固定层，不随正文滚动消失；
- 头部和区块在 loading 时使用骨架/占位结构，而不是整页空白；
- 外层 compact top bar 只负责状态栏渐变、返回/顶部操作与居中标题淡入；sticky header 只负责 tab 本体与内容入口，不再通过重复状态栏 padding 或额外顶部占位与外层头部叠加；
- 顶部渐变、居中标题淡入、头部大标题/按钮淡出都由同一滚动进度连续驱动，不允许出现 0/1 式的硬切过渡；
- tab 内容与 sticky header 共享同一条 `LazyColumn` 时，切换 tab 只切换 panel 与 pager，不修改外层滚动位置；各 tab 分别维护自己的内容滚动状态，避免把上一 tab 的偏移错误应用到当前 tab；
- 页面保持自然纵向滚动，不为了让更多区块首屏可见而强行压缩头像、文案或区块间距；
- HTML 参考如果对头部层次、图片区、信息块顺序有更明确要求，后续以它为视觉对照调整细节，但不改变这套结构原则。

这样做的原因是：
- 这是当前代码和主 spec 之间最明确的一处行为差距；
- 固定返回能力和详情壳层保留后，即使登录态或网络抖动导致区块加载较慢，页面也不会显得“没进去”；
- 把 compact top bar 与 sticky tabs 的职责彻底拆开后，才能避免标题、tab、状态栏三者互相覆盖，并把你给出的三阶段截图过渡落成稳定实现；
- 这也更适合 UI 多 agent 并行拆分，因为壳层和分区内容可以分开开发与验证。

替代方案是仅在成功态使用 scaffold，在 loading/error 态继续单独渲染，这会让三种状态拥有完全不同的布局骨架，增加 UI 复杂度，因此不采用。

### 7. HTML 仅作为视觉参考，颜色优先走 theme color roles，不做 1:1 像素复刻

本次视觉对齐遵循“参考气质和层次，不绑定静态截图”的原则：

- HTML 文件只用于确认视觉方向、区块层级和氛围感，不作为 1:1 布局或逐像素复刻约束；
- 头图保持全宽正方形且无圆角承载，作为详情页顶部主视觉区，而不是回退为小头像居中卡片；
- 页面主体继续以可滚动详情页为前提，允许用户通过纵向滚动浏览头部、简介、热门歌曲和专辑区块；
- 歌手简介卡片中的摘要预览固定为最多两行，完整正文继续放在可滚动浮层中承载；
- 背景、卡片、文案、按钮、占位与图标颜色优先使用 `MaterialTheme.colorScheme` 中的 `primary`、`secondary`、`surface`、`background` 以及对应 `on*` roles；
- 热门歌曲、专辑等列表 item 采用小间距卡片式组织，保持内容密度但不做拥挤贴边排布；
- 百科 tab 或其等价内容承载区域在视觉上尽量填满所在 section，避免出现明显的上浮留白和滚动回弹观感；
- sticky tab、头部固定层、返回入口和状态栏安全区必须正确协同，避免内容或交互元素卡进状态栏；
- 只有头部氛围层这类确实依赖主视觉联动的场景，才允许在 theme roles 基础上做动态渐变或透明度变化，而不是直接写死颜色值。
- 与这些 UI 约束配套的测试，应优先锁定用户可见行为和内容切换结果，而不是把内部 spacer、占位高度或过渡层级 tag 直接固化成产品契约。

这样做的原因是：
- 不同屏幕高度、状态栏和字体缩放下，首屏可见区域并不稳定，强行把所有关键信息塞进首屏会直接伤害可读性；
- 这个 change 关注的是详情页体验与结构升级，不是把参考 HTML 机械翻译成 Compose 像素图或逐像素复刻页面；
- 主题色 roles 能保证深浅层次、文本对比度和后续主题演进的一致性，避免页面颜色策略游离于现有设计系统之外。

替代方案是为了追求截图式首屏效果去压缩内容密度，或者在 section 中散落硬编码颜色。这会让不同设备上的阅读体验和主题一致性明显变差，因此不采用。

### 8. 为 multiagent 实现预先定义四类不重叠写入范围

为了兑现你后续“全部采用 gpt-5.4 multiagent 加快实现”的要求，本次设计直接把实现切面定义清楚：

- 数据层工作流：`ArtistDetailRepository`、远端数据源、mapper 以及对应 repository tests；
- 状态层工作流：`ArtistDetailViewModel` 与 ViewModel tests；
- UI 工作流：歌手详情 screen/section 文件与 Robolectric screen tests；
- 宿主接线工作流：`AppContainer`、必要的 manifest/intent/route 接线与入口验证。

其中最重要的约束是：
- `/ugc/artist/get` 与 `/artist/album` 都会触及数据层，因此不要把它们拆给两个同时写同一 repository 文件的 agent；
- 先把数据层边界拆清，再让状态层和 UI 层并行接入新状态；
- 接线工作流只在确实需要新增依赖或改 Activity 注册时触碰宿主文件，避免无意义冲突。

这样做的原因是：
- 当前 `feature/artist` 的改动天然横跨数据、状态、UI 三层，不先定边界就会在并行开发时互相抢文件；
- 先从设计阶段规定写入范围，比实现时临时协调更稳。

## Risks / Trade-offs

- `[百科接口依赖登录态，未登录时内容可能明显减少]` -> 头部主体继续依赖 `/artist/detail`，百科区块降级为空态或提示态，不影响整页可用性。
- `[专辑分页会增加 ViewModel 和 UI 状态复杂度]` -> 把专辑区块做成独立状态机，并把“首屏失败”和“续页失败”分开建模，避免和热门歌曲状态混用。
- `[HTML 参考可能与当前 Compose 结构存在局部差异]` -> 先以滚动壳层、全宽正方形且无圆角头图、两行简介预览、theme color roles、tab 切换滚动复位和安全区避让作为硬约束，再把 HTML 当作视觉参考做局部校准，不阻塞当前工件推进。
- `[固定头部与 sticky tab 容易再次通过叠加占位方式实现，导致状态栏/标题/tab 互相覆盖]` -> 明确 compact top bar 只承载状态栏渐变、标题和顶部操作，sticky header 只承载 tabs 与内容入口，二者共享单一顶部高度来源而不是各自重复占位。
- `[UI 回归测试若继续绑定内部 spacer 或过渡层 tag，会把错误布局固化成正确行为]` -> 调整测试口径，以当前 tab 内容、简介卡片可见性、旧内容隐藏和各 tab 独立滚动位置保持为主，减少对内部结构 tag 的依赖。
- `[头部增强接口失败会造成信息不完整]` -> 将 `/artist/detail/dynamic` 与 `/artist/follow/count` 视为软依赖，只降级增强字段，不影响基础 header。
- `[粉丝数展示规则容易在不同组件中写出不同格式]` -> 统一收敛到页面模型或独立 formatter，补充针对 `10000` 阈值的测试样例。
- `[数据层和 UI 层同时引入新能力，容易在单文件中产生冲突]` -> 通过前置拆分文件边界和 multiagent 写入范围，尽量减少并行编辑碰撞。
- `[百科从旧接口迁到新接口后，测试样例和 JSON 字段结构会整体变化]` -> repository tests 先锁住新接口映射，再让 ViewModel/UI 基于稳定页面模型推进。

## Migration Plan

1. 先更新 `artist-detail-page` 的 delta spec，把百科数据源切换、专辑分页区块和加载态壳层保留这些需求写清楚。
2. 实现数据层重构：拆分 repository 相关文件，接入 `/ugc/artist/get`、`/artist/album`、`/artist/detail/dynamic` 和 `/artist/follow/count`，补齐映射测试并明确头部增强软依赖的降级边界。
3. 实现状态层重构：在 ViewModel 中加入 `albumsState`，梳理整页重试、分区重试与续页触发，同时把头部增强字段格式化与降级策略收敛到稳定状态模型。
4. 实现 UI 重构：把 Activity 宿主与 screen/section 组合拆开，对齐 HTML 参考，补上 loading scaffold、百科区块和专辑区块，并落实全宽正方形且无圆角头图、小间距卡片、百科承载填满、compact top bar 与 sticky tabs 分离职责、顶部连续渐变过渡、tab 切换滚动复位和安全区避让约束。
5. 仅在需要时调整 `AppContainer` 或其他宿主接线，确保现有 artist detail 入口不回归。
6. 验证顺序优先采用 repository tests -> ViewModel tests -> Robolectric screen tests -> `:app:testDebugUnitTest`，最后再进入更大范围验证。

回滚策略：
- 如果新百科接口或专辑分页接入导致页面稳定性明显下降，可先保留 UI 骨架重构与状态拆分，临时关闭专辑区块展示，避免整页回退。
- 如果加载态骨架或分区渲染造成 UI 回归，可先保留数据层与状态层改动，局部回退具体 section UI，而不撤销整个接口迁移。
