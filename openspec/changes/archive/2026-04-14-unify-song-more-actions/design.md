## Context

当前仓库里单曲次级入口已经出现了多种形态：首页歌曲卡使用真正的更多菜单，本地歌曲列表使用独立详情按钮，每日推荐歌曲只有整行播放而缺少次级入口，搜索结果页则已经收口为“整行播放 + 三点直达详情”。这导致用户面对同样的歌曲对象时，对“更多”入口的预期不一致。此次改动范围覆盖首页歌曲、本地歌曲和每日推荐歌曲，三处都已经具备歌曲详情页、下一首播放和基础跳转能力，适合在不改搜索交互的前提下先收口为统一 contract。

## Goals / Non-Goals

**Goals:**
- 定义统一的单曲“更多”动作 contract，约束最小动作集合与按来源降级的规则。
- 让首页歌曲、本地歌曲与每日推荐歌曲都通过一致的三点更多入口承载次级动作。
- 让首页歌曲更多菜单补齐“查看歌曲详情”，并保留“下一首播放 / 查看歌手 / 查看专辑”。
- 让本地歌曲列表去掉单独的详情按钮，改成统一的三点更多入口，并提供“下一首播放 / 查看歌曲详情”。
- 让每日推荐歌曲列表在保留整行播放语义的前提下，通过三点更多入口提供“下一首播放 / 查看歌曲详情 / 查看歌手 / 查看专辑”。

**Non-Goals:**
- 不修改搜索结果页的主点击与三点行为。
- 不在本次统一“收藏 / 分享”等依赖登录态或来源差异更大的动作。
- 不扩展到最近播放、播放队列或播放器全局更多菜单。

## Decisions

### 决策一：新增 `song-overflow-actions` capability，而不是只在各页面 spec 各写一遍

首页和本地歌曲都需要同一套动作集合与降级规则。如果只在各页面 spec 中分别描述，后续再次扩展到每日推荐或播放队列时会继续复制粘贴。新增独立 capability 可以让“更多”动作 contract 成为稳定的共享约束，同时让各页面 spec 只负责描述自己如何承载这套动作。

### 决策二：搜索结果页不并入统一更多菜单

搜索结果页刚刚收口为“整行播放 + 三点直达歌曲详情”，这是用户已确认更连贯的交互。若本次把搜索三点再改成统一菜单，会立刻打破刚确认的体验，并把“详情”退回次级菜单层。为避免 scope 反复摇摆，本次明确把搜索排除在统一更多菜单之外。

### 决策三：本地歌曲的更多菜单按来源降级，不伪造在线动作

本地歌曲没有稳定的在线歌手 id、专辑 id，也不应该为了统一外观伪造“查看歌手 / 查看专辑”。因此本地歌曲更多菜单仅展示当前可靠可执行的动作：`下一首播放` 与 `查看歌曲详情`。在线歌曲则可在具备稳定目标 id 时继续展示歌手与专辑跳转。

### 决策四：本地歌曲详情入口改为三点更多，而不是保留 Info 图标

独立的详情 Info 图标和首页的三点更多入口属于两套交互体系，继续并存只会强化不一致。改为统一三点更多后，用户会把“单曲次级动作”理解为同一种交互，而不是把本地歌曲视为一套特殊列表。

### 决策五：每日推荐继续保持整行播放，三点更多只承载次级动作

每日推荐的主价值仍然是“推荐即播放”，并且当前页面已经围绕点击歌曲直接开始播放形成稳定心智。如果把整行点击改成详情，会破坏此前已经确认的主路径。因此每日推荐只补三点更多，不改变整行播放闭环。

## Risks / Trade-offs

- [风险] 首页和本地歌曲的动作排序不一致会让统一 contract 失效。 → 通过共享动作定义和定向测试锁住顺序。
- [风险] 本地歌曲新增更多菜单后，用户可能觉得多点了一次才能进详情。 → 保留主点击直接播放，把详情收进次级动作，符合“本地曲库列表主路径仍是播放”的定位。
- [风险] 每日推荐新增更多按钮后，歌曲行信息密度可能被挤压。 → 保持主信息区不变，只增加轻量三点入口并用 UI 测试锁住布局。
- [风险] 以后若扩展到最近播放或播放队列，动作能力可能继续分叉。 → 本次先把 capability 独立出来，后续扩展直接复用同一 contract。
