## Context

详情页播放不是“补一个按钮”这么简单。用户真正要的是：从歌单、歌手、专辑详情页发起播放时，当前播放列表要切换成对应上下文，而不是退化成单曲播放；同时大列表不能因为等完整元数据而拖慢首播。因此这部分值得独立成 change，而不是继续藏在在线播放或 UI change 里。

## Decisions

1. 详情页统一通过共享播放网关发起播放。
   - 歌单、歌手、专辑详情页都不直接复制 `PlayerViewModel` 的同步逻辑。
   - 统一走“替换当前队列 + 指定激活项 + 启动播放”的共享网关。
   - 只有当共享播放网关确认已成功替换当前播放列表或切换到目标激活项后，详情页才自动打开播放器展开页。

2. 详情页播放采用“两段式装配”。
   - 第一段：只要条目具备稳定 `songId`，就允许立即建立播放队列并开始播放。
   - 第二段：后台分页请求 `/song/detail`，按稳定 item id 异步补齐标题、歌手、专辑、封面与时长。
   - `/song/detail` 异步 enrichment 额外补齐稳定 `artistId`，供播放页歌手名跳转详情时复用，而不是再单独查询一次当前曲目详情。

3. 当前上下文已相同时只切换激活项。
   - 如果当前播放列表和详情页发起播放的上下文拥有相同稳定条目序列，则不重建整条队列。
   - 仅切到目标 index 并立即开始播放。

4. 歌单动态信息与播放量更新独立承载。
   - `/playlist/detail/dynamic` 失败不影响头部和歌曲列表主内容。
   - 播放量更新只在真实播放已成功启动后触发。

## Verification

- 覆盖歌单、歌手、专辑详情页的播放全部与点歌播放
- 覆盖成功替换播放列表后自动打开播放器展开页，失败时不跳转
- 覆盖同上下文只切激活项、不重复重建队列
- 覆盖异步 enrichment、稳定 `artistId` 回填、动态信息失败重试与播放量更新时机
