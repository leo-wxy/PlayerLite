## Context

当前歌单详情页已经具备 `headerState + tracksState` 的基础承载能力，`PlaylistDetailRepository` 也已经接通了 `/playlist/detail` 与 `/playlist/track/all`。但它仍然停留在“展示详情”的阶段：页面没有“播放全部”入口，歌曲行点击也不会把当前歌单装配成真正的播放列表，更无法把“在当前歌单中指定播放某首歌”的语义传到现有播放运行时。歌手详情页同样还没有“全部播放”入口，因此也无法把当前歌手热门歌曲集合直接替换进现有播放列表。专辑详情页虽然已经有歌曲列表和动态信息承载，但同样还没有把“播放全部 / 点击歌曲播放”接入现有播放列表替换语义。

现有播放链路的核心假设仍然是“队列项已经有可直接播放的 URI”。`PlaylistItem` 目前只保存 `id + uri + displayName + songId`，`PlayerViewModel.syncQueueToPlaybackProcess()` 也会直接把 `PlaylistItem.uri` 映射成 `MusicInfo.playbackUri` 后同步到 `playback-service`。这对本地文件、`content://`、固定 `http(s)` 资源成立，但对当前歌单详情场景并不成立，因为详情接口能稳定给到的是歌曲 id，而不是长期有效的音乐 URL。

用户最新的边界也把模型分层说得更清楚了：
- 在线歌曲可以有语义更完整的 `MusicInfo`。
- 本地播放后续可能需要 `LocalMusicInfo` 这类本地领域模型。
- 播放进程真正需要的并不是“完整 `MusicInfo`”，而是一份只包含可播放所需字段的最小基类 / 投影契约。

因此这次变化不能再简单理解成“让所有播放列表项都直接等于 `MusicInfo`”，而应该是“上层领域模型收敛命名，底层播放进程消费统一最小可播放基类”。

缓存、时长和主视觉链路也暴露出几个当前模型的缺口：
- `cache-core` 已经能够通过 `lookup` 暴露 `contentLength`、`durationMs` 等缓存元信息，但现有在线播放准备流程还没有把“缓存是否完整”作为第一层决策。
- 歌单详情接口返回的曲目本身已经带有 `durationMs`，但 `MusicInfo` 还没有承接这个字段，`TrackPreparationCoordinator` 仍然倾向于通过 FFmpeg 元信息探测拿时长，导致在线播放首播路径做了不必要的额外工作。
- 详情页和曲目行里已经有稳定的 `coverUrl` / `picUrl`，但播放队列共享结构没有保留这些信息，导致播放页如果想展示当前曲目封面，还需要额外追一层数据。
- 实际接口差异说明，仅依赖歌单/专辑/歌手各自详情接口返回的曲目行字段并不稳定：有的链路能稳定拿到封面，有的链路只能稳定拿到 `songId + title + artist + duration`，这会让不同详情页进入播放后的封面与主信息表现不一致。
- 参考文档里的原始字段命名（例如 `ar`、`al`、`dt`）对播放层来说过于抽象，如果直接把这些名字带进 `MusicInfo`，后续 app、播放服务和测试代码都会继续围绕接口细节耦合，而不是围绕播放器自己的领域语义。

用户这轮补充把问题边界拉得更清楚了：
- 在线歌曲需要通过 `/song/url` 或 `/song/url/v1` 按 song id 解析真实播放 URL，未登录或非会员时可能返回试听片段而不是完整音源。
- 歌单详情页还可以通过 `/playlist/detail/dynamic` 补齐评论数、收藏状态、播放数，并在合适时机用 `/playlist/update/playcount` 更新播放量。
- 403/referrer 的备注更偏 Web 直链场景，原生链路不能机械照搬 `<meta name="referrer">`，而应该在“是否使用 302 浏览器式直链”和“请求头怎么构造”上做正确选择。

仓库现状也给出几个设计约束：
- 播放列表语义属于 `app` 业务层，`playback-service` 负责播放执行与 `MediaSession` 映射，不应成为歌单业务和用户会话的真相源。
- `JsonHttpClient` 已经支持 `requiresAuth` 与额外 headers，`TrackPreparationCoordinator` 也已经支持受保护网络源的 `requestHeaders`，因此可以复用现有网络与服务层能力，而不是重新引入一条旁路。
- `PlayerRuntimeRegistry` 提供了进程内共享运行时，但歌单详情页当前没有接入一个稳定的“从详情页发起播放”的网关，因此不能简单依赖 `PlayerViewModel` 生命周期去偷用播放器逻辑。

## Goals / Non-Goals

**Goals:**
- 让歌单详情页提供“播放全部”入口，并让点击某首歌具备“以当前歌单为播放列表、以该歌曲为激活项开始播放”的语义。
- 让歌手详情页提供“播放全部”入口，并让该入口与歌曲行点击都以当前歌手热门歌曲集合作为新的播放列表整体替换当前队列。
- 让专辑详情页提供“播放全部”入口，并让该入口与歌曲行点击都以当前专辑详情页已加载曲目集合作为新的播放列表整体替换当前队列。
- 在不破坏现有播放模式、切歌、删除能力边界的前提下，为播放列表新增“从外部歌单整体替换当前队列”的入口。
- 让在线歌单歌曲以 song id 为稳定身份进入队列，而不是把临时解析出的播放 URL 当作长期真相。
- 扩展 `MusicInfo` 共享契约，使 app 与 `playback-service` 能共同表达在线歌曲身份、歌单来源上下文、请求头、时长、封面和试听片段信息。
- 为在线歌曲建立可复用的 URL 解析链路，优先支持歌单详情触发播放，并能在登录态、音质等级、403/无版权/试听片段等场景下给出稳定结果。
- 为详情页进入播放的元数据装配建立统一链路：以 `songId` 作为最低可播放前提，允许先快速设置播放列表并开始播放，再通过 `/song/detail` 分页异步补齐封面、标题、歌手、专辑与时长。
- 在真正解析拉流地址之前先做缓存完整性判断；已有完整缓存时直接走缓存，不再重复请求在线地址。
- 在缓存不完整时，对最近成功解析过的少量完整音源拉流地址做内存缓存，减少切回上一首时重复请求 `/song/url*`；试听解析结果不进入该缓存，避免挤占 10 首 URL 复用容量。
- 为歌单详情页补充动态信息加载能力，并将播放量更新入口收口到真实播放开始后的合适时机。

**Non-Goals:**
- 本次不实现歌单评论列表、收藏/取消收藏、分享、完整专辑/歌手联动播放等二级能力。
- 本次不新增全局音质选择 UI；首版只在内部确定默认解析等级和必要的回退顺序。
- 本次不采用 `/song/url/v1/302` 浏览器式重定向直链作为主方案，也不把 Web `<head meta>` 方案硬套到原生页面。
- 本次不把用户会话真相源迁入 `playback-service`，也不让详情页直接依赖 `PlayerViewModel` 实例完成播放。
- 本次不做整页预解析所有歌曲 URL 后再开始播放，避免长歌单放大首播等待。

## Decisions

1. 详情页播放入口通过共享的 app 侧播放网关进入现有运行时，而不是在 `PlaylistDetailActivity`、`ArtistDetailActivity`、`AlbumDetailActivity` 中直接复制 `PlayerViewModel` 的 `syncQueue` 逻辑。
   - 方案：在 `app` 的播放器运行时层新增一个可被详情页复用的播放网关，负责接收“详情上下文 + 曲目列表 + 激活项索引”的请求，统一完成队列替换、激活项设置、服务连接与播放启动。`PlaylistDetailViewModel`、`ArtistDetailViewModel` 和 `AlbumDetailViewModel` 只负责发出播放意图，不直接持有 `PlayerServiceBridge` 或复制 `PlayerViewModel` 的同步逻辑。
   - 原因：歌单详情页、歌手详情页和专辑详情页都是独立 Activity，直接向 `PlayerViewModel` 借逻辑会把页面生命周期耦合到播放器页面；如果在各自 feature 再复制一套 `syncQueue + statusText` 逻辑，后续播放模式、状态提示和失败语义又会分叉。共享网关可以继续复用 `PlayerRuntimeRegistry`、`PlaylistSessionCoordinator` 和 `PlayerServiceBridge`，同时保持宿主边界清晰。
   - 备选方案：
     - 在 `PlaylistDetailActivity` 中直接 new `PlayerServiceBridge` 并同步队列：实现快，但会复制运行时和状态提示逻辑。
     - 让详情页跳回主播放器页，再由 `PlayerViewModel` 接手：用户体验不自然，而且仍需要一条跨页面参数通道。

1.1 详情页播放入口统一采用“按 `songId` 立即入队 + `/song/detail` 分页异步回填元数据”的两段式装配，而不是等待全部详情 enrichment 完成后才开始播放。
   - 方案：`RuntimeDetailPlaybackGateway` 接到详情页播放请求后，先使用当前页面已知字段构建最小 `PlaylistItem` 列表；只要在线条目具备 `songId`，就允许立即替换当前播放列表并开始播放。随后网关在后台以分页方式请求 `/song/detail?ids=...`，将返回的 `name / ar / al / dt / picUrl` 等字段映射为统一 `MusicInfo` 语义，再按 item id 回填到当前播放列表。优先先回填当前激活曲目所在页，其余歌曲继续按页补齐，避免大列表阻塞首播。
   - 原因：用户已经明确“在线播放只要有 id 就能满足基本要求”，因此封面、标题等元数据不应再成为开播前置条件。同时，不同详情页接口返回的曲目字段稳定性不一致，只有把 `/song/detail` 作为统一 enrichment 源，才能让歌单、歌手、专辑三条入口进入播放后的元数据表现一致。
   - 备选方案：
     - 每个详情页在点击播放前自己先批量请求完整 `/song/detail` 再入队：实现分散，且长列表会显著拉长首播等待。
     - 完全不做异步 enrichment，只依赖页面已有字段：实现最省，但会继续保留“歌单链路正常、专辑链路缺封面”这类源数据差异。

2. 领域层与播放进程层分离：`MusicInfo` / `LocalMusicInfo` 负责承载上层语义，播放进程只消费最小可播放基类。
   - 方案：把模型分成两层：
     - 上层领域模型：在线歌曲使用语义更完整的 `MusicInfo`，本地播放预留 `LocalMusicInfo` 或等价本地条目模型。
     - 下层可播放契约：播放进程只消费统一的最小 `PlayableItem` 基类/接口，其中只保留真正用于播放和当前曲目投影的公共字段。
     `MusicInfo` 继续承载在线歌曲需要的稳定 `songId`、`durationMs`、`coverUrl`、歌单来源上下文、解析后的 URL / 请求头等信息；`LocalMusicInfo` 则可以只关心本地文件或 `content://` 所需字段。两者最终都通过 mapper 投影成同一种可播放基类，再同步给播放进程。
     `MusicInfo` 的字段命名不直接复用接口里的 `ar`、`al`、`dt` 等原始缩写，而是收口为播放器领域里的可读命名，例如 `title`、`artistNames`、`albumTitle`、`coverUrl`、`durationMs`，并通过 mapper 明确完成从 `/song/detail` 原始字段到领域结构、再到可播放基类的转换。
   - 原因：歌单歌曲 URL 是按接口实时解析出来的，持久化一个临时 URL 极易过期；而当前的 `PlaylistItem.uri` / `MusicInfo.playbackUri` 模型默认所有条目都等同于“长期可播直链”，这与在线歌单场景冲突。与此同时，本地播放并不需要完整在线 `MusicInfo` 的全部字段。如果强行让播放进程依赖完整 `MusicInfo`，后续引入 `LocalMusicInfo` 时仍会被在线字段绑住。把“领域信息”和“最小可播放信息”拆开，才能同时兼容在线歌单、本地播放、状态恢复、后台播放、“无需再用 FFmpeg 额外探时长”，以及播放页直接展示当前曲目封面的要求。
   - 备选方案：
      - 直接把解析后的 URL 落进 `PlaylistState`：实现简单，但重启后、隔段时间后或跨会话恢复时容易拿到失效 URL。
      - 保持 `durationMs` 只停留在详情页行模型里：播放器与缓存链路仍拿不到时长提示，FFmpeg 探测成本不会下降。
      - 保持封面信息只停留在详情页或搜索结果模型里：播放页切歌后仍需要额外补请求或回查数据源，无法保证主视觉稳定。
      - 直接沿用接口文档里的抽象字段名设计 `MusicInfo`：虽然 mapper 更少，但会把外部接口细节扩散到播放器内部。
      - 让播放进程直接依赖完整 `MusicInfo`：会让本地播放也被在线字段绑住，难以引入 `LocalMusicInfo` 这类本地模型。
      - 在点击播放时一次性解析整张歌单所有 URL 再建队列：长歌单首播等待太长，也会放大接口失败面。

3. 在线音乐在准备阶段先判断缓存是否完整；只有缓存不完整时才解析拉流地址并继续在线读取。
   - 方案：当 `playback-service` 准备在线歌曲时，先基于 `songId + level + clip-mode` 生成稳定 `resourceKey` 并查询缓存快照：
     - 若缓存快照显示内容已完整，则直接以缓存会话作为读取入口，不再请求 `/song/url*`。
     - 若缓存不存在或仍不完整，先查询一个仅存在于播放准备层的 LRU 内存缓存；命中且尚未过期时，直接复用最近一次解析出的 URL 与请求头。
     - 仅当磁盘缓存不完整且内存缓存 miss / expired 时，才真正进入在线 URL 解析流程。
     - URL 解析主路径使用 `/song/url/v1`，因为它能显式表达 `level`、`unblock` 等参数。
     - 当 `/song/url/v1` 返回无可用 URL、字段不完整或与当前账号/设备条件不兼容时，再按兼容策略回退到 `/song/url`。
     - 若解析结果仅包含试听片段，则允许当前播放继续使用该结果，但不把该结果写入 10 项 LRU URL 缓存，也不让其占用完整音源的复用槽位。
     - `/check/music` 不作为主播放入口，而是在解析失败时用于补一层“无版权 / 不可用”的可解释结论。
     - `/song/url/v1/302` 不作为首版主方案，因为当前原生链路更适合拿 JSON DTO 明确建模，且用户给出的 `<meta name="referrer">` 处理更偏浏览器跳转场景。
     - 内存缓存项上限固定为 10 首，key 继续使用 `songId + level + clip-mode`，value 至少包含解析后的 URL、请求头、试听片段信息，以及基于接口返回有效时长/有效期字段推导出的 `expiresAt`。
   - 原因：用户已经明确指出“是否有完整缓存”应该先于“是否获取拉流地址”判断。`cache-core` 现成的 `lookup`、`contentLength`、`durationMs` 元信息正适合承担这层前置决策。再加一个很小的内存缓存，可以显著减少“刚播过一首又切回上一首”这类高频回切场景的重复解析请求，同时又不会把临时 URL 长时间留在内存里。
   - 备选方案：
     - 每次播放在线歌曲都先解析 URL，再看缓存是否命中：会让完整缓存的场景也产生不必要的联网依赖。
     - 只看“是否有缓存文件”而不判断是否完整：容易把半截缓存误判成可直接播放的完整内容。
     - 无上限缓存所有已解析 URL：虽然命中率可能更高，但会增加内存占用，也更容易累积过期直链。

4. `MusicInfo.durationMs` 同时作为 UI 时长真相和缓存/播放准备提示，在线播放不再为了“拿时长”额外走 FFmpeg 探测。
   - 方案：详情接口返回的 `PlaylistTrackRow.durationMs` 在建队列时直接写入 `PlaylistItem` / `MusicInfo`，并沿着 MediaItem extras 传到服务侧：
     - cache session 打开时将其作为 `durationMsHint` 传入。
     - 播放准备阶段若 `MusicInfo.durationMs > 0`，则直接用该值构建时长相关的 UI/状态投影，不再为了“仅获取时长”额外执行 FFmpeg 元信息探测。
     - 只有当 `durationMs` 缺失、明显非法，或者后续真的需要 codec/sampleRate 等额外元信息时，才回退到现有的 source-level probe。
   - 原因：歌单详情接口本身已经提供时长，继续为了时长去读源并走 FFmpeg 只会增加准备成本，尤其在在线播放和缓存半命中场景下会放大首播延迟。把 `durationMs` 提前放进 `MusicInfo`，可以同时给 UI、缓存层和服务准备层提供统一提示。
   - 备选方案：
     - 继续一律从源里探测时长：实现路径最熟悉，但与当前接口数据重复，且会增加首播开销。
     - 完全不再探测任何元信息：虽然能省成本，但会把 codec/sampleRate 等后续仍可能需要的信息一并丢掉。

5. 歌单详情页在现有 `headerState + tracksState` 基础上增加独立 `dynamicState`，并仅在真实播放开始后触发播放量更新。
   - 方案：保留当前歌单头部与歌曲列表分段承载，再新增一个轻量 `dynamicState` 负责 `/playlist/detail/dynamic` 返回的评论数、收藏状态、动态播放数等信息。`/playlist/update/playcount` 不在页面进入时调用，而是在“从歌单详情页成功发起一次真实播放”后按 playlist id 做会话级去重更新。
   - 原因：动态信息不是详情页主体可用性的前置条件，不应让头部或歌曲列表为了评论数一起等待；而播放量更新更接近“播放行为回执”，若在仅打开详情页时就调用，会造成虚高计数，也不符合用户这次想补的是“歌单播放闭环”而不是“详情曝光即计播”。
   - 备选方案：
     - 把动态接口并进头部主请求：实现上更省状态，但接口慢或失败时会拖累首屏头部。
     - 每次进入详情页都更新播放量：接线简单，但语义错误。

6. 外部详情页触发的“播放全部 / 点歌播放”统一视为“替换当前播放列表为当前详情上下文”，删除与切歌继续由业务层列表状态驱动。
   - 方案：详情页播放行为统一走“replace queue with context”：
     - `播放全部` 使用当前歌单已知的完整曲目集合作为队列，并将激活项设为首曲。
     - 点击某首歌使用同一份歌单上下文替换队列，但将激活项设为被点击曲目。
     - 歌手详情页的 `全部播放` 使用当前歌手详情页已加载的热门歌曲集合作为队列，并将激活项设为首个有效曲目。
     - 专辑详情页的 `全部播放` 使用当前专辑详情页已加载的曲目集合作为队列，并将激活项设为首个有效曲目；点击曲目时则以被点击 index 作为激活项。
     - 若当前运行时中的原始队列已经与目标详情上下文拥有相同的稳定 item id 序列，则不重复替换队列，只切换激活项到目标 index 对应的曲目。
     - 队列建立后，删除、切歌、播放模式切换仍完全交给 `PlaylistSessionCoordinator` / `PlaylistController` 的业务层状态处理。
   - 原因：用户明确要求“单击某首歌曲，就是在当前歌单中指定播放某首歌”，并补充了“歌手详情 / 歌单的全部播放都直接替换当前播放列表”。这意味着详情页不是“临时单曲播放入口”，而是当前播放列表的来源之一。只要把队列替换动作收口清楚，后续删除、上一首、下一首和随机/列表模式就都能沿用现有业务层规则，而不需要在详情页自己维护第二份播放列表。
   - 备选方案：
     - 点击歌曲时仍退化为单曲播放：最省事，但与需求相反。
     - 让 `playback-service` 自己理解“歌单详情上下文”并生成列表：会把业务层队列真相源下沉到执行层。

7. 在线鉴权和 403/referrer 风险通过“原生请求头构造 + 避免浏览器式 302 跳转”处理，而不是引入 Web 页面层补丁。
   - 方案：在线播放解析链路继续复用 `UserSession.toAuthHeaders()` 产出的稳定 Cookie/CSRF 头，并在在线 URL 解析请求上增加一个播放专用 header builder，用于补充 `os=pc` 等需要的 Cookie 片段或参数。对于实际流媒体请求，`MusicInfo.requestHeaders` 会跟着队列同步到服务侧网络源；首版默认不显式设置 `Referer`，并避免使用需要浏览器 referrer 行为配合的 302 直链方案。
   - 原因：用户给出的 `<meta name="referrer" content="no-referrer">` 解决的是浏览器页面发起资源请求时的 referrer 传播，而当前原生播放链路本来就不应依赖页面 `<head>`。选择 JSON 解析接口而不是 302 跳转，可以把 URL、试听片段信息和请求头一次性收进 DTO，也更适合服务端准备阶段统一处理。
   - 备选方案：
     - 直接在原生页面或 Compose 宿主里模拟 Web meta 行为：语义不对，也难以真正约束 service 进程里的网络请求。
     - 首版直接用 302 接口把播放器指向重定向 URL：实现可能更短，但浏览器式 referrer/设备差异会更难控。

## Risks / Trade-offs

- [领域模型与最小可播放基类拆层后，mapper 数量会增加] → 把 mapper 收口在播放装配边界，避免 UI、缓存和播放进程各自做半套转换。
- [缓存完整性判断若使用了不稳定的 `resourceKey`，可能把不同音质或试听片段误当成同一缓存] → 将 `resourceKey` 固定为 `songId + level + clip-mode` 这类稳定组合，而不是使用临时 URL。
- [懒解析在线 URL 会把首曲准备延迟推到真正播放前] → 让详情页入口只负责尽快替换队列；解析成本延迟到当前曲目真正准备时承担，必要时可在首版优先预解析激活项而不是全量解析。
- [URL 内存缓存若不尊重接口返回的有效期，可能复用已经失效的直链] → 每个缓存项都绑定接口返回的有效时长/有效期字段换算出的 `expiresAt`，命中过期项时强制重新解析。
- [历史遗留的 full 磁盘缓存若曾被错误内容污染，后续会被“完整缓存命中”误复用] → 为在线缓存补充 clip-mode sidecar 标记，只有带可信 `full` 标记的完整 full 缓存才允许直接命中；旧的无标记 full 缓存一律视为不可信并重建。
- [若 `durationMs` 与真实音源时长不一致，可能导致进度条或结束判断出现轻微偏差] → 以接口时长作为首选提示值，仅在缺失或明显异常时回退到 source-level probe。
- [若队列项不保存封面信息，播放页在切歌或恢复时可能短暂丢失主视觉] → 将 `coverUrl` 直接纳入共享 DTO，并随队列、快照和恢复链路一起传递。
- [若等待全部 `/song/detail` 返回后才开始播放，大列表会明显拖慢首播] → 采用“只凭 `songId` 先入队开播，元数据后台分页回填”的两段式装配，并优先补齐当前激活曲目所在页。
- [若 `MusicInfo` 直接沿用外部接口的抽象字段命名，后续实现与测试会持续被 API 细节绑住] → 通过独立 mapper 把 `song/detail` 原始字段转换为播放器领域语义化命名。
- [若播放进程直接依赖完整在线 `MusicInfo`，本地播放模型会被迫携带无意义字段] → 让播放进程只消费最小可播放基类，在线/本地各自保留上层语义模型。
- [试听片段与完整音源共用一套播放链路，用户可能不理解为什么被截断] → 在 `MusicInfo` 中显式携带试听窗口信息，并把状态文案/失败提示留给 app 侧 UI 投影；同时不让试听结果进入 URL LRU 缓存，避免污染完整音源复用路径。
- [动态播放数和详情主接口里的播放数可能短时间不一致] → 以动态接口为优先展示值，主接口字段作为兜底；播放量更新采用会话级去重，避免频繁自增。
- [将歌单详情播放入口接到共享运行时后，若网关设计不好会再次复制 `PlayerViewModel` 逻辑] → 明确网关只暴露“替换队列并播放”的最小能力，`PlayerViewModel` 自己继续作为播放器页专用状态聚合器。
- [不同账号/设备对 `level`、会员音质、杜比/Hi-Res 的返回不一致] → 首版把解析等级做成内部配置和回退链路，不在 UI 上承诺所有高音质都能稳定可用。

## Playback Page Finalization

7. 播放页最终视觉收口采用“单条真实顶栏 + 顶部正方形封面 + 底部信息与播控分层”方案，而不是继续叠加壳层顶栏和页面内标题。
   - 方案：播放页由 `PlayerScreen` 自己承载完整顶栏，返回按钮、标题“歌曲”和分享按钮位于同一条安全区基线；外层 `PlayerExpandedScreen` 在播放器展开态不再额外叠加一层顶部按钮。封面维持顶部大尺寸正方形展示，缓冲/状态提示固定锚定在封面右下角，底部信息区与进度/播控区保持稳定位置，不再因为状态文案切换而整体抖动。
   - 原因：此前的双层 chrome 会造成状态栏重叠、标题无法和返回/分享对齐，以及按钮与封面互相压占。把顶栏收口到同一层后，视觉关系和安全区处理都更稳定，也更贴近用户确认过的参考方向。
   - 备选方案：
     - 继续保留外层悬浮返回/分享，再靠页面内 `padding` 调整标题：实现改动小，但很难彻底消除多层 chrome 叠压问题。
     - 把标题移出播放器内容层，仅由壳层标题栏承载：壳层职责会重新变重，且后续歌词页切换时仍会遇到布局耦合。

8. Seek / buffering 时采用“明确缓冲态 + 保持关键播控可操作”的交互，而不是把整排播控锁死成灰态。
   - 方案：进度拖动后页面继续进入明确的 buffering / preparing 态并冻结进度，但上一首、下一首、播放模式、播放列表这些关键播控仍保持可点；主播放按钮仍按当前播放/暂停语义响应，侧边四个按钮扩大点击区并使用稳定背景承载，以保证小屏设备上可识别性和可操作性。
   - 原因：用户已经明确指出“缓冲中不允许做任何操作”过于僵硬。对于在线播放，seek 后可能需要重新取流，但这不应阻断用户继续切歌、换模式或打开播放列表排查问题。
   - 备选方案：
     - buffering 时禁用全部播控：实现最保守，但会放大“卡住无解”的挫败感。
     - buffering 时完全不展示任何状态变化：会让用户误判成点击无效或进度失真。

9. 播放服务的“是否已经在播当前曲目”判断收紧为“当前状态属于该曲目”，而不是继续复用一个裸的全局 `PLAYING` 投影。
   - 方案：`PlaybackProcessRuntime` 额外记录当前真正启动过播放的 `activePlaybackTrackId`。`playCurrent()` 只有在 `playWhenReady = true`、`playbackState = PLAYING` 且 `activePlaybackTrackId` 与当前曲目一致时，才允许把当前请求视为“already active”并跳过；如果只是旧曲目 observer 迟到回写的 `PLAYING`，或者当前曲目仍处于 preparing，则必须重新 `open + launchPlay`。
   - 原因：最近的现场日志已经证明，仅靠共享 `playbackState == PLAYING` 判断“已经在播”会把旧曲目的残留状态误投到新曲目上，直接造成切歌后卡缓冲、当前曲目被 skip、不自动播放。
   - 备选方案：
     - 切歌时只做一次状态 reset：不够稳，因为旧 observer 下一拍仍可能把 `PLAYING` 写回来。
     - 在 UI 侧遇到卡缓冲就重复下发播放命令：只能掩盖症状，不能修复运行时对曲目归属的判断错误。

## Migration Plan

1. 升级上层领域模型：在线歌曲使用语义化 `MusicInfo`，本地播放预留 `LocalMusicInfo` 或等价本地模型，并补齐旧播放列表状态迁移。
2. 在 `playback-contract` 中定义统一最小可播放基类/接口，并让 app / playback-service 只在跨进程边界消费这份契约。
3. 为 `/song/detail` 等接口新增显式 mapper，把 `id`、`name`、`ar`、`al`、`dt` 等原始字段转换为 `MusicInfo` 的语义化命名，再投影为最小可播放基类。
4. 新增详情页播放网关，并把 `PlaylistDetailViewModel`、`ArtistDetailViewModel`、`AlbumDetailViewModel` 分别接到统一播放请求通道上；歌单详情、歌手详情和专辑详情都遵循“替换队列 + 用 index 设定激活项”的同一模型。
5. 为详情播放网关增加统一的 `/song/detail` enrichment 能力，按页请求 `songId` 元数据并回填当前播放列表；首版优先补齐当前激活曲目，再继续补其余页。

6. 详情页“播放全部”入口统一改成列表标题右侧的胶囊式带图标按钮，而不是继续放在 hero 区域里占用主视觉空间。
   - 方案：歌单详情页、歌手详情页和专辑详情页都在各自的歌曲区标题右侧放一个带播放图标的胶囊按钮；歌单/专辑使用“歌曲列表”区块，歌手使用“热门歌曲”区块。按钮采用主题主色系、圆角胶囊造型，并与标题保留一段固定间距，同时保留可测试 tag 与 content description。
   - 原因：用户明确给出了更接近胶囊按钮的视觉偏好，希望入口位于标题右侧且和整体色彩风格一致。这样入口既保留明显可点性，也不会把 hero 主视觉挤得过满。
7. 为在线播放解析新增 remote data source / repository / mapper，首版接通 `/song/url/v1`、`/song/url`、`/check/music`，并实现播放专用请求头构造。
8. 扩展在线播放准备层：先做 `CacheCore.lookup(resourceKey)` 完整性判断，再查询一个最多 10 项的 LRU URL 内存缓存，最后才决定是否解析 URL、如何打开缓存会话与如何补齐网络读取；仅完整音源解析结果进入该缓存，试听结果不占用容量。同时为在线磁盘缓存写入 clip-mode sidecar 标记，仅可信 full 缓存允许被直接复用。
9. 扩展 `TrackPreparationCoordinator` 或其相邻准备层，使最小可播放基类在进入网络源之前完成懒解析、试听片段建模、`durationMsHint` 透传和失败映射。
10. 扩展 `PlaylistDetailRepository` / `PlaylistDetailViewModel` / `PlaylistDetailScreen`，引入 `dynamicState` 并渲染评论数、收藏状态、播放数等信息。
11. 在详情页真实播放开始成功后接入 `/playlist/update/playcount`，并以 playlist id 做会话级去重。
12. 补充测试：播放列表状态迁移、歌单详情播放入口、`/song/detail` 分页 enrichment 回填、缓存完整性分支、URL 内存缓存命中/过期/LRU 淘汰、在线 URL 解析回退、试听片段建模、`durationMs` / `coverUrl` 直传、字段 mapper 正确性、领域模型到可播放基类的投影正确性、动态信息加载、播放量更新去重、删除当前在线项后的衔接行为。
