## ADDED Requirements

### Requirement: 在线曲目在缓存不完整时按歌曲标识解析播放地址
系统 SHALL 在准备受保护在线曲目时先判断缓存是否完整；仅当缓存不存在或不完整时，系统 MUST 基于歌曲标识解析真实播放地址，而不是假定歌单详情接口或本地播放列表已经持有长期有效的在线直链。

#### Scenario: 完整缓存命中时跳过播放地址解析
- **WHEN** 系统准备一个具备歌曲标识的在线曲目
- **AND** 该曲目的缓存已被判断为完整
- **THEN** 系统直接使用现有缓存作为读取入口
- **AND** 系统不再请求 `/song/url` 或 `/song/url/v1`

#### Scenario: 稀疏 data 文件不能仅凭文件长度伪装成完整缓存
- **WHEN** 某首在线歌曲的缓存 data 文件长度已经增长到与 `contentLength` 相同
- **AND** 该缓存的 `completedRanges` 仍存在洞，尚未连续覆盖 `0..contentLength`
- **THEN** 系统不会把该缓存判定为完整缓存
- **AND** 系统会继续走在线播放地址解析与缺失区间补拉，而不是错误地进入 `cache-only` 准备

#### Scenario: 缓存不完整时优先使用新版地址解析接口
- **WHEN** 系统准备一个具备歌曲标识的在线曲目
- **AND** 该曲目的缓存不存在或仍不完整
- **THEN** 系统优先请求 `/song/url/v1`
- **AND** 系统在获得可用结果后使用解析出的播放地址继续准备在线播放

#### Scenario: 新版地址解析不可用时回退旧接口
- **WHEN** `/song/url/v1` 未返回可用播放地址或当前账号/设备条件无法满足
- **THEN** 系统按兼容策略回退到 `/song/url`
- **AND** 仅在所有地址解析路径都无法获得可用结果时才判定该在线曲目准备失败

### Requirement: 在线地址解析结果支持短期内存复用
系统 SHALL 为最近成功解析的少量在线曲目保留带过期时间的内存地址缓存，以减少切回上一首时重复请求拉流地址；该缓存 MUST 受容量与有效期约束，而不是无限增长或无限期复用旧直链。

#### Scenario: 切回最近播放歌曲时复用未过期解析结果
- **WHEN** 当前曲目缓存不完整且用户切回最近播放过的上一首
- **AND** 该曲目的解析结果仍在内存缓存中且尚未过期
- **THEN** 系统复用缓存中的播放地址与请求头继续准备播放
- **AND** 系统不再重复请求 `/song/url` 或 `/song/url/v1`

#### Scenario: 解析结果过期后重新请求播放地址
- **WHEN** 某在线曲目的解析结果已超过接口返回的有效期
- **THEN** 系统重新请求在线播放地址解析接口
- **AND** 过期的缓存结果不再被用于实际播放

#### Scenario: 内存地址缓存容量受限
- **WHEN** 系统已经缓存 10 首在线曲目的解析结果且新的曲目解析成功
- **THEN** 系统按最近最少使用策略淘汰最旧的一个缓存项
- **AND** 系统不会无限增长该内存缓存

#### Scenario: 试听解析结果不占用地址缓存容量
- **WHEN** 在线地址解析结果表明当前歌曲仅提供试听片段
- **THEN** 系统可以使用该结果继续当前歌曲的在线播放准备
- **AND** 系统不会把该试听结果写入 10 首在线地址内存缓存
- **AND** 该试听结果不会挤占完整音源解析结果的复用容量

#### Scenario: 仅可信 full 磁盘缓存允许直接复用
- **WHEN** 系统发现某首在线歌曲存在看似完整的 `full` 磁盘缓存
- **AND** 该缓存缺少可信的 clip-mode 标记或无法证明自己对应完整音源
- **THEN** 系统不会把该缓存直接当成完整音源复用
- **AND** 系统会丢弃该不可信缓存并重新解析、重建在线缓存

#### Scenario: 明显短于歌曲详情时长的在线播放结果不能再当成完整音源
- **WHEN** 系统为在线歌曲解析到一个未显式标记 `freeTrialInfo` 的播放结果
- **AND** 该结果返回的实际时长明显短于当前歌曲详情或播放列表里已有的稳定时长
- **THEN** 系统不会继续把该结果当成可信 `full` 音源缓存或复用
- **AND** 系统会把它视为受限短片段或疑似截断流处理
- **AND** 该结果不会占用完整音源的 URL 复用容量

#### Scenario: 命中 full 磁盘缓存时仍需防止短资源伪装成完整音源
- **WHEN** 系统准备复用某首在线歌曲的 `full` 磁盘缓存
- **AND** 当前歌曲具备来自详情接口或播放列表的稳定时长提示
- **AND** 该缓存探测出的实际媒体时长明显短于稳定时长提示
- **THEN** 系统不会继续把该缓存当作完整音源播放
- **AND** 系统会丢弃该缓存并重新解析在线播放结果

#### Scenario: provider 发现真实总长更长时必须纠正旧 contentLength
- **WHEN** 系统准备复用某首在线歌曲的历史 `full` 磁盘缓存
- **AND** 该缓存或 source 内仍保留了一个较短且过时的 `contentLength` 提示
- **AND** 实际 Range 读取过程中 provider 已确认远端真实总长明显更长
- **THEN** 系统必须用真实总长纠正当前 source、缓存 session 与持久化 metadata
- **AND** 系统不得继续沿用旧的较短 `contentLength` 把后续数据误判为 EOF

#### Scenario: seek 后命中合法 offset 时 provider 不把空读误判为 EOF
- **WHEN** 用户在在线播放过程中执行 seek，新的读取目标仍位于已知内容长度之内
- **AND** 新 Range 连接首次读取发生了异常空读、瞬时 `0 byte` 或立即 EOF
- **THEN** provider 不会立刻把这次空读当成可信 EOF 返回给上层
- **AND** provider 会主动关闭当前连接并对同一 offset 执行至少一次重连重试
- **AND** 如果后续重连恢复出字节流，播放链路继续沿该 seek 目标恢复，而不是长时间停留在 buffering

#### Scenario: HTTPS Range 读取必须保留原始域名进行证书校验
- **WHEN** 系统为在线歌曲准备 HTTPS 播放地址并建立 Range 读取
- **THEN** provider 必须使用原始 HTTPS 域名发起连接
- **AND** 系统不会把该 HTTPS 域名偷偷替换成裸 IP 再依赖 `Host` 请求头继续连接
- **AND** 系统不会因为证书主机名校验失败把本次播放直接打成 `source.readDirect returned error`

#### Scenario: 独立 metadata probe 成功时不强制 rewind 真正播放 source
- **WHEN** 系统为在线歌曲准备播放 source 时额外创建了一条独立的 metadata probe source
- **AND** 当前曲目的时长或音频元信息已经从这条 probe source 成功读取
- **THEN** 系统不会再要求真正用于播放的 source 额外执行一次无意义的 rewind
- **AND** 不会因为 playback source 上一次未发生实际读取而把本次准备误判为 `Source rewind failed`

### Requirement: 在线准备区分完整音源、试听片段与不可用音源
系统 SHALL 区分完整可播音源、试听片段与不可用音源，并将结果映射为可解释的在线播放准备状态，而不是把所有失败都视为同一种通用网络错误。

#### Scenario: 未登录或非会员仅获得试听片段
- **WHEN** 在线地址解析结果表明当前歌曲仅提供试听片段
- **THEN** 系统仍允许该歌曲进入在线播放准备
- **AND** 共享播放结构记录试听片段的开始时间和结束时间
- **AND** 播放状态能够明确区分该歌曲不是完整音源

#### Scenario: 可疑短片段结束时不能静默伪装成正常整首播完
- **WHEN** 在线歌曲因为受限短片段或疑似截断流在远早于稳定时长提示的位置结束
- **THEN** 系统不会把这次结束继续解释为可信的整首自然播完
- **AND** 系统会优先尝试刷新受影响的缓存和解析结果

#### Scenario: 无版权时给出可解释失败
- **WHEN** 在线地址解析失败且可用性检查返回当前歌曲不可用
- **THEN** 系统以明确的不可用结论结束本次在线播放准备
- **AND** 系统不把该失败误判为需要重新登录
