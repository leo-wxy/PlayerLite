## Context

当前用户中心仍是一个以 `UserSessionUiState` 为核心输入的静态页面：`MainActivity` 通过 `PlayerViewModel.userSessionUiStateFlow` 把登录态头部信息传给 `UserCenterScreen`，页面本身只负责展示头像、摘要、登录/退出按钮和一段“后续补充”的占位文案，还没有承载任何登录后在线内容。与此同时，仓库里已经有稳定的在线请求模式：`AppContainer` 用同一个 `JsonHttpClient` 注入 `AuthHeaderProvider`，认证型接口通过 `requiresAuth = true` 走现有 `Cookie` / `X-CSRF-Token` 透传链路，`UserRepository` 负责持久化会话、刷新用户信息以及在会话失效时统一降级。

这次变更不是重做账户体系，而是在现有会话与主壳层结构不变的前提下，把用户中心从“会话承载位”扩展成“可浏览个人在线内容的一级页”。这带来几个技术约束：

- 不能把用户中心内容加载逻辑继续堆进 `PlayerViewModel`，否则播放控制、会话展示和个人中心内容会进一步耦合。
- 新增接口必须沿用现有认证请求方式，至少保证 `Cookie` 自动进入请求头，并在会话失效时与现有登录态降级机制保持一致。
- 用户要求页面整体风格偏向云音乐效果，但当前仓库还有一个单独的 `refresh-login-and-user-center-style` change 在持续调整账户页视觉，因此本次设计应复用账户页已建立的视觉 token，而不是再造一套互相打架的局部样式。
- 三类内容需要通过 Tab 切换展示，这意味着状态模型不仅要表达“已登录/未登录”，还要表达当前选中的 Tab、每个 Tab 是否已加载、加载结果和失败状态。

## Goals / Non-Goals

**Goals:**

- 在不改变现有登录流程、用户会话恢复和主壳层导航的前提下，为用户中心增加“收藏的歌手”“收藏的专栏”“用户歌单”三类登录态内容。
- 为用户中心新增一套独立于播放状态的内容状态模型，支持 Tab 切换、首次加载、缓存复用、失败重试和未登录降级。
- 复用现有 `JsonHttpClient` + `AuthHeaderProvider` 鉴权链路，让新增接口和现有受限接口保持同样的 `Cookie` 透传行为。
- 让用户中心已登录内容区形成偏云音乐的个人页体验，包括头部资料区、Tab 切换栏和统一的内容卡片区，同时不把局部风格外溢成全局主题变更。
- 为后续继续补充点击跳转、更多个人内容模块或明细页接入保留稳定的数据与 UI 骨架。

**Non-Goals:**

- 不改变手机号/邮箱登录、跳过登录、退出登录、会话持久化和主壳层双 Tab 结构。
- 不把“收藏内容展示”扩展成新的全局用户域架构，也不把在线播放/播放控制状态挪到用户中心专属 ViewModel。
- 不要求本轮一次性完成收藏内容卡片点击后的详情跳转；本轮重点是内容拉取、状态管理和一级页展示。
- 不直接修改全局 `PlayerLiteTheme` 或首页/播放器页的整体视觉风格，只在账户域内部收敛偏云音乐的局部页面表现。
- 不把三类内容一次性堆成同一长列表，而是明确以 Tab 作为一级切换方式。

## Decisions

### 1. 新增独立的 `UserCenterViewModel` 与 `UserCenterRepository`，不把个人中心内容塞进 `PlayerViewModel`

当前 `PlayerViewModel` 已经承担播放控制、播放页状态、用户会话头部投影等职责。如果继续把“用户中心 Tab 状态 + 三类接口加载 + 重试/降级逻辑”放进去，会把个人中心内容和播放域进一步耦合，导致后续维护时难以判断哪个状态属于播放，哪个状态属于账户内容。

本次设计选择：

- 保留 `PlayerViewModel -> UserSessionUiState` 这条现有链路，继续为用户中心头部提供登录态、昵称、摘要和头像。
- 新增 `UserCenterViewModel` 只负责个人内容相关状态，例如当前选中的 Tab、每个 Tab 的加载状态、错误提示和映射后的展示列表。
- 新增 `UserCenterRepository` 负责聚合三类远端请求和领域映射，UI 层不直接碰远端数据源。
- `MainActivity` 同时收集 `userSessionUiStateFlow` 与 `UserCenterViewModel.uiStateFlow`，再把两者传入 `UserCenterScreen`。

这样做能让“用户头部身份状态”和“用户中心内容状态”保持解耦，同时不需要重构当前登录链路。

**备选方案：**

- 直接扩展 `PlayerViewModel`：改动入口最少，但会继续放大该 ViewModel 的职责边界。
- 让 `UserCenterScreen` 自己在 Composable 中发请求：实现快，但会把请求、副作用和重试逻辑直接推到 UI 层，不符合仓库当前 `ViewModel + Repository` 的组织方式。

### 2. Tab 切换使用“单选中 Tab + 分 Tab 缓存”的状态模型，而不是一次性并发加载全部内容

用户明确要求三类内容通过 Tab 切换展示，因此 UI 上任意时刻只需要渲染一个激活中的内容面板。若进入用户中心时一次性并发拉取三个接口，虽然切换会更即时，但会增加首屏等待、无谓流量和错误处理复杂度，而且用户未必会点完三个 Tab。

本次设计选择：

- 定义稳定的 `UserCenterTab` 枚举，并明确已登录内容区的展示顺序为 `PLAYLISTS -> ARTISTS -> COLUMNS`。
- 由 `UserCenterUiState` 记录 `selectedTab` 和每个 Tab 对应的 `UserCenterTabContentState`。
- 每个 Tab 拥有独立的 `isLoading`、`items`、`errorMessage`、`hasLoaded` 等字段。
- 首次进入已登录用户中心时默认选中并拉取 `用户歌单` Tab；当用户切换到尚未加载的 Tab 时再触发请求；已经成功加载过的 Tab 直接复用缓存，除非用户显式重试或触发刷新。

这套模型既满足 Tab 切换，又避免因为“一个接口失败”把三个内容面板全部打成错误态。

**备选方案：**

- 进入页面时并发加载全部 Tab：切换体验最好，但首屏成本高，也会放大失败处理分支。
- 每次切换都重新请求：实现最简单，但会导致来回切换抖动明显，也浪费网络请求。

### 3. 新增用户中心远端数据源沿用现有 `JsonHttpClient` 鉴权链路，并把会话失效映射成统一降级信号

仓库里已有认证型接口的稳定模式：`JsonHttpClient` 通过 `requiresAuth = true` 触发 `AuthHeaderProvider`，由当前 `UserSession` 生成 `Cookie` 和 `X-CSRF-Token` 请求头。用户特别强调“记住要加 cookie”，所以新接口不能走裸请求，也不能在 UI 层手动拼接头部。

本次设计选择：

- 在 app 侧新增 `UserCenterRemoteDataSource`，其三个远端方法分别对应收藏歌手、收藏专栏和用户歌单。
- 所有这三类请求都基于共享的 `JsonHttpClient` 实例，统一使用 `requiresAuth = true` 发起，这样会自动复用当前登录会话的认证头。
- 远端层复用和 `NeteaseUserRemoteDataSource` 相同的响应码校验思路：`200` 视为成功，`301/302` 视为会话失效，其余状态转成普通请求失败。
- `UserCenterRepository` 遇到会话失效时不上抛裸网络错误，而是转换成可识别的失效信号；`UserCenterViewModel` 收到后触发 `UserRepository.logout()` 或等价的统一降级流程，让头部和内容区一起回到未登录态。

这样既能满足“必须带 cookie”的实现要求，又能保证用户中心扩展内容遵守现有会话体系。

**备选方案：**

- 把这三个接口直接加进 `UserRepository`：会让 `user` 模块从会话域膨胀成个人页内容域，边界会变得模糊。
- 在 ViewModel 里直接调用 `JsonHttpClient`：实现路径短，但鉴权、映射和失效处理都会散落在 UI 状态层。

### 4. 三类接口先映射到统一的卡片型 UI 模型，页面以“头部资料区 + Tab 栏 + 单内容面板”承载偏云音乐风格

三类内容的原始字段结构不同，但从首版展示目标看，它们都可以被投影成“封面/头像 + 标题 + 副标题 + 附加标签”的卡片型信息单元。若一开始就为每个 Tab 建三套完全不同的数据模型和布局，会增加 UI 和测试复杂度，也不利于形成统一的云音乐式内容区观感。

本次设计选择：

- 仓储层把三类接口先映射为统一的 `UserCenterCollectionItemUiModel`，字段至少包含 `id`、`title`、`subtitle`、`imageUrl` 和可选 `badge` / `meta`。
- `UserCenterScreen` 继续保留当前头部资料区，但已登录态不再插入独立说明卡；个人资料卡片保持完整四角圆角并与下方 Tab 内容面板分组展示，Tab 栏作为内容面板头部，当前 Tab 的列表/空态/错误态直接衔接在下方。
- 当用户中心内容区整体向上滚动时，筛选 Tab 栏通过 `LazyColumn` 的吸顶头部保持在顶部；说明文案随列表正常滚动离场，避免把整块高头部一并钉在屏幕顶端。
- 在移动端布局下，资料头与 Tab 内容面板保留一圈更小的横向留白，并在两者之间维持小间距分组；目标是既保持紧凑，又避免过于贴连或完全贴边带来的生硬感。
- Tab 栏使用稳定的可测试节点和单一选中态，只渲染当前 Tab 的内容，避免在一个滚动容器里叠三段大列表。
- 页面风格继续使用账户域已有的 `AccountVisualStyle` / 头部 Hero 视觉作为基础，再补充更明确的红色强调、Tab 指示器和内容卡片层次，让整体更接近云音乐式个人页氛围，而不改造全局主题。
- 内容 item 间距采用更紧凑的列表节奏，减小 Tab 区与首个 item、以及 item 之间的垂直留白，优先保证“个人内容直接连在一起”的浏览感。

这能让首版先把“内容真的出来了”和“整体看起来像音乐个人页”两件事一起完成，同时保留后续为不同 Tab 做定制卡片的空间。

**备选方案：**

- 为每个 Tab 定义完全独立的 item 模型和布局：表达力更强，但首版实现与测试成本明显更高。
- 继续用当前静态卡片加说明文字，不引入 Tab：实现最省，但不满足用户提出的交互要求。

### 5. 用户中心内容失败只影响当前内容面板，不影响头部；但会话失效必须触发整页统一降级

用户中心页面包含两类失败：一类是普通网络失败或空内容，另一类是会话失效。两者对 UI 的处理方式不应该一样。普通失败不应把头部资料区抹掉，否则用户会感觉整页“崩了”；但会话失效又必须和现有登录体系保持一致，不能只让某个 Tab 显示错误而头部还假装在线。

本次设计选择：

- 对普通错误，保留头部资料区和 Tab 栏，只在当前 Tab 面板展示错误提示与重试入口；其他已缓存 Tab 继续可用。
- 对空内容，使用与页面风格一致的空态文案和图形占位，不把空列表视作错误。
- 对会话失效，`UserCenterViewModel` 触发统一登出/降级，清空 Tab 内容缓存，让页面切回未登录态，并重新显示登录引导。

这样可以兼顾稳定体验与会话一致性，不会把“一个接口偶发失败”和“账号真的失效了”混为一谈。

**备选方案：**

- 所有失败都整页降级：逻辑简单，但普通网络波动会导致体验过于激烈。
- 所有失败都只做局部提示：会让会话失效场景与系统其他登录态行为不一致。

## Risks / Trade-offs

- **[用户中心状态入口增加，MainActivity 需要额外持有一个 ViewModel]** -> 通过把内容状态限制在用户中心域内，换取 `PlayerViewModel` 不继续膨胀，整体可维护性更高。
- **[默认只加载一个 Tab，首次切换到其他 Tab 会有等待]** -> 使用分 Tab 缓存、显式 loading 态和轻量骨架屏减轻切换首载感知；后续如果证据表明切换频繁，再升级为预取策略。
- **[统一卡片模型可能遮蔽不同内容类型的个性字段]** -> 首版只抽取共性字段，保留 `badge/meta` 扩展位；后续若某一类内容需要更强表达，再局部拆分。
- **[账户域局部风格和另一个 user-center-style change 发生冲突]** -> 以当前账户域视觉 token 为基础追加 Tab 和内容区，而不是另起新的主题分支；实现阶段优先复用现有 Hero/卡片样式。
- **[后端对“专栏”接口命名不稳定或字段差异较大]** -> 远端层按语义方法命名，接口 path 和 mapper 细节隔离在数据源内部，避免把服务端命名直接污染到 UI 和状态层。

## Migration Plan

1. 定义 `UserCenterTab`、`UserCenterUiState`、`UserCenterTabContentState` 和统一的内容卡片 UI 模型，明确用户中心内容状态边界。
2. 新增 `UserCenterRemoteDataSource` / `UserCenterRepository`，接入三类认证型接口，并补充响应码校验、空态映射与会话失效处理。
3. 在 `AppContainer` 中装配新的用户中心仓储，新增 `UserCenterViewModel`，让其观察登录态并按需加载默认 Tab 与切换 Tab 内容。
4. 重构 `UserCenterScreen`：保留现有头部资料区的风格基础，引入 Tab 栏、内容面板、空态/错误态和重试入口，并把内容区视觉收敛为偏云音乐的账户页风格。
5. 调整 `MainActivity` 的用户中心分支，同时收集 `userSessionUiStateFlow` 与 `UserCenterViewModel.uiStateFlow`，把登录、退出登录和 Tab 内容状态统一传给页面。
6. 补充单元测试和 UI 测试，至少覆盖：登录态下默认 Tab 加载、Tab 切换缓存、`requiresAuth` 请求头透传、普通失败局部展示、会话失效整页降级、未登录态不触发受限接口请求。

## Open Questions

- “收藏的专栏”对应的服务端接口和字段命名是否已经在当前后端中稳定存在，还是需要在实现前再次确认具体 path 与返回结构？
- 首版是否只要求内容展示，还是需要同步接入点击卡片进入歌手/歌单/专栏详情页的跳转能力；如果需要，需与正在进行的详情页 change 协调边界。
