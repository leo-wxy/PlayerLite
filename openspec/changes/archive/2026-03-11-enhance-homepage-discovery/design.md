## Context

当前首页概览态由 `HomeOverviewScreen` 直接渲染一组占位内容：品牌封面、登录提示、当前文件名、播放状态，以及底部“进入播放页”按钮。这个结构满足了 Home 域与播放展开态的基础切换，但还没有承载“首页-发现”接口内容、默认搜索关键词轮播，或首页作为内容入口时需要的加载/空态/异常态。

现有前台主结构已经具备几个可复用约束：

- `MainActivity` 和 `MainShellState` 已将首页与播放展开态建模为同一 Home 域，不能为了发现页改成完全独立的一级路由。
- 仓库要求 `MainActivity` 保持轻量，Compose UI 尽量无状态，因此首页发现数据与搜索轮播状态不能继续堆进 Activity 或直接写在 Composable 里。
- `AppContainer` 已通过 `network-core` 和 feature repository 的方式组织用户会话相关依赖；首页发现如果需要远端接口，也应延续同一数据接入方式，而不是在 UI 中直接发请求。
- 当前仓库里还没有现成的首页搜索页或首页发现数据模型，因此本次设计既要补齐首页发现承载，也要为未来搜索能力预留边界。

## Goals / Non-Goals

**Goals:**

- 将首页概览态从静态占位内容升级为“发现内容首页”，按“首页-发现”接口的区块结构展示内容。
- 在首页顶部增加悬浮搜索框，并支持默认搜索关键词轮播展示。
- 为首页发现内容建立独立的数据模型、加载状态和错误恢复机制，避免把远端 DTO 直接暴露给 UI。
- 保持首页与播放展开态仍属于同一 Home 域，并继续保留进入播放展开态的稳定入口。
- 让首页切回播放展开态、从播放展开态返回首页时，不因重新取数或布局重排产生明显闪动。

**Non-Goals:**

- 不在本次设计中实现完整的搜索结果页、搜索历史或搜索建议接口。
- 不重构 `playback-service`、`PlayerRuntime` 或现有播放链路。
- 不改造用户会话、登录流程和底部双 Tab 的整体导航模型。
- 不在本次变更里定义所有发现卡片的最终视觉细节，重点先落稳定的数据结构与页面骨架。

## Decisions

### 1. 为首页发现引入独立的 `HomeViewModel + Repository` 链路，而不是继续扩展 `PlayerViewModel`

首页发现内容与播放控制、用户会话属于不同职责域。当前 `PlayerViewModel` 已承载播放状态、播放命令和用户会话投影；继续把首页发现接口、关键词轮播和首屏加载逻辑塞进去，会让播放器状态模型继续膨胀，也不符合仓库“页面 orchestration 与 UI 职责清晰分明”的约束。

本次设计选择新增独立的首页状态入口：

- 在 `app` 的 Home 相关目录下新增 `HomeViewModel` 或同等职责的状态持有者，向 UI 暴露 `StateFlow<HomeOverviewUiState>`。
- 在 `AppContainer` 中新增首页发现 repository 的构造入口，延续已有的 `JsonHttpClient + RemoteDataSource + Repository` 依赖组织方式。
- `MainActivity` 只负责收集 `HomeOverviewUiState` 并把它传给 `HomeOverviewScreen`，不直接参与取数与业务状态拼装。

这样做的好处是：首页发现的数据刷新、缓存、错误恢复可以独立演进，后续即使搜索页或更多首页模块加入，也不需要把 `PlayerViewModel` 变成一个“全首页 ViewModel”。

**备选方案：**

- 继续扩展 `PlayerViewModel`：改动入口少，但会混合播放、用户、发现内容三类职责，长期维护成本高。
- 在 Composable 内直接调用 repository：短期快，但会让页面变成状态与渲染耦合体，不利于测试，也违背 Compose UI 尽量无状态的约束。

### 2. 首页概览态改为“可滚动发现内容 + 顶部悬浮搜索框 + 底部播放入口”的三层布局

现有首页是一个垂直居中的静态 Column，无法同时满足发现内容浏览、搜索框悬浮和底部播放入口常驻的需求。本次设计将首页概览态改成三层结构：

- **底层内容层**：使用可滚动容器承载发现区块，推荐采用 `LazyColumn`，便于按 section 渲染不同模块并支持后续扩展。
- **顶部悬浮层**：搜索框作为浮在内容层上的独立组件，贴合顶部安全区定位，内容列表通过 top padding 预留空间，避免首屏内容被覆盖。
- **底部入口层**：继续保留进入播放展开态的入口，但从“页面主体内容的一部分”调整为浮在底部安全区附近的固定 CTA，避免被发现流内容挤走。
- **区块稳定性**：横向推荐区块统一采用容器级 `contentPadding + spacedBy` 管理左右留白和项间距，避免给每个 item 单独加外边距导致首个卡片阴影被裁切；同时各类卡片使用固定高度，避免不同文案长度在滚动时触发明显的整屏抖动。
- **Banner 展示方式**：Banner 区块不再复用普通横向卡片列表，而是采用固定高度的 pager/carousel 展示，保留横向切页语义并统一首尾留白。
- **Banner 视觉铺满**：Banner 图片本身应在卡片内部做 full-bleed 填充，文字信息直接浮在图片之上，而不是让图片只占卡片上半部分。
- **Banner 外侧留白**：Banner carousel 仍保留轻量容器级左右 padding 用于阴影呼吸空间，但该 padding 应小于普通横向区块，让 banner 更接近整行铺满。
- **Banner 循环方式**：Banner carousel 需要提供“真循环”观感，用户连续左右滑动时不会在首尾处明显撞墙，而是通过虚拟页映射等方式保持连续切换体验。
- **每日推荐快捷入口样式**：Banner 下方的“每日推荐/私人FM/歌单”等快捷入口改为纯文字低高度卡片，不再显示图片；每个卡片根据稳定 seed 生成浅色背景，提升区分度同时避免随机闪烁。
- **普通推荐卡封面比例**：Banner 之后的普通推荐卡片封面移除额外外间距，直接满宽铺在卡片顶部，并固定为 1:1 比例，保证浏览时视觉更整齐。
- **文本行数约束**：为进一步稳定卡片高度，首页各类 item 的标题/副标题统一限制为最多一行并使用省略号收口，不再让多行文本参与卡片高度竞争。
- **Banner 辅助信息样式**：Banner 底部辅助信息采用纯文字标签/徽标样式，不依赖额外图标资源，以避免出现“图标缺失但留白怪异”的展示问题。

这能同时满足用户提出的“清空当前主页占位内容”“主要上方添加悬浮搜索框”“首页内容按发现接口展示”，并保持现有 Home 域进入播放展开态的交互约定不丢失。

**备选方案：**

- 把搜索框放进列表第一个 item：实现简单，但不符合“悬浮搜索框”的要求，滚动后也会丢失首屏入口。
- 把播放入口融进发现流：页面更统一，但会削弱播放器入口的稳定性，也与现有 `user-center-tab-shell` 中首页进入播放展开态的 requirement 不一致。
- 横向列表继续依赖 item 外边距：实现直接，但首个/末尾卡片阴影容易被裁切，整体左右留白也更不均衡。

### 3. 用“接口 DTO -> 统一 Section UI Model”的中间映射层承接首页-发现接口

首页-发现接口天然会包含多种内容区块，如果直接让 UI 渲染远端 DTO，页面会强依赖后端字段命名和区块枚举，后续接口变动时 UI 和测试都容易连带破坏。因此本次设计选择引入统一的 section 映射层：

- 远端层只负责解析原始接口响应。
- repository 将响应转换为 `HomeDiscoveryPayload` / `HomeSectionUiModel` 这类面向 UI 的稳定模型。
- UI 按 section 类型分发到对应的渲染组件，例如 banner、歌单网格、歌曲横滑区块、榜单条目等。
- 对当前版本暂不支持的 section 类型，repository 侧以“跳过未知区块 + 保留日志/埋点入口”的方式处理，避免整个首页因单个未知区块失败。

这种做法的核心价值是让 spec 可以围绕“首页按发现接口区块展示”的行为建模，而不是把实现绑死在某一版接口结构上。

**备选方案：**

- UI 直接渲染网络 DTO：开发速度快，但后续接口调整会把页面、预览、测试全部绑在一起。
- 首版只硬编码一两种区块：实现最省事，但无法兑现“按照首页-发现接口进行展示”的目标，也会让后续扩展变成重写。

### 4. 默认搜索关键词由“接口优先，本地兜底”的策略提供，轮播由 UI 状态层驱动

用户明确要求搜索框内容根据默认搜索关键词轮播展示，但当前仓库中还没有独立搜索能力或关键词源。为减少阻塞，本次设计将关键词来源拆成两层：

- 如果“首页-发现”接口本身返回默认搜索词列表，则 repository 直接透出这组词。
- 如果接口暂未返回或返回为空，则 repository 提供一组本地兜底默认词，保证搜索框总能展示可轮播内容。
- UI 状态层只消费一个稳定的 `List<String>` 与当前轮播索引；轮播定时切换由 `HomeViewModel` 或与其配套的状态逻辑驱动，而不是让 Composable 自己维护不可测试的临时状态。

这样可以确保“悬浮搜索框 + 关键词轮播”不依赖完整搜索功能落地，也便于后续把关键词来源切换成服务端配置。

**备选方案：**

- 只用写死文案：实现最简单，但无法与真实首页数据源对齐。
- 完全依赖服务端关键词：语义更统一，但一旦接口缺字段或为空，首页搜索框会退化成空壳。

### 5. 首页发现数据采用“首进加载、保留上次成功结果、显式重试”的状态策略

首页位于主 Tab 中，用户会频繁在首页、用户中心与播放展开态之间切换。如果每次回到首页都强制重新请求，会让搜索框轮播与内容区块频繁闪烁；如果完全不重试，又会让异常态缺乏恢复能力。为平衡体验与复杂度，本次设计采用三态数据策略：

- 首次进入首页时触发加载。
- 成功后缓存最近一次成功结果，在 Home 域内切换到播放展开态再返回时直接复用，不重复闪屏。
- 失败时保留可识别的异常态和重试入口；如果之前已有成功数据，则优先保留旧内容并辅以轻量刷新失败提示，而不是整页清空。

这个策略适合首页发现这种“首屏浏览内容”场景，也能减少因为页面结构切换导致的感知抖动。

**备选方案：**

- 每次 `HomeOverviewScreen` 重组都重新加载：实现简单，但体验最差。
- 只加载一次且不支持重试：实现轻，但异常态不可恢复。

### 6. 保持 `HomeSurfaceMode` 不变，只替换首页概览态内容承载

本次变更聚焦首页能力升级，而不是重新设计 Home 域路由。因此不新增新的一级页面状态，也不改变 `HomeSurfaceMode.OVERVIEW / PLAYER_EXPANDED` 的整体建模。实现上只替换 `OVERVIEW` 模式下的内容承载方式：

- `HomeOverviewScreen` 接收新的 `HomeOverviewUiState`、搜索点击回调与播放入口回调。
- `PLAYER_EXPANDED` 相关动画、返回行为和底部 Tab 隐藏逻辑保持现状。
- 首页发现内容层和底部播放入口共用同一个 Home 容器，保证从首页到播放展开态的视觉关系仍属于同一域。

这样风险更低，也符合 proposal 中“清除当前主页内容但保持首页仍属于现有 Home 域”的要求。

**备选方案：**

- 为首页发现再新增一个独立路由层：会让 Home 域状态更复杂，也不利于维持首页与播放展开态的单域关系。
- 顺手重写 Home 动效：收益有限，但会显著扩大本次变更范围。

## Risks / Trade-offs

- **[首页-发现接口结构不稳定]** → 通过 repository 中间映射层隔离远端 DTO，并对未知 section 做跳过处理，降低 UI 直接受影响的风险。
- **[悬浮搜索框与滚动内容发生遮挡]** → 通过顶层 overlay + 列表预留 top inset 的方式规避，同时在实现中校验状态栏和不同屏幕尺寸下的安全区。
- **[底部播放入口被发现内容弱化]** → 保持播放入口为独立底部浮层，而不是列表内容的一部分，确保进入播放展开态的路径稳定。
- **[关键词轮播影响可读性]** → 控制轮播节奏，只切换占位/提示文本，不改变输入框整体布局；后续如用户反馈干扰，可退回静态默认词。
- **[首页缓存内容可能不够新]** → 以“保留上次成功结果 + 支持显式重试”的方式折中，不为了实时性牺牲首页返回体验。
- **[搜索点击行为尚未有完整目标页]** → 设计上预留 `onSearchClick` 边界，避免把搜索入口与具体搜索结果页强耦合；若本轮不接搜索页，需在 specs 中明确首版行为。

## Migration Plan

1. 在首页相关 feature 中新增首页发现状态模型、repository 接口与 remote data source，对接现有 `network-core` 能力。
2. 扩展 `AppContainer`，提供首页发现依赖的构造入口，并在 Activity / ViewModel 层完成注入。
3. 将 `HomeOverviewScreen` 从静态占位布局替换为“发现流 + 悬浮搜索框 + 底部播放入口”的新结构。
4. 补充关键词轮播、加载态、空态、异常态与重试路径，并确保 Home 域切换回首页时复用最近成功结果。
5. 保持 `HomeSurfaceMode` 与现有播放展开态逻辑不变，回归验证首页进入播放展开态、从播放展开态返回、底部 Tab 显隐和用户中心切换。

## Open Questions

- “首页-发现”接口的稳定区块类型与字段清单是否已经固定，还是需要前端先按通用 section 模型兜底？
- 搜索框在首版点击后应进入哪个目标：已有搜索页、待建搜索页占位，还是仅展示 UI 不执行跳转？
- 默认搜索关键词是否包含在“首页-发现”接口响应中；如果没有，产品侧希望使用哪组本地兜底词？
- 底部播放入口首版是继续保持按钮形态，还是希望在本轮直接演进为带当前歌曲摘要的 mini player 入口？
