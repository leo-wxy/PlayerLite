## Context

当前个人页实现位于 `app` 模块的 `UserCenterScreen` / `UserCenterProfileHeader`，已经具备以下基础能力：

- 复用 `UserSessionUiState` 展示登录态头像、昵称与摘要。
- 通过 `UserCenterViewModel` + `UserCenterRepository` 承载“用户歌单 / 收藏歌手 / 收藏专栏”三类 Tab 内容。
- 通过 `AppContainer` 共享 `JsonHttpClient` 与 `AuthHeaderProvider`，认证型请求统一复用当前会话的 `Cookie` / `X-CSRF-Token`。

但现状仍偏“能力骨架”：

- 页面头部只有基础资料与本地歌曲入口，缺少参考稿 `cc.html` 中那种“头像主视觉 + 会员/状态信息 + 快捷入口 + 内容分区”的完整个人页节奏。
- 已登录态内容仍主要围绕收藏 Tab，缺少“个人首页只展示用户自己的歌单 + 高频快捷入口跳转到独立子页面”的清晰结构。
- 当前“收藏专栏”数据源使用 `/dj/sublist`，与这次确认的“收藏专栏”语义不一致；应切换到 `/topic/sublist`。
- 仓库当前对 `SearchRouteTarget` 的落地能力主要覆盖歌手 / 歌单 / 专辑详情，歌曲、视频、播客等最近播放项目还没有通用的详情或播放落地路径。

这次设计不是重写账户体系，也不是重做整个用户域导航，而是在现有用户中心结构上做增量升级：参考 `cc.html` 调整页面结构和视觉节奏，并补齐个人页直接需要的接口调用、数据映射和状态组织方式。

## Goals / Non-Goals

**Goals:**

- 在现有 `UserCenterScreen` 基础上升级为更完整的个人页，补上头部强化信息、快捷入口与更明确的子页面导航。
- 保持当前 `UserCenterViewModel` / `UserCenterRepository` 的页面级组织方式，但扩展其数据能力，支持：
- 用户资料补充信息与统计（账号、等级、订阅数量、绑定信息等）。
- 最近播放内容（首版以最近播放歌曲子页面为主，并保留向视频 / 声音 / 歌单 / 专辑 / 播客扩展的统一结构）。
- 个人首页主体只展示“用户自己的歌单”；“喜欢 / 最近 / 本地”通过快捷入口进入各自子页面，其中“本地”复用现有能力。
- 继续沿用现有认证请求链路，所有登录态接口通过统一的 `requiresAuth = true` 请求方式发出，并在会话失效时遵守现有统一降级逻辑。
- 让页面结构与 UI 节奏参考 `cc.html`，但不脱离现有账户域视觉 token 与 Compose 组织方式。

**Non-Goals:**

- 不重构 `UserRepository` 的登录、持久化、退出登录与会话恢复流程。
- 不在本次设计里新建复杂的多级导航体系；仅为“喜欢 / 最近 / 本地”补充必要的独立子页面入口。
- 不要求首版把用户给出的所有接口都直接可见地渲染到一个页面中；优先接入当前个人页直接需要的接口，并把其余同类接口纳入可扩展的数据骨架。
- 不在本轮解决“歌曲详情页”或“视频详情页”缺失的问题；对当前没有落地页的内容类型，首版允许使用明确提示或仅展示不跳转。
- 不改变底部双 Tab 主结构，也不把个人页拆成新的 Activity。

## Decisions

### 1. 继续使用 app 侧页面聚合仓储承载个人页内容，而不是把所有新增接口都下沉到 `user` module

当前 `user` module` 更适合承载“会话建立、会话恢复、用户基础资料刷新、退出登录”这类账户基础设施；`app/feature/main` 中的 `UserCenterRepository` 则已经承担了“个人页内容聚合”的角色。此次新增的数据分成两类：

- 会话相关的稳定资料补充：`/user/account`、`/user/detail`、`/user/subcount`、`/user/level`、`/user/binding`
- 页面内容型数据：`/record/recent/*`、`/artist/sublist`、`/topic/sublist`、`/mv/sublist`、`/user/playlist`

本次选择保持现有边界：

- `user` module` 继续只负责当前登录会话和 `UserInfo` 的基础生命周期。
- 个人页补充展示所需的“页面投影数据”由 `feature/main` 新增或扩展 `UserCenterRepository` 聚合。
- 如果某些接口的结果最终需要反哺全局会话摘要（例如等级或部分用户统计），由页面层显式映射到个人页 UI 状态，而不是立即修改 `UserInfo` 的持久化模型。

这样做的原因是：本次目标是升级个人页，而不是扩张用户域基础设施；把所有接口都塞进 `user` module` 会让会话域和页面内容域边界变得模糊。

**备选方案：**

- 把所有新增接口都并入 `UserRepository`：能集中“用户相关”代码，但会把 repository 职责从会话域膨胀到页面内容域。
- 在 UI / ViewModel 里直接发请求：实现快，但数据映射、错误处理和鉴权逻辑会散落在页面状态层。

### 2. 把个人页数据拆成“头部摘要块 + 快捷入口块 + 用户自己的歌单块”三个稳定分区，并把“喜欢 / 最近 / 本地”外提为子页面

参考 `cc.html`，本次个人页不再只靠一个 Hero 头部和下方收藏 Tab 支撑整页信息密度，而是调整为三类稳定分区：

- 头部摘要块：头像、昵称、会员/账号状态、等级或核心统计摘要
- 快捷入口块：一排固定入口，首版优先承载“喜欢 / 最近 / 本地”
- 用户自己的歌单块：主页主体只展示当前用户自己创建的歌单

其中：

- `本地` 入口直接复用现有 `LocalSongsActivity`
- `最近` 入口进入新的最近播放子页面，首版展示最近播放歌曲
- `喜欢` 入口进入新的“喜欢”子页面，以 Tab 形式切换展示不同类型的收藏内容（例如歌手 / 歌单 / MV / 专栏）

这意味着页面结构从“静态说明型个人页”转成“摘要 + 导航 + 自己的歌单”的个人中心布局，但仍然基于当前 `LazyColumn` 组织，不引入新的复杂滚动容器。

**备选方案：**

- 只微调头部样式，不新增分区：实现成本更低，但无法承载用户明确要求补充的接口与信息密度。
- 继续把所有内容堆在主页里：会让首页重新变回信息混杂的综合页，也不符合这次“喜欢 / 最近 / 本地分流到子页面”的要求。

### 3. 最近播放采用“独立子页面 + 统一最近播放域模型 + 首版只渲染歌曲”的扩展方案

用户提供了完整的最近播放接口族：

- `/record/recent/song`
- `/record/recent/video`
- `/record/recent/voice`
- `/record/recent/playlist`
- `/record/recent/album`
- `/record/recent/dj`

首版页面直接需要的是“最近”子页面，而不是一次性把六类最近播放都展开到个人主页主屏。设计上采用两层模型：

- 远端层保留按类型请求的方法，便于后续按接口族扩展。
- 页面层先统一映射到 `RecentPlaybackItemUiModel` 一类的稳定模型，至少包含 `id`、`type`、`title`、`subtitle`、`coverUrl`、`meta`、`action`。

首版渲染策略：

- 快捷入口点击后进入“最近”子页面，并在该页面请求和展示最近播放歌曲。
- 数据结构和 repository API 预留其他最近播放类型的扩展入口，但不要求首版全部占据主屏区域。
- 当前没有稳定详情页或播放落地页的内容类型，`action` 允许使用 `ContentEntryAction.Unsupported` 或后续再补的跳转动作。

这样做既不浪费用户已经给出的接口边界，也避免首版主页被多种“最近播放”内容撑爆。

**备选方案：**

- 只接 `/record/recent/song`，不设计统一模型：短期更快，但后续要接其他最近播放类型时会重复改 repository / UI 状态。
- 一次性把六类最近播放全部做成多 Tab / 多列表：信息量过大，且当前页面没有对应的完整落地交互支撑。

### 4. 主页歌单与“喜欢”入口共享用户歌单数据，但首页只保留“用户自己的歌单”

当前 `UserCenterRepository` 已经把用户歌单映射成统一的 `UserCenterCollectionItemUiModel`。这次需要进一步细分：

- 主页只展示“用户自己的歌单”，即由当前用户创建的歌单集合。
- “喜欢”入口不在主页内直接展开，而是进入新的“喜欢”子页面，并通过多 Tab 分区展示收藏内容。
- 主页不再强推“收藏 Tab”，收藏内容收敛到“喜欢”子页面。

保留统一卡片模型的原因是：

- 用户自己的歌单和“喜欢”入口都可投影成统一的歌单型卡片结构。
- 统一模型更有利于在主页列表与子页面之间共享视觉样式。

同时，这次不会直接把收藏歌手、收藏专栏、收藏 MV 放回主页，也不会接入 `/video/sub`、`/mv/sub` 的写操作主流程，避免本轮页面改版与范围控制相冲突。

**备选方案：**

- 继续保留“收藏 Tab 主页化”：与用户最新要求冲突。
- 把“喜欢”单独做成全新的歌曲域数据链路：长期可能更完整，但当前缺少用户明确要求，且会显著扩大范围。

### 5. 用户资料增强信息采用“页面附加摘要”而不是立即改造持久化 `UserInfo`

当前 `UserInfo` 已经持有一部分用户详情字段，如 `level`、`playlistCount`、`follows`、`followeds`、`eventCount`、`listenSongs` 等，但本次个人页还需要更多来自：

- `/user/account`
- `/user/subcount`
- `/user/level`
- `/user/binding`

这些数据里有一部分适合只用于当前页面摘要展示，而不一定适合作为全局持久化用户模型的一部分。设计上采用：

- `UserSessionUiState` 继续承载轻量的全局头部摘要。
- 个人页单独新增 `ProfileOverviewUiModel` 或同类页面摘要模型，承载会员标识、订阅计数、等级进度、绑定渠道摘要等更丰富的展示信息。
- 页面在已登录态进入时按需请求这些信息，并和 `UserSessionUiState` 合并投影；失败时只影响相应摘要区域，不影响整个登录态判定。

这样可以避免为了页面 UI 一次性扩展和持久化大量字段，同时保留未来把稳定字段上收进 `UserInfo` 的空间。

**备选方案：**

- 直接把所有新字段并入 `UserInfo` 并持久化：全局可复用性更高，但当前没有证据表明这些字段都需要跨页面长期保存。
- 完全不请求附加资料，只用现有 `UserInfo`：实现更省，但无法满足这次页面升级需要的完整度。

### 6. 页面动作沿用现有 `ContentEntryAction`，但对“没有落地路由的内容类型”显式保守处理

当前 app 已有稳定的页面动作协议：`ContentEntryAction.OpenDetail`、`OpenUri`、`Unsupported`。但目前 `searchRouteIntent()` 仅稳定支持：

- 歌手详情
- 歌单详情
- 专辑详情

歌曲、视频、播客、声音等内容没有同等成熟的详情落地。因此本次设计不引入新的全局路由协议，而是：

- 对已有落地页的内容继续走 `OpenDetail`。
- 对仅需展示的快捷入口和未完成落地页的内容，允许先走 `Unsupported` 并给出明确提示。
- 对“最近播放歌曲”这类高频内容，优先把数据展示和版块搭起来；如果后续确认要直接进入播放器或触发播放，再在实现阶段补充更合适的 action 类型。

这样做能保证这次改版不被“还没有详情页/播放落地页”的问题阻塞。

**备选方案：**

- 现在就扩展一套新的全局路由协议去承载歌曲 / 视频 / 播客：长期可扩展，但会把变更范围从个人页升级扩大到全局导航。
- 让所有新内容都默认不可点击：最安全，但会损失明显的交互反馈价值。

## Risks / Trade-offs

- **[页面信息块增多，首屏高度上升]** → 通过把快捷入口和最近播放控制在紧凑节奏内，并保留 sticky Tab 内容区，避免页面变成冗长仪表盘。
- **[最近播放类型扩展过早抽象可能带来过度设计]** → 统一模型只覆盖最小公共字段，首版主渲染歌曲，其他类型先复用接口边界而不强推 UI。
- **[用户资料增强信息和持久化模型脱节]** → 首版以页面投影为主，等真实复用场景出现后再决定哪些字段要上收进 `UserInfo`。
- **[收藏专栏接口切换可能遇到字段结构差异]** → 在 remote data source 内隔离 mapper；若真实返回与预期不一致，只影响该内容源，不波及页面主结构。
- **[缺少歌曲/视频详情落地页会限制点击行为]** → 首版明确把“展示优先、路由保守”作为策略，对未支持的点击提供稳定提示，不假装已支持。
- **[页面级 repository 继续扩张]** → 通过按“头部摘要 / 最近播放 / 收藏内容”分层组织接口与 mapper，避免 repository 退化成单个巨大文件；必要时再拆 `ProfileOverviewRepository` 或专门的数据源。

## Migration Plan

1. 扩展用户歌单数据映射，区分“用户自己的歌单”和“喜欢入口”所需信息。
2. 新增最近播放歌曲子页面的数据链路与页面壳层，并复用现有 action / route 机制处理点击反馈。
3. 调整 `UserCenterViewModel` 与主页状态，让主页只展示头部、快捷入口和用户自己的歌单列表。
4. 重构 `UserCenterScreen` / `UserCenterProfileHeader` 的页面结构，参考 `cc.html` 调整视觉节奏。
5. 让“本地”入口复用现有 `LocalSongsActivity`，让“喜欢 / 最近”入口进入各自子页面。
6. 补充与当前收缩后范围匹配的测试，验证主页列表、快捷入口与最近页加载行为。

## Open Questions

- 最近播放首版是否只展示“最近播放歌曲”，还是需要在子页面内显式露出切换到视频 / 歌单 / 专辑 / 播客的入口？
- “喜欢”子页面最终要承载“我喜欢的音乐歌单入口”还是“喜欢的歌曲列表”；当前先按复用歌单能力实现。
- 最近播放歌曲点击后，产品预期是进入播放器、触发播放、进入歌曲详情，还是先只展示不跳转？
