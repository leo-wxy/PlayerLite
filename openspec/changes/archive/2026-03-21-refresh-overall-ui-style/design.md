## Context

这次变更属于跨首页、搜索页、播放器展开页、播放列表半浮层与主壳层 minibar 的统一视觉改造，影响范围横跨 `ui/theme/`、`feature/main/`、`feature/player/ui/` 与 `feature/search/`。当前工程已经有一套 `MaterialTheme.colorScheme`，但现有色板仍是偏旧的品牌表达，且各页面大量直接依赖通用 `surface`、`surfaceVariant`、`primary` 组合，导致首页、搜索页、播放器和播放列表虽然功能已完成，视觉语言却没有形成统一的音乐产品风格。与此同时，`app` 与 `feature-search` 目前各自维护主题入口，页面里还混入了局部红色常量和 `copy(alpha = ...)` 组合；如果这次只做静态换色，后续想要继续扩品牌主题或支持换肤时，依旧会回到分散的页面级维护。

这次改造的约束也很明确：

- 这是样式统一，不是信息架构重写。首页主壳层继续保留双 Tab，底部导航只收窄整体宽度，不扩成四入口。
- 搜索仍沿用既有 `SearchRepository` / `SearchViewModel` / 独立搜索页流程，首页搜索入口只负责导航。
- 搜索结果页在快速切换结果类型时不能出现“当前 tab 短暂空白”的中间态，未完成加载的当前类型至少要稳定展示局部 loading。
- 搜索结果页的结果类型栏需要保持统一的宽度节奏，像 `MV` 这类短标签也不能因为文案更短而与中文两字标签错位。
- 首页发现流仍沿用 `HomeDiscoveryRepository` / `HomeViewModel` 数据边界，样式调整不改变接口映射和区块顺序。
- 播放器、播放列表和 minibar 继续复用现有播放状态、队列和交互语义，不在本次设计里改动播放规则。
- 参考图会以本地资源形式保留在 `openspec/changes/refresh-overall-ui-style/references/assets/`，并通过局部 `.gitignore` 避免进入提交。

受影响的主要协作者是后续实现 UI 的开发者，以及继续补充参考图、微调风格和验证效果的当前 change 维护者。

## Goals / Non-Goals

**Goals:**

- 建立一套围绕 `primary #e53935`、`secondary #616161`、`tertiary #0087a0`、`neutral #f9f9fb` 的共享主题 token，并明确它们在首页、搜索页、播放器和播放列表中的语义分工。
- 建立一套可换肤就绪的共享主题 contract，以 “brand palette -> semantic tokens -> page/component tokens” 的分层方式统一首页、搜索页、播放器和播放列表的颜色来源。
- 把首页发现流、顶部搜索入口、banner 推荐位、快捷入口、推荐卡片、纵向歌曲列表和底部 minibar 统一到同一视觉语言下。
- 把搜索页空查询态和搜索结果态统一到同一视觉体系，包括返回键、搜索框、历史标签、热门搜索榜单、结果类型切换栏和紧凑结果列表。
- 把播放器展开页与播放列表半浮层统一到同一主题方向，包括强调色、高亮项、操作按钮、背景基底与信息层级。
- 为后续实现保留稳定的分层：先落主题 token，再落共享组件皮肤，最后逐页套用，降低后续局部微调成本。

**Non-Goals:**

- 不改动首页发现流的数据来源、区块编排逻辑和点击目标契约。
- 不改动搜索接口调用、历史搜索持久化、热搜缓存、结果类型切换和结果跳转语义。
- 不改动播放列表队列规则、拖拽排序语义、播放模式语义与播放器状态流。
- 不在本轮把双 Tab 主壳层升级为四 Tab 或多入口信息架构。
- 不在本轮引入新的设计系统依赖、第三方主题库或全新导航框架。
- 不在本轮实现运行时换肤入口、皮肤选择页或皮肤持久化能力。

## Decisions

### Decision: 先建立可换肤就绪的共享主题 contract，而不是逐页直接写死颜色

采用方案：

- 新增轻量 `design-system` 共享模块，承载 app 与 `feature-search` 共用的主题 contract，避免继续维护两套平行 color scheme。
- 在共享模块里定义可扩展的 `brand palette`，先落默认皮肤使用的核心色板，再提供 light / dark 映射函数。
- 在主 `MaterialTheme.colorScheme` 之上，为本次 change 补充更贴近业务语义的 `semantic tokens` 与必要的 `page/component tokens`，例如首页搜索框底色、结果页弱分隔色、minibar 进度强调色、播放列表高亮行背景等。
- 页面与组件优先依赖这些语义 token，而不是每个界面直接拼接 `copy(alpha = ...)` 风格常量。

这样做的原因：

- 这次改造是跨多个 capability 的样式统一，逐页写死颜色会让后续微调变成多点同步。
- 工程已经采用 Material 3 主题，沿用现有主题入口并抽出共享主题 contract，能在不引入重型设计系统的前提下，为后续皮肤扩展保留稳定入口。

备选方案：

- 备选 A：直接在各页面局部写常量色值。好处是快，坏处是后续统一调色会变成全局搜改，且容易造成首页、搜索页、播放器出现同名不同色。
- 备选 B：新建一套完全独立于 `MaterialTheme` 的自定义设计系统。好处是理论上更纯粹，坏处是对当前工程过重，会把“样式调整”扩大成架构迁移。

### Decision: 样式落地按“共享 primitives -> 页面壳层 -> 列表项/卡片”三层推进

采用方案：

- 先整理通用视觉 primitives：圆角、阴影、描边、弱分隔线、胶囊按钮、搜索框容器、榜单容器、卡片标题/副标题层级。
- 再处理页面壳层：首页搜索框与 banner 外层、搜索页顶部搜索栏和类型栏、播放器主信息区和播放列表 sheet 头部。
- 最后处理页面内 item：首页推荐卡、纵向歌曲行、搜索历史 chip、热门搜索榜单行、搜索结果行、播放列表行与 minibar。

这样做的原因：

- 这次参考图虽然分属不同页面，但重复出现了圆角容器、轻底色、红色强调动作、浅灰背景和更紧凑的标题/副标题层级。
- 先统一 primitives，后面页面级微调时更容易保持一致，也更适合在 OpenSpec `specs` 里拆 requirement。

备选方案：

- 备选 A：逐张参考图按页面一次做完。坏处是容易在不同页面上重复实现同类视觉，后续再统一又要返工。

### Decision: 首页与搜索页保持现有行为边界，只替换视觉承载方式

采用方案：

- 首页发现流继续由 `HomeDiscoveryRepository` + `HomeViewModel` 提供数据，首页搜索入口只做导航。
- 搜索页与搜索结果页继续沿用独立搜索流；空查询态与结果态共享同一主题语义，但不把搜索数据逻辑混回首页。
- 搜索结果页继续沿用现有 `SearchViewModel` 和 `HorizontalPager` 承载结果类型，但切换类型时必须以原子状态更新和稳定 pager 同步避免当前页落回 `Idle` 空白。
- minibar 继续挂在 Home 域主壳层内，不改变“主体打开播放器、按钮独立响应、歌曲区支持横滑切歌”的现有语义。

这样做的原因：

- 记忆和现有 spec 都已经确认：首页和搜索的数据边界、Home 域播放器层级、双 Tab 壳层都是稳定约束。
- 这次目标是更快得到可看的样式结果；如果同时动数据流和交互架构，会显著放大风险。
- 搜索结果空白问题本质上是结果状态机与 pager 同步窗口，不需要改接口或搜索流程语义，直接在现有边界内收敛即可。

备选方案：

- 备选 A：把首页搜索、热搜、结果做成同页联动。视觉上可能更统一，但会直接越过当前 `search-page` capability 边界。
- 备选 B：把 minibar 抽离为全局独立浮层。会破坏现有 Home 域层级和过渡逻辑。

### Decision: 底部导航继续保留双 Tab，只收窄宽度和统一胶囊样式

采用方案：

- 沿用 `首页` / `我的` 双 Tab 主壳层，不增加新的一级入口。
- 调整底部导航容器的整体宽度、圆角、阴影与留白，让它更像参考图中的悬浮胶囊，但内部仍只承载两个入口。
- 若未来一级入口数量增加，再在新的 change 里重新定义宽度、分布与图标节奏。

这样做的原因：

- 用户已经明确要求先保留两个入口，只为后续可能扩容预留宽度调整空间。
- 现有 `user-center-tab-shell` spec 和既有 Home 域实现都建立在双 Tab 架构上，现在改入口数会把 style change 升级成 IA 变更。

备选方案：

- 备选 A：直接照参考图改成四个入口。好处是更像图，坏处是与当前产品结构冲突，且会连带影响路由、文案和状态恢复。

### Decision: 本地参考图采用 change 内局部目录管理，并默认忽略图片文件

采用方案：

- 在 `openspec/changes/refresh-overall-ui-style/references/` 下保留说明文档。
- 实际参考图统一放到 `references/assets/`。
- 用 change 内局部 `.gitignore` 忽略 `assets/*`，只保留目录结构和命名约定。

这样做的原因：

- 参考图后续还会增加，放在 active change 目录最便于和 `proposal/design/specs` 一起查看。
- 忽略实际图片文件可以避免本地参考素材误进入提交，同时保留工程内稳定路径。

备选方案：

- 备选 A：把图片放仓库根目录或 docs 静态资源目录。缺点是容易和可提交资产混在一起。
- 备选 B：完全不在仓库存放。缺点是后续对照困难，且多轮样式迭代容易丢上下文。

## Risks / Trade-offs

- [风险] 主题 token 改动可能影响非本次目标页面的通用组件观感
  → 缓解：优先通过页面语义 token 和局部 surface token 承载新风格，避免所有组件都直接继承强烈的全局变化。

- [风险] 新增共享主题模块会带来一层新的 Gradle 依赖关系
  → 缓解：模块范围仅限 Compose 主题 contract 与 token，不承载业务逻辑，也不引入新的第三方依赖。

- [风险] 首页、搜索页、播放器同时改样式，单次实现范围较大
  → 缓解：后续 tasks 按 capability 分批落地，先做可见壳层，再做 item 细节，不要求一口气把所有页面打磨到最终版。

- [风险] 参考图带来的视觉方向可能与真实数据内容长度不完全匹配
  → 缓解：在 spec 中明确标题、副标题、省略、跑马灯和固定高度等规则，保证真实数据下布局仍稳定。

- [风险] minibar 和底部导航同时变轻变窄，可能造成点击热区拥挤
  → 缓解：保持现有交互边界不变，并在实现时优先守住最小点击尺寸与按钮间距。

- [风险] 后续继续补参考图时，当前设计文档与最新视觉可能短暂不一致
  → 缓解：把可提交结论始终回写到 active change artifacts，参考图目录只作为本地对照，不把图片本身当作规格真源。

## Migration Plan

- 第一步：补齐 `specs`，把 `ui-theme-tokens` 作为新 capability 建出来，并把首页、搜索页、播放器、播放列表与主壳层的视觉要求拆入对应 capability。
- 第二步：创建轻量 `design-system` 共享主题模块，先落默认皮肤的 brand palette、light / dark color scheme 与 semantic tokens。
- 第三步：实现主题 token 与共享 primitives，尽量先完成颜色、圆角、描边、阴影和基础容器。
- 第四步：按页面分组套用到首页 / minibar、搜索页 / 搜索结果页、播放器 / 播放列表。
- 第五步：通过已有页面截图、轻量 UI 验证与必要的回归测试检查行为未被破坏。
- 回滚策略：若局部页面风格偏差过大，可先回退对应页面对新 token 的套用，而保留主题 token 和 specs 结构不动。

## Open Questions

- `neutral #f9f9fb` 在深色模式下是仅作为浅色方案主基底，还是要为 dark scheme 同时定义一套对位中性色映射，当前仍需在 specs 中写清。
- 未来真的要增加第二套皮肤时，是直接以静态 palette 扩展，还是再叠加用户设置与持久化，当前明确不在本轮实现范围内。
- 首页 banner、快捷入口、推荐歌单和纵向歌曲列表的视觉靠拢程度是否要统一到同一批次完成，还是允许先做壳层后做卡片细节，当前可在 tasks 阶段继续拆。
- 搜索结果页右侧更多操作入口是否只做视觉统一，还是还要同步统一弹出菜单皮肤，这次 proposal 尚未细化到菜单层。
