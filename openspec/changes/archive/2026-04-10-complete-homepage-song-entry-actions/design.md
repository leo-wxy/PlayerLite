## Context

首页发现接口已经返回真实的 `song` 资源块，例如 `HOMEPAGE_BLOCK_STYLE_RCMD` 和 `HOMEPAGE_BLOCK_NEW_ALBUM_NEW_SONG` 中的新歌资源，但当前首页实现仍按通用发现卡片处理它们。现状有三个断点：

- 首页 mapper 只把 `album / artist / playlist` 识别为站内详情实体，`song` 资源没有进入专门的站内行为分支。
- 首页没有针对歌曲 block 的独立展示结构，真实歌曲资源会退回通用推荐卡，而不是承载封面、歌曲信息和推荐理由的音乐卡片。
- 首页歌曲右侧“更多”按钮是空点击，占位大于能力。

另一方面，仓库里已经有两条可复用的能力链路：一条是 `DetailPlaybackGateway` 驱动的“用当前上下文替换队列并从指定 index 播放”，另一条是歌单 / 专辑 / 艺人详情页与每日推荐页已经采用的 `PlaylistItem` 在线条目投影方式。这意味着首页歌曲块不需要发明新的播放模式，只需要在首页域内补一个“歌曲资源块 -> 站内歌曲卡片入口”的映射与事件编排层。

除了功能断点，首页视觉也有明显问题：banner、歌单卡、快捷入口、状态卡和搜索框都在复用近似的白面板语义，导致首页从顶部到底部看起来像一串统一厚度的容器堆叠，而不是有明确主次的音乐推荐流。这次会在不打破现有设计系统的前提下，针对 Home 页做一次局部视觉收口。

## Goals / Non-Goals

**Goals:**
- 让首页 `song` 资源块稳定渲染为横向滑动的歌曲推荐卡片，而不是继续落回通用推荐卡。
- 每个首页歌曲卡片都展示封面，并以三行文本承载标题、歌手/专辑和推荐理由。
- 用户点击首页歌曲卡片时，系统以当前区块歌曲列表替换当前播放队列，并从被点击歌曲开始播放后打开播放器。
- 首页歌曲“更多”菜单首批支持“下一首播放”“查看专辑”“查看歌手”三种动作。
- 当首页歌曲缺少专辑、歌手或当前播放上下文时，系统给出明确降级结果，不保留无反馈入口。
- 拉开 Home 页顶部搜索、banner、歌曲模块、歌单模块和快捷入口的视觉层级，减少整页“统一白卡片”的平均感。

**Non-Goals:**
- 不为首页歌曲新增独立歌曲详情页。
- 不在本次 change 中补齐首页其他 block 的站内能力，例如热评、话题、播客、广播或音乐日历。
- 不在本次 change 中扩展首页歌曲“更多”到收藏、分享、加入歌单等完整动作集。
- 不重做用户中心、底部导航或播放器页视觉。

## Decisions

### 1. 首页歌曲资源采用“播放优先”动作，而不是复用 `SearchRouteTarget.Song`

首页歌曲资源的主动作定义为“播放当前区块上下文中的这首歌”，而不是“打开歌曲详情”。原因是当前仓库没有 song detail 路由，而首页接口的 `song` 资源天然已经携带构建播放列表所需的最小元数据。若继续复用 `SearchRouteTarget.Song`，主点击会再次落入不可打开兜底，无法解决首页最核心的断点。

备选方案是先补 song detail 路由，再让首页歌曲点进详情页。但这会把 P0 范围从首页闭环扩大到新详情页定义，不符合本轮目标。

### 2. 首页歌曲区块使用横向歌曲卡片，而不是纵向歌曲行

首页歌曲区块保持首页模块语义，使用横向滑动展示，而不是切成详情页风格的纵向行列表。每个歌曲卡片宽度约为屏幕宽度的 `60%`，并统一包含：
- 歌曲封面
- 标题
- 歌手 / 专辑
- 推荐理由
- 更多入口

这样做的原因是：
- 首页仍然是推荐流，横向滑动更符合首页模块节奏。
- `song` 资源与歌单 / 专辑卡片语义不同，需要独立视觉层级，但不应直接变成详情页列表。
- 封面和推荐理由会让首页歌曲块更像“推荐内容”而不是单纯文本操作入口。

备选方案是改成纵向紧凑歌曲行。这个方案信息密度更高，但首页氛围会明显更像列表页，不符合本轮期望。

### 3. 首页视觉采用“轻搜索 + 强 banner + 分层内容卡”方向，而不是继续统一白卡堆叠

这次 Home 页视觉会按内容语义分层，而不是继续让所有模块共享同一层白卡厚度：

- 顶部搜索入口改成更轻、更贴背景的悬浮搜索框，减弱它对首屏的白块占比。
- banner 保持主视觉英雄位，继续是最重的内容模块。
- 歌曲推荐区块改成更有节奏的横向歌曲卡片，强调封面和推荐理由。
- 推荐歌单区保持内容卡，但密度、圆角和底色会与歌曲卡、快捷入口拉开，不再所有模块看起来像同类容器。
- 快捷入口维持低高度功能入口风格，减少厚重感。

备选方案是维持现有布局，只替换颜色和圆角。这种成本低，但很难真正解决首页层级平均的问题。

### 4. 复用 `DetailPlaybackGateway` 完成首页歌曲区块的“整组替换队列 + 指定激活项播放”

首页歌曲块会像每日推荐和各类详情页一样，把当前区块歌曲列表一次性投影为 `PlaylistItem` 列表，并通过 `DetailPlaybackGateway` 发起播放。这样可以复用现有的队列替换、远端同步和播放器打开逻辑，避免在首页单独造一套播放入口。

首页歌曲区块对应的 `PlaylistItem` 需要补齐：
- 稳定 `id`
- `songId`
- 标题、歌手、专辑、封面、时长
- `primaryArtistId`、专辑 id
- `contextType / contextId / contextTitle`

这样“查看歌手 / 查看专辑”和后续可能的队列元数据回填都能沿用既有契约。

### 5. 首页歌曲“更多”使用最小动作菜单，不保留空入口

首页歌曲行右侧保留“更多”，但菜单范围只收敛到三项：
- 下一首播放
- 查看专辑
- 查看歌手

其中“查看专辑 / 查看歌手”是已有站内详情能力的复用；缺少目标 id 时对应菜单项直接隐藏或禁用，并给出稳定反馈，不做假入口。

菜单承载形式优先复用 Compose modal bottom sheet / action sheet 模式，由首页宿主统一分发事件。这样既不会把首页歌曲行做成多按钮杂糅，也能和现有播放器“更多操作”保持类似的交互语义。

### 6. “下一首播放”仅在当前存在有效播放上下文时生效

本次 change 为播放列表增加“把单首在线歌曲插入当前激活项之后”的能力，但它只在当前播放列表存在有效激活项时生效。若当前没有正在使用的播放上下文，则首页歌曲“下一首播放”给出明确提示，不隐式创建一个新的静默队列。

这样做的原因是：
- 语义更稳定，避免“下一首播放”在无播放上下文时退化成不可预期的入队行为。
- P0 不需要同时定义“空队列下的加入队列”产品规则。
- 实现可以聚焦在播放列表有序插入与当前播放项保持不变这条核心逻辑。

## Risks / Trade-offs

- [首页歌曲接口字段不稳定] → 统一从 `resourceExtInfo.song` / `songData`、`artists`、专辑对象中做多源兜底解析，并在缺字段时退回明确降级行为。
- [首页歌曲更多菜单动作会扩大范围] → 菜单严格限制为三项，不在本次 change 内继续扩成收藏、分享、加歌单。
- [“下一首播放”会触及播放列表核心规则] → requirement 明确只覆盖“有激活项时插入到下一位”这一条路径，并通过 `playlist-core` / `playback-orchestrator` 定向测试保护当前播放语义。
- [首页歌曲块和详情页播放入口重复实现] → 播放主链路统一复用 `DetailPlaybackGateway` 和 `PlaylistItem` 投影，只把首页特有的 mapper、横向卡片 UI 与事件壳层留在首页域。
- [首页视觉调整范围失控] → 只收口 Home 页发现流与顶部搜索入口，不重做用户中心、底部导航或播放器页。
