## Context

当前工程已经完成了本地/`content://`/通用 `http(s)` 音源播放、播放列表持久化，以及前后台播放分层；但这些能力默认假设“音源可直接访问”，并没有一套统一的用户身份体系来承载受登录态约束的在线播放接口。对于这次需求，真正要补齐的不是一个孤立的 `LoginActivity`，而是一条从“登录入口 -> 用户会话 -> 用户资料 -> 登录状态观察 -> 在线能力前置校验”的完整链路。

仓库现状也决定了这次设计需要控制边界：
- `app` 已经承担页面与 UI 投影职责，不适合继续堆叠登录协议、Cookie 管理与用户会话持久化。
- `playback-service` 当前关注播放执行与 `MediaSession` 映射，不应成为用户账号真相源。
- 现有持久化实践以轻量本地存储为主（如播放列表 `SharedPreferences`），尚未引入通用用户域模型。
- 用户已经明确：**本地播放不需要登录**，登录只是**在线播放能力的前置卡口**。

因此，这次设计的核心不只是“做一个登录页”，而是明确：
1. 哪个模块负责通用网络调用；
2. 哪个模块负责用户会话与用户资料；
3. 在线播放入口在哪里校验登录态；
4. 本地播放如何完全绕开这条登录链路。

## Goals / Non-Goals

**Goals:**
- 新增独立 `user` module，承载登录、会话持久化、用户详情、退出登录与登录状态监测。
- 新增独立 `network` 基础 module，统一承载通用网络客户端、认证拦截、Cookie/会话附着与错误映射。
- 在 `app` 中新增 `LoginActivity`，作为在线播放相关账号入口。
- 建立用户会话持久化与恢复能力，使应用重启后能够恢复当前登录态。
- 为在线播放能力提供统一的登录前置校验，未登录时拦截在线播放入口。
- 明确本地播放、已有本地列表恢复与其他不依赖账号的能力不受登录态影响。

**Non-Goals:**
- 本次不直接实现完整在线播放业务（如在线歌单、在线搜索、在线播放页完整 UI）。
- 本次不把所有在线业务接口一次性都接完；首阶段只围绕登录、会话、用户详情与在线播放前置校验建立基础设施。
- 本次不把用户会话真相源下沉到 `playback-service`；服务层只消费必要的认证投影或受保护请求能力。
- 本次不引入完整账号体系 UI（如注册、修改密码、找回密码、多账号切换），除非手机登录接口本身强依赖。

## Decisions

1. 新增 `:network-core`（名称可在实现阶段微调）作为通用网络基础模块。
   - 方案：抽出独立网络模块，负责 `OkHttpClient`、基础请求配置、统一错误转换、认证拦截与 Cookie/会话附着；`user` 与后续 `online-playback` 相关能力都通过它发起请求。
   - 原因：用户登录只是第一批在线接口，后续在线播放、歌单、搜索等在线能力大概率都会复用同一套网络基础设施。若把网络请求直接放进 `user` module，后续业务模块会反向依赖 `user`，边界会越来越乱。
   - 备选方案：
     - 直接把所有网络请求能力放进 `user` module`：短期快，长期会让 `user` 演变成“在线业务大杂烩”。
     - 先全部写在 `app`，后续再抽：会让登录卡口、Cookie 管理与会话恢复散落在 UI 和调用点里。

2. 新增 `:user` module 作为用户域真相源，而不是让 `app` 直接持有登录状态。
   - 方案：`user` module 负责定义 `UserSession` / `UserProfile` / `LoginState` 等模型，以及 `UserRepository`、登录用例、退出登录用例、用户详情读取、登录状态观察能力；`app` 只消费暴露出来的状态与命令。
   - 原因：用户态属于可复用业务域，不应该和单个页面生命周期绑定。独立模块后，后续在线播放入口、个人页、设置页都可复用同一套用户会话能力。
   - 备选方案：
     - 让 `PlayerViewModel` 或 `LoginActivity` 直接持有登录状态：会导致页面销毁与会话语义耦合。
     - 让 `playback-service` 成为登录态真相源：职责错误，且与服务进程生命周期耦合过深。

3. 用户会话采用“内存快照 + 本地持久化”双层模型。
   - 方案：`UserRepository` 进程内维护 `StateFlow<LoginState>` 作为实时状态，同时把必要的会话信息（例如持久 Cookie、用户基础信息、最近一次登录标识）序列化到本地存储；启动时先恢复本地会话，再按需拉取用户详情校验其有效性。
   - 原因：在线播放前置校验、UI 显示和退出登录都需要可观察状态；同时，应用重启后必须恢复用户态，不能要求每次重新登录。
   - 备选方案：
     - 只放内存：进程被杀后就丢失登录态。
     - 只在每次启动时临时探测服务端：会增加冷启动耦合，也无法作为统一 UI 状态源。

4. 显式定义 `UserInfo` 模型，并以接口返回字段为准做领域收敛。
   - 方案：在 `user` module 中定义明确的 `UserInfo` / `UserSession` / `LoginState` 模型，其中 `UserInfo` 以“手机登录返回结果 + 获取用户详情返回结果”的稳定字段并集为基础，优先收敛 `userId`、昵称、头像、账号标识、会员/等级等用户态展示与鉴权常用字段；接口原始响应与领域模型之间增加映射层，不让 UI 或调用方直接依赖松散 JSON。
   - 原因：用户已经明确要求“根据接口返回信息定义好 `UserInfo` model”。若没有统一模型，后续登录成功、用户详情读取、会话恢复与在线播放前置校验就会分别消费不同形态的数据结构，导致状态漂移。
   - 备选方案：
     - 直接在各调用点消费原始 JSON：开发快，但字段分散且难以维护。
     - 只保留登录响应，不单独建 `UserInfo`：会让用户详情接口返回无法稳定复用。

5. 本地持久化先采用轻量存储，而不引入重型数据库。
   - 方案：沿用仓库当前轻量持久化思路，使用 `SharedPreferences`（或等价轻量 KV 存储封装）存储用户会话与用户资料快照，配合 JSON 编码；接口稳定后再评估是否升级到更强的持久化方案。
   - 原因：本次用户数据结构相对简单，重点是会话恢复与清理，而不是复杂查询；使用轻量存储可以减少依赖与迁移成本。
   - 备选方案：
     - 直接引入 Room：过重，当前没有关系型查询诉求。
     - 直接引入 DataStore：也可行，但会引入新的基础设施与迁移成本；若实现阶段发现 Flow/事务性收益明显，再评估替换。

6. 登录态前置校验放在“在线播放入口”而不是“全局播放入口”。
   - 方案：把是否需要登录的判定收口在在线能力入口，例如在线资源选择、在线播放发起、受保护接口请求创建前；当请求来源是本地文件、`content://` 或已有可直接访问的匿名 URL 时，不走登录卡口。
   - 原因：用户已经明确“本地播放不需要登录”。因此登录校验必须是**在线播放特有前置条件**，而不是全局播放总开关。
   - 备选方案：
     - 所有播放都先检查登录：会错误阻断本地播放。
     - 完全不做入口校验，等网络层失败：用户体验差，也不利于在 UI 上做一致引导。

7. `LoginActivity` 只承载登录交互，不直接承担用户会话真相源。
   - 方案：`LoginActivity` 负责输入手机号登录所需字段、展示接口约束提示、提供“跳过”入口，并触发登录请求与结果反馈；真正的登录结果写入、会话持久化和用户详情刷新都由 `user` module 完成。
   - 原因：这样可以保持 Activity 轻量，并避免“登录页面关闭后状态丢失”的页面级设计问题。
   - 备选方案：
     - 在 `LoginActivity` 内直接做 Cookie 存储与资料缓存：实现会更快，但后续无法复用，也不符合仓库现有架构偏好。

11. 未登录启动应用时可先进入 `LoginActivity`，但允许用户跳过后进入主界面。
   - 方案：应用启动后在会话恢复完成且确认未登录时优先打开 `LoginActivity`；用户若选择“跳过”，则回到主界面继续使用本地播放。真正受保护的在线播放能力仍在独立入口处再次要求登录。
   - 原因：这样既能把登录能力前置暴露给用户，又不会违背“本地播放不需要登录”的核心约束。
   - 备选方案：
     - 启动应用后完全不提示登录：用户难以发现在线能力依赖账号。
     - 启动应用后强制登录且不允许跳过：会误伤本地播放路径。

12. `LoginActivity` 首屏采用浅色音乐产品风的欢迎页式布局，而不是工具型表单页。
   - 方案：页面主视觉与标题、副标题、登录表单整体垂直居中；`跳过` 作为右上角描边胶囊按钮并扩大点击区域；主标题固定为“登录后解锁在线播放”，副标题固定为“本地播放仍可直接使用”；去掉长说明块与首屏退出登录按钮。
   - 原因：当前登录页虽然功能完整，但视觉层级像调试表单，不符合播放器欢迎页气质。用户已经明确希望它整体更像音乐产品欢迎页，并且更方便单手点击跳过。
   - 备选方案：
     - 保留当前下半区卡片式表单，仅微调颜色与圆角：实现简单，但无法根治“整体都丑”的问题。
     - 用更重的插画与活动页视觉：辨识度高，但容易喧宾夺主，也不适合当前播放器主流程。

13. 登录入口支持手机号与邮箱两种账号方式切换，但保持同一欢迎页结构。
   - 方案：在登录操作区顶部提供二段切换 `手机号 / 邮箱`；手机号模式展示 `手机号 + 密码`，邮箱模式展示 `邮箱 + 密码`；登录按钮和跳过入口不因切换而改变布局。
   - 原因：用户已经明确要求可以切换邮箱和手机号登录，同时不希望破坏欢迎页的整体气质。把登录方式切换收在表单顶部，可以在不增加页面复杂度的前提下扩展能力。
   - 备选方案：
     - 将邮箱登录单独放到二级页面：流程更长，不利于登录页首屏完成操作。
     - 通过一行小字链接切换登录方式：实现轻，但不够正式，也容易降低可发现性。

14. 主界面右上角登录入口在已登录时显示真实头像，而不是继续显示固定人像图标。
   - 方案：前台 `UserSessionUiState` 增加 `avatarUrl` 投影；未登录时保持当前登录入口图标，已登录且头像可用时显示圆形在线头像，加载失败或头像缺失时回退为默认人像图标；点击行为保持不变，仍作为当前账户入口。
   - 原因：用户已经明确希望主界面右上角能展示在线头像，同时说明该入口暂时仍只是跳转入口，真正的二级页面后续再补。把它定义为“只改展示、不改交互”可以控制本次改动范围。
   - 备选方案：
     - 继续使用固定人像图标：实现最轻，但失去已登录态识别。
     - 头像旁再叠昵称或状态点：信息更多，但会破坏当前顶栏简洁度。

8. 在线请求统一通过认证拦截器附着会话，而不是让每个调用点手动拼 Cookie。
   - 方案：`network` module 提供统一认证附着能力，从 `user` module 的会话存储中读取当前认证信息，并在需要时附着到在线接口请求；退出登录时统一清空。
   - 原因：手动在各调用点拼接 Cookie/头信息会很快失控，而且会让登录态失效处理分散。
   - 备选方案：
     - 每个 API 手动传 Cookie：短期容易写，长期必然重复且容易漏。

9. 登录状态监测采用“恢复时校验 + 关键接口失败时降级”的双轨方式。
   - 方案：应用启动后若存在本地会话，优先恢复，再通过“获取用户详情”或等价轻量接口校验会话是否仍有效；在线请求若返回明确未登录/失效信号，则回写为未登录状态并触发统一清理。
   - 原因：仅靠本地快照无法知道会话是否已过期；仅靠每次在线播放失败再处理又过晚。
   - 备选方案：
     - 只看本地有没有会话：会出现“本地显示已登录，实际请求已失效”的假状态。
     - 每次启动强制重新登录：用户体验差。

10. 对 `playback-service` 的改动保持最小：不让它管理用户态，只让它消费已授权的在线请求能力。
   - 方案：服务层继续以“播放执行”为核心；如果后续在线播放资源在进入播放链路前已经完成登录态校验并具备可访问 URL/认证上下文，服务只负责按现有在线 source 机制播放。若未来需要服务端直接发受保护请求，再通过网络模块提供认证读取能力，而不是在服务里重复实现一套用户仓库。
   - 原因：服务层职责已经较重，把用户态也塞进去会破坏现有分层。
   - 备选方案：
     - 让服务层直接持有并管理登录态：会造成第二真相源，并增加跨进程同步复杂度。

## Risks / Trade-offs

- [新增两个模块会拉高初始集成成本] → 通过把 `network` 控制为纯基础设施、`user` 控制为纯用户域，避免未来再次返工。
- [轻量持久化保存会话信息存在结构演进成本] → 为会话存储增加版本字段与统一编解码入口，避免后续直接散落读写。
- [登录态恢复后可能与服务端真实状态不一致] → 启动恢复后补一次用户详情校验，并在关键接口返回未登录时立即降级清理。
- [在线播放前置卡口放错位置会误伤本地播放] → 明确只在在线入口做登录校验，不在通用播放入口做全局拦截。
- [接口文档细节可能包含额外约束（签名、验证码、Cookie 字段等）] → 设计阶段只锁定模块边界与状态流转；具体参数与错误码在 specs/实现阶段按文档精化。
- [跨进程播放未来若需要直接消费受保护接口，可能引入新的同步复杂度] → 当前先保持服务层只消费投影后的在线资源访问能力；若后续必须让服务层独立发认证请求，再在网络模块上扩展跨进程读取接口。

## Migration Plan

1. 新增 `network` 基础 module，并下沉通用 HTTP 客户端、认证附着与错误映射。
2. 新增 `user` module，定义用户模型、登录仓库、持久化存储与状态流。
3. 在 `app` 中新增 `LoginActivity` 与登录入口，并接入 `user` module。
4. 实现会话恢复、用户详情读取、退出登录与登录状态监测。
5. 在在线播放入口接入登录前置校验；未登录时引导到 `LoginActivity`，本地播放路径保持不变。
6. 若在线资源最终进入后台播放服务，再补充最小必要的认证投影接入，但不把用户态真相源迁入服务层。
7. 补充测试：持久化恢复、退出清理、详情读取、前置卡口与本地播放绕行。

回滚策略：
- 若登录流程接入失败，可先保留 `network` / `user` 基础模块但关闭在线播放前置入口；
- 若会话恢复存在问题，可暂时降级为“仅当前进程会话有效”，但不影响本地播放与已有播放链路；
- 若在线前置卡口误伤本地播放，应优先回滚入口校验逻辑，而不是回滚整个本地播放链路。

## Open Questions

- 参考接口文档中的“手机登录”最终采用哪些字段组合（手机号 + 密码 / 验证码 / 其他附加参数）？这需要在 specs 阶段按文档进一步收口。
- 会话持久化到底以 Cookie 为主，还是还需要额外保存 token / uid / csrf 等派生字段？需要根据接口文档与实际返回确认。
- `LoginActivity` 首版是否只做登录，不展示用户中心页；还是需要同时在某个入口展示当前用户详情与退出登录按钮？
- 在线播放入口的首版触发点具体放在哪里：独立页面、搜索入口，还是其他后续在线功能入口？
