## 1. 网络基础与用户域建模

- [x] 1.1 新增独立的 `network` 基础 module，统一承载 HTTP 客户端、基础请求配置、认证附着入口与错误映射。
- [x] 1.2 新增独立的 `user` module，定义 `UserInfo`、`UserSession`、`LoginState` 等领域模型，并完成手机登录响应与用户详情响应到稳定模型的映射。
- [x] 1.3 实现用户会话与 `UserInfo` 快照的本地持久化编解码，支持恢复必要的 Cookie、账号标识与用户资料。
- [x] 1.4 提供统一的 `UserRepository` / 用例接口，对外暴露登录、用户详情刷新、退出登录与登录状态观察能力。

## 2. 登录流程与会话恢复

- [x] 2.1 在 `app` 中完善 `LoginActivity` 与对应 UI 状态流，除手机号登录输入、提交、加载与失败提示外，还需要提供明确的“跳过”入口。
- [x] 2.2 接入手机登录接口，登录成功后建立用户会话并刷新 `UserInfo`，登录失败时保持本地播放与本地列表能力不受影响。
- [x] 2.3 在应用启动或依赖初始化阶段恢复本地用户会话，并通过用户详情接口或等价轻量校验确认当前登录态是否仍有效。
- [x] 2.4 实现主动退出登录与会话失效降级逻辑，统一清理持久化状态、同步前台登录状态并回退为未登录。
- [x] 2.5 将 `LoginActivity` 重构为浅色音乐产品风的欢迎页式登录页：右上角文字“跳过”、主视觉与登录操作区整体居中、首屏只保留简短登录说明。
- [x] 2.6 为 `LoginActivity` 增加手机号 / 邮箱登录方式切换，并分别对接对应的登录请求与输入校验。
- [x] 2.7 将主界面右上角登录入口改为“已登录展示真实头像、未登录展示默认登录图标”，并在头像缺失或加载失败时回退默认人像图标。

## 3. 在线播放前置卡口与播放链路协同

- [x] 3.1 未登录启动应用时优先进入 `LoginActivity`，允许用户点击“跳过”后进入主界面；真正受保护的在线播放能力仍在独立入口处要求登录。
- [x] 3.2 保证本地文件、`content://` 音源与已有本地播放列表恢复路径完全绕开登录校验，不残留错误的全局“禁止播放”状态。
- [x] 3.3 移除主界面当前临时的受保护在线播放测试入口与“未登录”占位展示，仅保留真正需要的登录入口或登录跳转时机；已授权的在线播放上下文与认证附着能力仍由前台业务层完成准入判断后再接入现有播放链路。
- [x] 3.4 在服务重建或授权缺失场景下，对受登录态约束的在线音源安全降级为空闲态，同时不影响已有本地播放会话。
