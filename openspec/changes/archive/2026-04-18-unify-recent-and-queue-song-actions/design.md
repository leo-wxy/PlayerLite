## Context

当前仓库已经把首页歌曲、本地歌曲和每日推荐歌曲统一到单曲次级入口，但最近播放和播放队列仍然停留在旧交互：最近播放缺少清晰的“主点击播放 / 次级入口详情”分工；播放列表半浮层右侧仍然是删除按钮。与此同时，播放队列共享模型 `PlaylistItem` 只有 `primaryArtistId`，没有 `albumId`，这让播放队列里的“查看专辑”没有稳定数据来源。本次改动需要同时触及用户中心、播放器队列 UI 和共享播放契约，属于跨模块但边界清晰的一次收口。

## Goals / Non-Goals

**Goals:**
- 让最近播放中的歌曲主点击稳定按当前最近播放列表直接开始播放，并把右侧三点收口为直达歌曲详情的弱入口。
- 让播放列表半浮层中的队列项使用队列语义的三点更多入口，承载 `查看歌曲详情 / 查看歌手 / 查看专辑 / 移出播放队列`。
- 为在线歌曲共享队列项补齐稳定 `albumId`，让播放队列动作不再依赖临时推断。
- 保持播放队列整行点击切歌的主路径不变。

**Non-Goals:**
- 不修改搜索结果页交互。
- 不把“下一首播放”带入当前播放队列项菜单。
- 不在本次统一收藏、分享或登录态相关次级动作。
- 不扩展到喜欢页歌曲列表或播放器全局更多浮层。

## Decisions

### 决策一：最近播放主点击直接播放当前列表，三点仅承载详情入口

最近播放列表更接近“最近听过的播放上下文”，主点击应该优先服务回到播放，而不是再次进入详情。因此最近播放整行点击改为按当前最近播放列表替换当前队列并从点击项开始播放；右侧三点则不再弹更多菜单，而是作为轻量详情入口，直接进入歌曲详情页。

### 决策二：播放队列使用队列语义的更多菜单，而不是照搬统一四动作

当前播放队列中的歌曲已经处在现有队列里，“下一首播放”在这里语义冲突。队列项更多菜单因此采用变体：`查看歌曲详情 / 查看歌手 / 查看专辑 / 移出播放队列`，既保留统一的详情跳转能力，又不引入重复或冲突动作。

### 决策三：共享 `PlaylistItem` 增加 `albumId`，不为播放队列额外再造一套歌曲模型

播放队列已经围绕 `PlaylistItem` 作为共享条目模型运转。若为了更多菜单再单独定义一套队列歌曲模型，只会扩大桥接成本。直接在 `PlaylistItem` 上补齐 `albumId`，可以让最近播放、搜索、首页、每日推荐、歌曲详情等已有队列投影路径继续复用同一共享模型。

### 决策四：最近播放单独使用歌曲专属 UI model，而不是继续把歌曲字段塞进通用收藏卡模型

`UserCenterCollectionItemUiModel` 已被喜欢页、自建歌单和收藏实体复用。最近播放这次需要稳定的 `songId / primaryArtistId / albumId`、直接播放上下文和详情入口，继续把歌曲专属字段塞回通用收藏模型会让边界再次发散。因此最近播放会升级成自己的歌曲列表项模型，而不是继续硬复用收藏卡模型。

## Risks / Trade-offs

- [风险] `PlaylistItem` 增加 `albumId` 会波及多处创建逻辑。 → 使用默认空值保持二进制兼容，并只补在线歌曲投影路径。
- [风险] 播放队列把删除按钮改成三点更多后，用户删除路径会变深一层。 → 将“移出播放队列”保留在菜单中，并在视觉上保持三点入口靠右稳定可见。
- [风险] 最近播放从通用卡模型拆出歌曲模型会增加一点仓库和测试改动。 → 换取最近播放播放上下文和详情入口边界清晰。
- [风险] 最近播放三点不再展示菜单，和首页/本地/每日推荐的更多入口不同。 → 在 spec 中明确最近播放次级入口承担“详情直达”而不是“统一菜单”，避免伪统一。
- [风险] 队列和来源列表的三点语义不完全一致，可能被误解为“不统一”。 → 在 spec 中明确“来源列表优先服务播放上下文，当前队列走队列语义变体”，避免伪统一。
