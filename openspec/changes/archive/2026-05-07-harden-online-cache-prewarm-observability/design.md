## Context

在线播放缓存、完整缓存命中和下一首预热横跨播放准备、缓存读写、预热调度和前台状态投影。当前实现已经具备基础能力，但仍需要把几个关键边界固化成可回归行为：

- 完整缓存存在时，当前播放不应再依赖在线 URL 解析或重新下载。
- 播放上下文变化后，旧预热任务的状态回调不得覆盖新上下文。
- 缓存与预热日志应能解释“为什么缓存条这样展示、为什么预热启动或跳过”，但不能长期输出低价值噪声。

## Goals

- 用测试覆盖完整缓存命中时的本地播放路径，避免网络解析失败导致已缓存歌曲无法起播。
- 用测试覆盖预热取消、上下文隔离和过期结果丢弃，避免旧任务污染当前状态。
- 收敛缓存进度、完整缓存命中和预热链路日志，保留关键诊断字段并减少重复日志。

## Non-Goals

- 不新增播放 UI 中的预热状态展示。
- 不改变现有播放控制 API 或跨进程 wire contract。
- 不引入新的日志框架或远程埋点系统。
- 不重做缓存淘汰策略和预热预算策略。

## Decisions

### 完整缓存命中以播放准备行为验证为准

完整缓存优先本地播放不是只看缓存库是否返回 complete，而是要验证播放准备链路在完整缓存可信时确实选择缓存源，并且不因为 URL 解析失败进入播放失败路径。测试应使用 fake cache / fake resolver 固定输入，避免依赖外部网络。

### 预热隔离使用上下文签名和 active job 双重门禁

预热任务发布状态时必须确认当前 active signature 仍匹配，并且发布者仍是当前 active job。切歌、队列变化、音源变化、音质变化等上下文切换后，旧任务即使晚到也只能被丢弃或记录为旧上下文取消，不能覆盖当前预热快照。

### 日志收敛按状态边界输出，不按读取循环输出

缓存进度和预热日志只在状态边界、关键决策或异常时输出。日志应包含稳定字段，例如 `key`、`trackId`、`quality`、`cachedBytes`、`totalBytes`、`progress`、`complete`、`reason`、`target`、`signature`。读取循环内的高频细节只保留在必要的调试日志中，避免影响长期排查效率。

## Risks / Trade-offs

- 过度收敛日志可能隐藏底层读取细节；保留关键异常和状态边界字段作为折中。
- 预热异步测试可能出现时序不稳定；测试应优先使用可控 fake 和 dispatcher，不依赖真实网络或 sleep。
- 完整缓存验证如果只测缓存层会产生假阳性，因此必须覆盖播放准备链路是否真的绕过网络解析失败。
