## Context（背景）

当前播放器流程偏向“即点即播”，但缺少可跨重启保留的播放列表一等状态。该变更将引入播放列表管理与持久化能力，会影响播放编排、启动初始化和存储处理。主要约束是：兼容现有单条目播放行为、在持久化数据过期或损坏时可安全恢复、并尽量控制运行时开销。

## Goals / Non-Goals（目标 / 非目标）

**Goals（目标）:**
- 定义统一的播放列表状态模型（有序条目、当前激活索引及必要元信息）。
- 将上一首/下一首等导航行为统一收敛到播放列表状态驱动。
- 实现本地持久化与启动恢复，并在恢复前执行数据校验。
- 通过“单条目映射为单元素播放列表”保持历史播放入口兼容。

**Non-Goals（非目标）:**
- 多端同步或云端播放列表同步。
- 账号体系、分享能力、协作编辑能力。
- 丰富的播放列表元数据管理（封面编辑、标签、文件夹等）。
- 与播放列表状态/持久化无关的播放引擎重构。

## Decisions（关键决策）

1. 统一状态形态：采用单一内存态 `PlaylistState`（items、currentIndex、metadata）作为播放列表与播放导航的唯一事实来源。
   - 原因：避免队列状态与播放器状态漂移。
   - 备选方案：维护独立队列指针与播放指针并定期对账；因同步复杂度高而放弃。

2. 持久化协议：使用独立 key（如 `playlist_state_v1`）存储带版本号的 JSON。
   - 原因：便于未来结构演进与显式迁移。
   - 备选方案：直接持久化内部对象；因与实现细节耦合过深而放弃。

3. 恢复策略：恢复前严格校验 payload，剔除无效条目、修正越界索引，失败时回退到安全空播放列表。
   - 原因：即使本地数据异常，用户仍能进入可用状态。
   - 备选方案：恢复失败即中断；因用户体验较差而放弃。

4. 播放集成：将单条目播放入口规范化为单元素播放列表，使历史流程与新流程共用同一套状态迁移。
   - 原因：减少分支逻辑并提升行为一致性。
   - 备选方案：保留单条目与播放列表两套逻辑；因状态迁移重复而放弃。

5. 写入策略：在播放列表变更时触发持久化，配合轻量防抖，并在可用生命周期节点进行兜底 flush。
   - 原因：平衡写入频率与数据可靠性，避免频繁同步写本地存储。
   - 备选方案：每次动作都同步写入；因性能与开销不优而放弃。

## Risks / Trade-offs（风险 / 权衡）

- [本地存储配额压力] -> 仅持久化必要字段，避免冗余媒体元数据。
- [结构版本演进风险] -> 引入版本字段与显式迁移处理逻辑。
- [媒体引用失效或缺失] -> 恢复时逐条校验并跳过无效项，保证整体可用。
- [初始化阶段竞态] -> 在依赖播放列表状态的导航指令生效前完成 hydration。
- [防抖写入在异常退出时丢失末次变更] -> 在可用生命周期钩子中补充强制 flush。

## Migration Plan（迁移计划）

1. 先引入播放列表领域状态与适配接口，不改变现有外部调用入口。
2. 实现带版本的持久化读写与恢复校验。
3. 将播放导航逻辑切换为基于播放列表状态迁移。
4. 在启动阶段完成状态恢复，并验证异常持久化数据的回退路径。
5. 如需回滚，可临时关闭恢复/持久化路径并退回纯内存播放行为。

## Open Questions（开放问题）

- 恢复时是否需要包含“每个条目的播放进度”，还是仅恢复顺序与当前激活项？
- 持久化播放列表规模是否需要设置上限，若需要，上限是多少？
- 对恢复时判定为无效的条目，是静默丢弃，还是提供非阻塞提示？
