## Context

当前仓库已经具备首页发现、搜索、歌手 / 专辑 / 歌单详情、独立播放器页与本地歌曲页等核心页面，但“歌曲”本身仍缺少稳定的一等对象承载。具体表现为：

- 搜索结果模型已经定义了 `SearchRouteTarget.Song`，但宿主路由只真正支持歌手、专辑和歌单。
- 最近播放歌曲仍然映射为“当前版本暂不支持打开最近播放歌曲”的占位动作。
- 播放器当前歌曲的分享与收藏还是 toast 占位，不是实际可执行能力。
- 本地歌曲页维护独立的 `LocalSongEntry` / `PlaylistItem` 路径，只把歌曲当作可播放条目，不具备统一详情承载。
- 仓库已有歌曲基础信息解析和歌曲百科能力，但还没有独立歌曲详情 feature 把这些能力组织成稳定页面。

这会让“歌曲”继续停留在播放上下文附属物，而不是可稳定进入、可回流、可承载动作的一级内容对象。另一方面，后续还计划继续处理统一“更多”能力；如果在歌曲对象未收口前直接处理“更多”，会把收藏、分享、查看详情、查看专辑/歌手等动作继续分散在各页面里。

## Goals / Non-Goals

**Goals:**

- 新增独立的歌曲详情页能力，将歌曲作为可直接进入和承载动作的一等对象。
- 引入统一歌曲引用模型，覆盖在线歌曲与本地歌曲，并允许按来源做能力降级。
- 让搜索结果中的歌曲、最近播放中的歌曲和播放器当前歌曲进入统一歌曲详情与动作闭环。
- 让本地歌曲在保持“直接播放”为主路径的前提下接入统一歌曲对象体系和详情承载。
- 将播放器当前歌曲的收藏与分享从占位提示改为真实可执行动作。

**Non-Goals:**

- 不在本次变更中实现下载 / 离线能力。
- 不在本次变更中完成统一跨页面的完整“更多”菜单设计与动作集合。
- 不顺手补齐 MV、专栏、播客、电台、视频等非歌曲内容类型的详情承载。
- 不重构现有播放编排主链路，不引入新的平行播放 runtime。

## Decisions

### Decision 1: 新增独立歌曲详情 feature，而不是继续把歌曲动作散落在现有页面

本次新增独立歌曲详情能力，推荐以 `:feature-song-detail` 形式承载页面、状态模型、repository contract 和详情动作模型。宿主 `app` 只负责 Activity 宿主和 route bridge。

这样做的原因是：

- 歌曲已经具备跨页面统一承载价值，继续在搜索页、最近播放页、播放器页分别补丁式接动作，会重新把歌曲能力做散。
- 现有歌手 / 专辑 / 歌单详情已按独立 feature 组织，歌曲详情继续放在 `app` 内会让详情体系不对称。
- 后续要处理完整“更多”能力时，歌曲详情页和歌曲对象可以作为稳定动作挂点。

备选方案是先不做独立歌曲详情页，只补“点击即播”和分享 / 收藏动作。该方案交付更快，但会把歌曲继续停留在“只有动作、没有一级页面对象”的状态，因此不采用。

### Decision 2: 统一的是歌曲引用对象，不是强行把本地歌曲伪装成在线歌曲

本次引入统一歌曲引用模型，例如 `SongRef`：

- `Online(songId)`
- `Local(playbackUri)`

所有与歌曲相关的页面入口和动作回调，先落到统一 `SongRef`，再由详情页或宿主动作处理器根据来源分流。在线歌曲与本地歌曲共用页面壳和基础动作模型，但按来源降级：

- 在线歌曲支持百科、收藏、分享、查看专辑、查看歌手。
- 本地歌曲支持基础信息、播放、加入队列和可执行的本地分享；没有稳定在线 id 的动作不展示。

这样可以让本地歌曲接入统一详情体系，而不会伪造不存在的在线能力。备选方案是只让在线歌曲进入详情页、本地歌曲继续维持完全独立路径。该方案实现更简单，但会让“歌曲对象统一”只做一半，因此不采用。

### Decision 3: 各入口统一汇到 SongRef，但主点击语义按场景保留

本次不把所有歌曲入口的主点击都改成同一种行为，而是统一对象、保留场景心智：

- 搜索结果里的歌曲：主点击进入歌曲详情页。
- 最近播放里的歌曲：主点击进入歌曲详情页。
- 本地歌曲页：主点击仍然直接播放，另提供进入歌曲详情页的次级入口。
- 首页歌曲推荐区块：主点击继续保持当前 block 播放闭环，不改成进详情页。
- 播放器当前歌曲：新增稳定的“查看歌曲详情”入口。

这样做的原因是：搜索和最近播放更偏内容回流入口；本地歌曲页和首页歌曲推荐区块更偏“立即播放”入口。统一对象不等于强行统一所有场景的主点击语义。

### Decision 4: 详情数据优先复用现有歌曲基础信息与百科链路，不新建第二套歌曲域接口

在线歌曲详情页的数据来源优先复用仓库已有地基：

- 基础信息复用现有歌曲详情解析能力
- 歌曲百科复用 `/song/wiki/summary`
- 播放动作复用现有 `PlayerRuntime` / `PlaylistController` / detail playback gateway

本地歌曲详情页直接使用本地扫描结果和当前 `PlayableItem` 快照，不额外请求在线详情接口。

这样可以避免为歌曲详情页再造一套平行的歌曲域模型，也能让本次工作聚焦在“对象与页面闭环”，而不是扩散成新的数据层重写。

### Decision 5: “更多”能力只保留最小真实动作，完整统一作为后续变更

本次不会试图定义所有页面、所有歌曲来源的完整“更多”菜单。当前 change 只要求：

- 详情页和播放器当前歌曲暴露真实可执行的核心动作
- 现有占位的分享 / 收藏动作变成真实动作
- 现有已存在的歌曲次级动作入口不再返回占位提示

后续统一“更多”菜单时，再以歌曲详情页和 SongRef 作为稳定挂点，统一动作集合、展示位置和跨页面一致性。

## Risks / Trade-offs

- [Risk] 在线 / 本地歌曲共用详情页壳后，页面状态分支可能变多  
  → Mitigation：在状态模型里显式区分 `OnlineSongDetail` 与 `LocalSongDetail` 能力面，而不是靠页面层到处判空。

- [Risk] 搜索、最近播放和播放器页同时接入 SongRef，会增加一轮路由与动作翻译改动  
  → Mitigation：把 Song route 与详情入口收成稳定 bridge，避免每个页面自己拼 `Intent`。

- [Risk] 本地歌曲加入统一对象后，用户可能期待与在线歌曲完全相同的收藏 / 百科能力  
  → Mitigation：详情页按来源做明确动作显隐，不展示无法真实执行的在线能力。

- [Risk] 本次不一并完成完整“更多”能力，后续仍需要第二条 change 补齐动作集合  
  → Mitigation：本次先把歌曲对象与详情页立住，为下一条“更多”能力变更提供稳定挂点。

- [Risk] 播放器当前歌曲的真实收藏动作依赖在线歌曲具备稳定 songId  
  → Mitigation：只有存在稳定在线 songId 时才展示在线收藏入口；本地歌曲不伪造在线收藏能力。

## Migration Plan

1. 新增歌曲详情 feature 边界
   - 建立统一 `SongRef`、详情状态模型、repository contract 与 route 入口
   - 接入 Activity 宿主与 bridge

2. 先打通只读回流入口
   - 搜索结果歌曲点击进入详情页
   - 最近播放歌曲点击进入详情页
   - 本地歌曲接入统一对象体系和次级详情入口

3. 再打通动作闭环
   - 详情页接入播放、加入队列、查看专辑 / 歌手
   - 播放器当前歌曲接入查看详情、分享与收藏真实动作

4. 回归与验证
   - 保证首页歌曲主点击、本地歌曲主点击和现有播放链路不回归
   - 编译和关键测试通过后再进入后续“更多”能力变更

5. 回滚策略
   - 若歌曲详情 feature 接入导致多入口路由不稳定，可先保留 SongRef 和详情模块实现，仅回滚搜索 / 最近播放 / 播放器页入口切换
   - 优先回滚入口接线，不回滚统一对象模型与详情 feature 基础设施

## Open Questions

- 在线歌曲的真实收藏动作最终应落到“喜欢的音乐”还是独立收藏接口，需要以现有接口能力和产品预期再核一次。
- 本地歌曲详情页是否需要首版就补系统分享，还是先只保证详情与播放闭环。
- 歌曲详情页是否应在首版承载评论数、分享数等动态元信息，还是只承载基础信息与百科摘要。
