## Context

当前仓库已经完成播放器主链路、后台播放服务、在线音质切换、source adapter、最近播放和播放队列等基础能力。后续如果继续扩发现流、下载中心或社区能力，但播放、在线资源和队列三条主链路不稳定，会把失败面放大到更多入口。因此本次先把长期播放器版本第一阶段收口为三个闭环：

- 播放闭环：断点续播、切歌稳定、缓冲 / 失败重试、弱网恢复。
- 在线资源闭环：搜索落到稳定可播上下文、音质切换、版权失效 / 下架兜底、URL 过期刷新。
- 队列与歌单闭环：下一首、最近播放、收藏 / 歌单、单曲来源切换。

本次设计沿用现有模块边界，不新增大型产品面，也不引入重型多源聚合。

## Goals / Non-Goals

**Goals:**
- 让播放服务和 app 侧编排围绕同一个播放目标 generation 工作，避免旧 source completion、旧快照或重复 prepare 覆盖当前曲目状态。
- 让在线播放准备层统一处理资源失效、URL 过期和主备 source 单曲兜底，而不是把分支散到页面层。
- 让音质切换继续按同一首歌、同一播放位置重准备，并在资源刷新后保持当前播放上下文。
- 让播放队列、最近播放和单曲来源切换有明确边界，避免短期播放上下文污染长期收藏 / 歌单资产。

**Non-Goals:**
- 不实现完整下载中心、批量下载、断点续传或 Wi-Fi 下载策略。
- 不做跨设备同步或账号云资产建模。
- 不扩展重发现流、排行榜、社区、评论、MV、播客或电台。
- 不做多音源聚合搜索，也不自动批量替换整条歌单来源。

## Decisions

### 决策一：先稳定播放目标，再处理资源和队列语义

播放闭环是另外两个闭环的地基。若切歌、恢复或弱网重试仍可能被旧事件覆盖，那么 URL 刷新、主备 source 兜底和最近播放记录都会出现误判。因此实现顺序先收口播放目标 generation、恢复快照和 retry / failed 投影，再处理在线资源兜底，最后统一队列和最近播放边界。

备选方案是按页面入口修复搜索页、详情页和播放器页各自的问题，但这样会让重试、兜底和来源切换逻辑散落在 UI 层，后续仍要回收。

### 决策二：资源失效统一归到在线播放准备层

搜索、详情和推荐入口只负责产出统一歌曲引用或共享播放队列项。URL 过期、无版权、下架、无直链、试听片段和主 source 失败都由在线播放准备层解释，并向播放服务返回可恢复失败、不可恢复失败或备用 source 成功结果。

备选方案是在搜索结果或详情页提前判断版权和直链可用性，但页面层无法覆盖恢复播放、切歌、后台服务重建和音质切换等非页面入口。

### 决策三：主备 source 只做单曲级兜底，不改全局当前 source

当前版本采用“主音源 + 备用源”策略，但兜底只针对当前单曲的一次解析或播放会话生效。成功兜底后，共享队列项可记录当前单曲的来源 override 以支持恢复；全局当前 source、整条歌单、收藏和历史资产不被静默改写。

备选方案是主 source 失败后直接把全局当前 source 切到备用 source，但这会让设置页状态、后续搜索、历史记录和歌单来源都被一次单曲失败污染。

### 决策四：最近播放以成功开播为准

最近播放应代表本机真实听过的内容，而不是点击过的内容。点击搜索结果、详情页歌曲或下一首播放失败时，不写入最近播放；只有播放服务确认当前目标进入可播放执行态后，app 侧再写入本机最近播放缓存。

备选方案是点击即写入，实现成本低，但会把版权失效、URL 过期失败和备用 source 兜底失败都记录为“最近听过”，与播放器型产品定位冲突。

## Risks / Trade-offs

- [风险] 播放 target generation 会触及 app 侧编排、service runtime 和 native 回调边界。 → 先补旧 completion / 重复 prepare / 切歌回跳的定向测试，再分层收口。
- [风险] 资源失败分类不足时，可能把登录失效、版权不可用和网络抖动混成同一种错误。 → 在线准备层先定义最小错误分类，再映射到 retry、refresh URL、备用 source 或最终失败。
- [风险] 单曲来源 override 持久化过重会提前变成多源库管理。 → 只保留当前单曲恢复所需的最小 source 上下文，不做全量来源资产建模。
- [风险] 最近播放改为成功开播后记录，UI 点击后列表更新不再即时。 → 以真实播放事件为准，避免失败点击污染本机历史。
- [风险] 本次 change 跨多个现有 capability。 → 实施顺序固定为播放闭环、在线资源闭环、队列与歌单闭环，每个阶段都有定向验证。
