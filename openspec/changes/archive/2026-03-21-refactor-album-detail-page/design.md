## Context

当前专辑详情页原本是单条 `LazyColumn`：头部 hero、动态信息、歌曲列表顺序直排。这个结构可以承载基础信息，但不具备歌手详情页已经验证过的交互能力：头部与内容区的连续滚动接力、sticky tab、横向 pager 与纵向列表的手势分流，以及各 tab 独立滚动位置。

这次重构的直接目标不是扩展数据源，而是先把专辑详情页的交互骨架对齐到歌手详情页已经跑通的模式，同时保持现有 `AlbumDetailViewModel`、`AlbumDetailRepository`、分页加载和播放行为不回退。用户已经明确专辑简介不再走浮层，而是作为单独 tab 承载。

## Goals / Non-Goals

**Goals:**
- 让专辑详情页采用与歌手详情页一致的交互骨架：外层头部、sticky tab、`HorizontalPager`、内外层滚动接力。
- 将专辑简介从原先的点击浮层切换为独立的“简介” tab，与“歌曲” tab 并列承载。
- 复用已验证的 `DetailVerticalScrollHandoff`，统一专辑页与歌手页的纵向接力和横向手势判定。
- 保留现有专辑详情的数据状态、分页加载和播放入口，不在本轮重构中引入额外数据层改造。

**Non-Goals:**
- 本轮不接入 `/digitalAlbum/detail`、`/digitalAlbum/sales`、`/ugc/album/get` 等新增接口。
- 本轮不重写 `AlbumDetailViewModel` 或 `AlbumDetailRepository`。
- 本轮不补专辑详情测试，也不处理与当前交互骨架无关的视觉精修。

## Decisions

### 1. 继续使用 `MusicDetailScaffold` 作为外层容器，避免新建第二套详情壳层
专辑页仍使用 `MusicDetailScaffold + bodyListState` 承载 hero 和 sticky tab，只把正文部分替换成 pager。这样可以最大程度复用现有详情页基础能力，并把改动收敛在 `AlbumDetailActivity.kt` 内。

备选方案是完全照搬歌手页的 overlay top bar 与自定义 shell，但这会显著放大改动范围，也会把当前问题从滚动链路扩散到状态栏、top chrome 和更多视觉细节。

### 2. 专辑页固定为两个 tab：`歌曲` 与 `简介`
`歌曲` tab 承载播放全部入口和分页歌曲列表；`简介` tab 只承载专辑简介正文。动态元信息上移到外层头部底部、sticky tab 上方，和 header 一起滚动。这样既符合用户已经明确的交互要求，也能保持 tab 语义直接、稳定。

头部动作区保留第二个胶囊位，但当前只展示评论数摘要，不直接承载评论页跳转。这样可以先稳定头部信息层次，后续再在同一位置补评论入口而不必重新调整整体布局。

备选方案是引入第三个 `动态` tab，但当前动态信息只有一组统计值，拆成独立 tab 只会增加手势与状态复杂度，没有足够收益。

### 3. 复用共享的 `DetailVerticalScrollHandoff`
专辑页不再实现一套新的 nested scroll 规则，而是直接复用歌手页已经验证过的 handoff helper，包括：
- 上滑时外层头部优先消费，再把剩余位移交给当前 tab 列表
- 下滑时先让当前 tab 列表回顶，再由外层头部接手展开
- 横向手势使用相同偏置，避免轻微斜向滑动误判为纵向滚动

这保证了歌手页和专辑页后续复用同一套交互约束，减少分叉维护。

### 4. 头部折叠只基于 hero，简介内容只留在简介 tab
专辑页的外层折叠距离只根据 hero item 的剩余可见高度计算，不再把正文或简介内容混进折叠模型。头部不再重复展示简介摘要，也不再保留单独的简介入口；简介正文只在“简介”tab 中承载，避免头部和 tab 重复表达同一内容。

这样做的原因有两个：
- 先避免重演歌手页之前“中间内容影响折叠距离”的问题
- 简介既然已经提升为 tab，就应该进入同一套详情上下文，而不是再开一层独立浮层

## Risks / Trade-offs

- [顶部预留区视觉可能偏空] → 当前专辑页复用了 sticky tab 的顶部避让逻辑，但没有完整引入歌手页的 collapsing top bar；如果真机上顶部留白感明显，再单独补 top chrome。
- [头部内容继续膨胀] → 动态卡片已经上移到 header，若后续继续往头部塞正文信息，会重新把折叠模型和内容模型混在一起，因此保持 header 只承载概览信息与动作入口。
- [简介入口可发现性下降] → 头部移除了单独的简介按钮，简介内容完全通过 tab 进入；这提升了页面收敛度，但依赖用户理解 `简介` tab 的存在。
- [数据层能力仍落后于 proposal] → 这次只完成了交互骨架对齐，proposal 中提到的数字专辑、销量、登录态百科仍待后续单独落地。
