## Context

当前首页底部播放入口由 `HomePlayEntryCard` 承载，形态是“当前播放 + 文件名 + 状态文案 + 进入播放页按钮”的卡片式结构，位于 [MainShellScreen.kt](/Users/wxy/Projects/player-lite/app/src/main/java/com/wxy/playerlite/feature/main/MainShellScreen.kt)。它已经能表达“这里可以进入播放页”，但离用户想要的 QQ 音乐式 minibar 还很远：没有左侧封面、没有胶囊式单行承载、没有右侧局部控件区，也无法在首页直接完成播放/暂停和列表相关操作。

当前 Home 域的导航边界是清晰的：
- 首页概览态和播放展开态都属于同一个 `HomeSurfaceMode`，由 [MainShellState.kt](/Users/wxy/Projects/player-lite/app/src/main/java/com/wxy/playerlite/feature/main/MainShellState.kt) 控制。
- 点击首页播放入口时，`MainActivity` 通过 `shellState.openPlayer()` 进入 Home 域内的播放展开页，而不是跳到独立路由。
- 播放列表面板 `PlaylistBottomSheet` 目前只挂在 `PlayerScreen` 内部，因此首页概览态还没有直接打开列表面板的能力。

当前播放 UI 状态也有明确边界：
- `PlayerUiState` 已经提供首页迷你条做基础交互所需的大部分字段：`selectedFileName`、`currentTrackTitle`、`currentTrackArtist`、`currentCoverUrl`、`playbackState`、`displayedSeekMs`、`durationMs`、`playlistItems`、`activePlaylistIndex`、`showPlaylistSheet` 等。
- 这些 `currentTrack*` 字段已经是播放器运行时基于 `MusicInfo` / `PlayableItemSnapshot` 投影出来的当前播放上下文，不需要首页再重新发明一套元数据来源。
- 播放页已经有 `resolvePlayerTrackText(...)`，可以在运行时字段缺失时作为本地文件名兜底解析。

用户这轮又补了两个关键约束：
- 样式方向可以大部分参考 QQ 音乐底部 minibar，尤其是胶囊容器、左侧封面、中间当前歌曲信息区和右侧组合控件区。
- 封面元数据问题由另一个任务处理，因此本 change 可以先实现 UI；如果当前播放项没有真实封面，首版允许置空或使用占位封面，不在这次顺手扩展 `PlaylistItem`、`MusicInfo` 或播放元数据链路。

## Goals / Non-Goals

**Goals:**
- 将首页底部播放入口从卡片式 CTA 改为更接近 QQ 音乐底部 minibar 的胶囊式迷你播放条。
- 左侧承载封面，中间承载歌曲名和歌手信息，右侧承载“进度 + 播放/暂停 + 列表入口”的组合控件区。
- 当歌曲名过长时，中间歌曲信息区以单行跑马灯承载标题，不拉高迷你条高度。
- 点击整条迷你播放条时进入 Home 域内的播放展开页。
- 在 minibar 中间歌曲展示区域补齐左右横滑切歌手势，左滑下一首、右滑上一首。
- 点击右侧播放/暂停与列表入口时执行各自的局部动作，不与整条点击手势冲突。
- 在不修改播放业务真相和不引入新的播放元数据模型的前提下，先把首页 UI 和交互闭环做出来。
- 让首页迷你播放条与现有首页发现内容、搜索框和播放展开页风格更一致，而不是继续保留一块独立的大按钮卡。

**Non-Goals:**
- 本次不解决真实封面、真实歌手名、完整在线元数据的长期来源问题，这部分由另一个任务负责。
- 本次不修改 `PlaylistItem`、`MusicInfo`、播放服务共享契约或播放列表持久化格式。
- 本次不重做播放展开页本体布局，重点仅限首页概览态底部迷你播放条及其相关入口接线。
- 本次不新增新的播放列表能力，只复用现有 `PlaylistBottomSheet` 和已有播放/暂停、队列操作回调。
- 本次不要求 1:1 还原 QQ 音乐，每一处视觉细节仍要适配当前应用已有样式体系。

## Decisions

### 1. 首页播放入口从 `HomePlayEntryCard` 收敛为单行胶囊式 `HomeMiniPlayerBar`

首页现有的播放入口会从纵向卡片改成单行胶囊条，布局固定为三段：
- 左侧：封面容器；
- 中间：歌曲名与歌手信息；
- 右侧：组合控件区。

这样做的原因是：
- 用户给出的参考图本质上是“持续驻留的迷你播放器”，不是“一个说明卡片加跳转按钮”。
- 现有入口占高较大，而且信息分散在 badge、两段文案和按钮上，视觉重心不稳定。
- 单行胶囊条更容易贴近当前首页底部浮起的产品感，也更适合与播放展开页形成共享视觉锚点。

备选方案：
- 继续保留当前卡片结构，只在底部按钮周围补图标和进度。这样改动小，但视觉上仍然像“卡片 + CTA”，不够像 minibar。
- 把迷你条放到系统底栏上方的独立悬浮层。这样更像常见音乐 App，但会额外牵动底栏与页面内容避让，不适合作为本 change 的首版。

因此本次选择在当前首页内容区域底部，将入口重构为单行胶囊式 `HomeMiniPlayerBar`。

### 2. 迷你条展示优先复用运行时播放上下文，文件名解析只作为兜底

首版不会新增新的播放元数据模型，而是直接复用现有数据：
- 标题与歌手：优先使用 `PlayerUiState.currentTrackTitle / currentTrackArtist`，也就是运行时已经从 `MusicInfo` / 播放快照同步过来的字段；仅在这些字段缺失时，才回退到 `resolvePlayerTrackText(playerState.selectedFileName)`。
- 进度：直接由 `displayedSeekMs / durationMs` 计算比例。
- 播放状态：直接由 `playbackState` 映射为播放中、暂停中或不可播。
- 封面：优先使用 `PlayerUiState.currentCoverUrl`，若当前播放上下文没有稳定封面，再回退到活动播放项上的封面或默认占位封面。

这样做的原因是：
- 用户这轮又明确提出“minibar 展示内容需要和 `musicinfo` 做下结合”，因此首页不能继续只靠 `selectedFileName` 做字符串猜测。
- 当前 `PlayerUiState` 已经承载了运行时同步好的标题、歌手与封面字段，不需要首页为同一首歌再做第二套推断逻辑。
- 仍然保留文件名解析作为兜底，可以兼容本地文件和缺失 metadata 的场景，而不扩大 `MusicInfo` 契约改动范围。

备选方案：
- 这次一并扩展共享 `MusicInfo` 契约或重新设计运行时元数据同步。展示会更完整，但会明显扩大范围，并和另一个任务重叠。
- 没有真实封面时直接隐藏左侧封面区域。这样实现更简单，但会破坏用户明确要求的左侧封面布局。

因此本次选择“优先复用运行时播放上下文，文件名解析只做兜底，封面首版允许占位”。

### 3. 右侧组合控件区采用“环形进度包裹播放/暂停按钮 + 邻接列表按钮”的结构

右侧控件区会靠近参考图，但保留当前需求里的进度表达：
- 一个圆形主按钮作为播放/暂停入口；
- 圆形按钮外围用环形进度表达当前播放进度；
- 圆形按钮右侧放一个更轻的列表按钮；
- 两者作为一个视觉分组共同落在迷你条右端。

这样做的原因是：
- 用户已经把需求明确成“进度和列表都放一起”，这比最初的“单个圆形进度”更接近参考图。
- 用环形进度包裹播放/暂停按钮，可以同时满足“像 QQ 音乐的右侧主按钮”与“保留进度可视化”这两点。
- 列表按钮与主按钮分开，可以减少误触，并且保留参考图里“右侧两个相邻控件”的节奏。

备选方案：
- 把进度单独画成一整圈，再把列表入口叠进同一个圆里。视觉会过满，点击区域也更难切清。
- 完全照抄参考图，只保留暂停按钮和列表按钮，不显示进度。实现更简单，但会丢掉你前面明确要求的进度表达。

因此本次选择“环形进度 + 主播放按钮 + 邻接列表按钮”的右侧组合控件区。

### 4. 整条迷你条与中间歌曲区、右侧局部控件采用显式手势分区

交互边界会明确切成两层：
- 点击迷你条主体区域：进入 Home 域内的播放展开页。
- 在迷你条中间歌曲展示区域左右横滑：切换上一首 / 下一首。
- 点击右侧播放/暂停按钮：仅切换播放/暂停。
- 点击右侧列表按钮：仅打开播放列表相关面板。

这样做的原因是：
- 这是用户明确提出的交互要求。
- 如果整条条目都是一个大的 `clickable`，右侧局部控件很容易被父点击吞掉。
- 右侧是高频操作区，必须优先保证播放/暂停和列表的点击命中稳定。
- 横滑切歌应该复用现有播放器上一首 / 下一首回调，而不是额外发明一条新的切歌状态链路。
- 用户最新澄清已经把手势范围限定在“歌曲展示部分”，因此封面区只承担展示与点击进入播放页，不承担左右横滑切歌。

备选方案：
- 点击右侧任意区域都先进入播放页，再在播放页里操作。这样实现最省，但会直接违背迷你条右侧局部操作的意义。
- 让整条条目都不可点，只保留右侧按钮。这样会损失首页到播放页的主要入口。
- 把横滑手势挂到整条 minibar，包括封面区和右侧按钮区。这样实现简单，但会显著增加误触和手势冲突。

因此本次选择“主体点击仍可进入播放页，中间歌曲展示区承载横滑切歌，右侧局部控件独立消费点击”的分区结构。

### 5. 歌曲展示区横滑切歌采用阈值触发，并提供轻量位移反馈

横滑手势落在 minibar 中间歌曲展示区时，组件会提供一个轻量的横向位移反馈；当拖动距离超过阈值后：
- 左滑触发 `下一首`；
- 右滑触发 `上一首`；
- 若当前没有可切换的上一首或下一首，则仅回弹，不触发切歌。

这样做的原因是：
- 用户需要的是“左右横滑切歌的效果”，不是纯粹的隐形手势；轻量位移反馈可以让交互更可感知。
- 采用阈值触发能避免普通点按时手指轻微抖动就误切歌。
- 现有播放器已经有稳定的 `skipToPreviousTrack` / `skipToNextTrack` 回调，minibar 只需要把手势映射过去即可。

备选方案：
- 用一小段水平位移就立刻切歌。响应快，但误触风险高。
- 做完整的滑块式揭示动画。效果更强，但实现和验证成本更高，不适合这一轮增量需求。

因此本次选择“歌曲展示区轻量位移反馈 + 阈值触发 + 回弹复位”。

### 6. 长歌名通过单行跑马灯承载，而不是继续静态省略

中间歌曲标题在首版仍保持单行，但当标题长度或排版宽度超过歌曲展示区域可用空间时：
- 标题启用单行跑马灯滚动；
- 歌手信息仍保持次级文本和单行静态展示；
- 迷你条整体高度、右侧控件组宽度与封面尺寸不因长标题发生抖动。

这样做的原因是：
- 用户已经明确要求“歌名过长要加跑马灯效果”，因此仅靠 `Ellipsis` 会损失核心信息。
- 迷你条高度有限，二行扩展会明显破坏参考图的胶囊节奏。
- 跑马灯只作用在标题层，可以保住当前“标题 > 歌手”的信息层级而不挤压右侧控件区。

备选方案：
- 继续保持单行省略。实现最简单，但直接违背用户新增要求。
- 改成两行标题/歌手动态增高。信息更完整，但会破坏首页悬浮迷你条的稳定高度。

因此本次选择“长标题单行跑马灯，歌手信息单行静态承载”。

### 6. 播放列表面板从 `PlayerScreen` 内部提升到 Home 壳层共享承载

因为首页迷你条也需要打开列表入口，而 `PlaylistBottomSheet` 当前只挂在 `PlayerScreen` 内部，所以首版会把播放列表面板的挂载层级上提到 Home 壳层共享位置，让首页概览态和播放展开态都复用同一个 `showPlaylistSheet` 状态与同一套面板组件。

这样做的原因是：
- 首页迷你条上的列表按钮不能依赖“先进入播放页再打开列表”，否则这个按钮几乎失去独立存在的意义。
- 当前 `PlayerUiState.showPlaylistSheet` 已经在 ViewModel 里存在，状态源不需要重做，只需要调整挂载位置。
- 共享挂载位置可以避免首页和播放页各自复制一份 `PlaylistBottomSheet`，减少双份遮罩和状态竞争。

备选方案：
- 首页列表按钮点击后直接 `openPlayer()` 并顺便打开列表。交互路径更绕，也与参考图里的“迷你条局部控件”不一致。
- 在首页概览态再复制一份 `PlaylistBottomSheet`。实现看似快，但后续会出现两个宿主位置、两个 testTag 和两套路径。

因此本次选择把播放列表面板提升到 Home 壳层共享承载。

### 7. 本 change 采取 UI-first 落地，不阻塞于元数据补齐任务

这次实现的优先级是“先把首页迷你条 UI 做到位，并且把交互闭环跑通”，因此设计会刻意避免依赖另一个任务：
- 没有真实封面时，用占位封面；
- 没有真实歌手时，用现有文件名解析结果；
- 不等待 richer metadata、在线封面或 `MusicInfo` 字段扩展完成后再开始做 UI。

这样做的原因是：
- 用户已经明确同意“可以先实现 UI”。
- 迷你条样式、排版、手势分区和列表共享承载，已经足够构成一个独立且可验证的 UI change。
- 先把 UI 宿主结构搭好，后续另一个任务只需要把真实 metadata 填进来，而不用再改布局层级和交互边界。

备选方案：
- 等另一个任务完成后再做首页迷你条。能一次性吃到真实元数据，但会把当前 UI 进度整体后移。

因此本次选择 UI-first。

## Risks / Trade-offs

- **[没有真实封面时视觉会比参考图弱]** → 保留固定尺寸的封面承载位，并使用统一占位封面，后续元数据任务只需要替换数据源，不用重做布局。
- **[文件名解析出的歌手信息不一定准确]** → 首版继续复用 `resolvePlayerTrackText(...)`；这被明确视为临时策略，不在本 change 里承诺完整 metadata。
- **[将列表面板上提后，首页与播放页可能出现遮罩层级问题]** → 统一把面板挂到 Home 壳层单一位置，并复用已有 `showPlaylistSheet` 状态，避免双份面板同时存在。
- **[整条点击与右侧局部按钮可能发生手势冲突]** → 通过显式分区组件和独立 testTag 验证，保证右侧按钮优先消费点击。
- **[横滑切歌与主体点击手势冲突，导致轻微拖动也进入播放页]** → 在歌曲展示区使用明确的滑动阈值和回弹判定；只有未达到切歌阈值且未发生显著拖动时，点击才进入播放页。
- **[右侧控件区过挤，导致小屏上中间文案被压缩]** → 固定右侧控件组宽度，并限制标题单行省略、歌手单行省略，优先保证整体胶囊条稳定。
- **[跑马灯和横滑都落在歌曲展示区，可能产生手势与动画观感叠加]** → 跑马灯只作用于标题文本本身，横滑位移作用于歌曲展示容器；横滑中不改变右侧控件区位置，结束后统一回弹。
- **[只做首页迷你条而不改播放页，视觉仍可能存在轻微落差]** → 本次接受这种阶段性不一致，只要首页与播放页的主色、圆角和信息层级明显 closer enough；播放页深调由其他 change 继续收口。

## Migration Plan

1. 在 Home 概览态引入新的迷你条组件，替换现有 `HomePlayEntryCard` 的卡片式入口结构。
2. 抽出首页迷你条所需的轻量 view data 解析逻辑，基于现有 `PlayerUiState` 计算标题、歌手、播放状态和进度比例。
3. 在迷你条右侧实现“环形进度包裹播放/暂停按钮 + 邻接列表按钮”的组合控件区，并补齐独立点击回调。
4. 在 minibar 中间歌曲展示区加入横向位移反馈和阈值式左右滑切歌手势，并复用现有上一首 / 下一首回调。
5. 为长歌名标题接入单行跑马灯承载，保持迷你条高度与右侧组合控件区稳定。
6. 将 `PlaylistBottomSheet` 的挂载位置从 `PlayerScreen` 内部提升到 Home 壳层共享位置，让首页迷你条和播放展开页都能复用。
7. 补充首页迷你条的交互和 UI 验证，覆盖整条点击进入播放页、歌曲展示区左右横滑切歌、播放/暂停按钮点击、列表按钮点击、长歌名跑马灯和空封面占位场景。
8. 后续另一个 metadata 任务完成后，只需要替换封面和歌手等真实数据来源，不需要重新设计迷你条布局。

## Open Questions

- 首页迷你条首版的占位封面是使用纯色唱片/专辑占位图，还是复用当前播放器主视觉的简化版本；这只影响视觉细节，不影响本次结构设计。
- 右侧列表按钮首版是否直接复用当前播放页的列表 icon，还是做一个更贴近参考图的线性 icon；这属于视觉细节，可在实现时按现有图标系统落地。
