## Context

当前播放器展开页已经具备稳定的播放状态来源与 UI 承载：`PlayerRuntime` 持有主播放状态，`PlayerViewModel` 负责把远端状态同步到 `PlayerUiState`，`PlayerScreen` 再基于 `currentSongId`、`currentArtistId`、`displayedSeekMs`、`songWikiUiState` 等字段渲染页面。现状里播放页信息区仍保留固定文案“歌词待补充”，首页底部 minibar 也已经有独立的封面、歌曲信息和右侧控件区，但整体密度偏大；与此同时，系统 `MediaSession` / 通知仍只展示静态歌名与状态文案，没有跟随当前歌词时间投影显示标题。

这次变更同时涉及四个边界：

- 播放页 UI：需要把静态占位升级为顶栏“歌曲 / 歌词”切换、当前行歌词摘要和独立歌词主界面承载。
- 数据链路：需要为在线歌曲接入 `/lyric?id=<songId>`，并处理无需登录、无歌词、请求失败、歌曲切换等状态。
- 本地保留策略：歌词下载不能无限增长，需要限制最多仅保留 100 首，并在后续重复进入同一歌曲时优先复用本地结果。
- 跨进程展示投影：首页 minibar 与系统 `MediaSession` 需要复用同一份“当前歌词行 + 歌名/歌手”的展示投影，保证应用内外当前播放展示一致。

仓库内已有最接近的参考链路是 `SongWikiRepository`：它复用 `AppContainer + JsonHttpClient` 注入方式，由 `PlayerViewModel` 发起加载，再通过 `PlayerUiState` 回推给 `PlayerScreen`。歌词能力仍沿用这个 app 侧数据流，不把歌词解析、缓存和当前行计算塞进 `playback-service` 或 native/player 模块；但为了让系统媒体入口同步显示当前歌词，需要新增一条轻量的 display metadata 投影通道，把 app 侧已经算好的显示文案同步给 `MediaSession`。

## Goals / Non-Goals

**Goals:**

- 为具备在线 `songId` 的当前播放歌曲提供歌词获取、解析、展示和本地保留能力。
- 在播放展开页补齐歌词相关交互，包括顶栏“歌曲 / 歌词”切换、歌手下方当前行歌词以及独立歌词主界面的状态切换与自动滚动。
- 复用现有播放进度状态来驱动歌词高亮，避免引入新的计时源或播放器侧专用歌词时钟。
- 为歌词本地资源建立明确的保留上限和淘汰策略，最多保留 100 首。
- 在不破坏首页迷你播放条现有手势与操作语义的前提下收紧其整体高度和内部控件尺寸。
- 让首页 minibar 的主标题与系统 `MediaSession` 标题都按当前歌词时间展示当前一句歌词，并把次级信息统一收敛为“歌名 - 歌手”。
- 收敛完整歌词页当前行高亮的视觉强调，保留可识别性但避免跳出整页节奏。

**Non-Goals:**

- 不把歌词解析、缓存和当前行计算迁移进 `playback-service`、FFmpeg/native 播放实现或 Media3 队列协议；服务侧只消费 app 已经算好的 display metadata 投影。
- 不新增独立的歌词管理页、下载列表页或全局设置页；首版只覆盖当前播放上下文中的歌词能力。
- 不在首版引入逐字歌词、双语歌词对照、桌面歌词或锁屏歌词。
- 不扩大为通用文本缓存框架；歌词保留策略只服务当前歌词能力。

## Decisions

### 1. 歌词数据流沿用 `PlayerViewModel -> Repository -> PlayerUiState -> PlayerScreen`

歌词能力挂在现有播放器 app 侧状态链路上，而不是进入后台播放服务。具体做法是：

- 在 `feature/player` 下新增歌词仓库与远端数据源，复用 `AppContainer` 中的 `JsonHttpClient` 注入。
- 在 `PlayerUiState` 中补充歌词展示状态，例如当前歌词 tab、歌词加载态、已解析歌词内容、是否可下载/是否来自本地缓存等。
- 由 `PlayerViewModel` 监听当前歌曲切换，在 `currentSongId` 变化时触发歌词读取流程，并在歌曲切换时重置旧歌词态。
- `PlayerScreen` 仅消费歌词 UI 状态并触发 tab 切换、重试、下载等事件，保持现有“状态下发 + 回调上抛”的 Compose 边界。

这样做的原因是歌词展示天然依赖 UI 结构和当前页面状态，而不是播放服务自身的最小播放职责；同时仓库里已有 `SongWikiRepository` 作为近似模式，可以减少额外架构分叉。

备选方案：

- 把歌词同步逻辑放进 `playback-service`，让服务直接输出当前行歌词。放弃原因是歌词不会影响音频解码和播放控制，却会扩大跨进程状态面，并增加协议负担。
- 在 `PlayerScreen` 内直接请求接口。放弃原因是会把网络与缓存职责压进 UI 层，且不利于测试和歌曲切换时的状态收敛。

### 2. 当前行歌词使用现有 `displayedSeekMs` 推导，不新增独立计时器

歌词高亮和“歌手下面显示对应时间的歌词”都以 `PlayerUiState.displayedSeekMs` 为唯一时间源。歌词解析后保留为按时间升序排列的行列表，ViewModel 或 UI 层基于当前 `displayedSeekMs` 计算当前命中行，并把该结果映射到：

- 播放信息区歌手下方的一行当前歌词摘要。
- 歌词 tab 内的当前高亮行与必要的邻近行展示。

这样做可以直接复用现有 `remoteSyncJob + uiProgressJob` 已经持续刷新的进度投影，不需要再为歌词单独维护 coroutine ticker，也能保证 seek 拖动时歌词跟随 `displayedSeekMs` 变化同步刷新。

备选方案：

- 新建歌词专用定时器。放弃原因是会和现有进度刷新形成两套节拍，增加错位与抖动风险。
- 仅在 Compose 层通过 `seekValueMs` 临时计算。放弃原因是状态分散，重试、缓存命中和切歌重置更难统一管理。

### 3. 远端歌词请求新增独立 `LyricRepository`，并明确 `requiresAuth = false`

歌词接口为 `/lyric?id=<songId>`，按用户说明不需要登录，因此远端数据源会显式传入 `requiresAuth = false`，避免复用认证接口时把无关 Cookie/Header 带进去。返回结果由仓库层完成这些职责：

- 解析标准时间标签歌词，过滤空行、非法时间戳和纯元数据头。
- 对无歌词、纯音乐、接口失败等场景统一映射成可展示的空态或错误态，而不是把原始 JSON 直接暴露给 UI。
- 为同一 `songId` 提供“本地命中优先、远端兜底”的读取策略。
- 歌词获取时机采用“两段式触发”：`currentSongId` 变化后立即查本地；仅当本地未命中时，才发起远端请求。默认对远端请求增加一个很短的稳定窗口，用于过滤快速切歌；但如果用户已经打开播放页且歌词区域可见，则跳过等待并立即请求当前歌曲歌词。

这样可以把“歌词接口无需登录”的边界固定在仓库层，避免后续 UI 或业务层误沿用 `requiresAuth = true` 的在线详情接口默认值。
而“两段式触发”则兼顾了两个目标：一方面避免在加入队列或快速连续切歌时为每一首都发起歌词请求，另一方面又保证用户进入播放页时能尽快看到歌词，而不是等到手动点开歌词 tab 后才开始加载。

备选方案：

- 直接复用 `SongWikiRepository` 或把歌词塞进歌曲百科仓库。放弃原因是语义不同，缓存与失败策略也不同，会让仓库职责变混杂。
- 统一走带认证请求。放弃原因是和接口契约不符，也可能引入不必要的登录态耦合。
- 只在用户点击歌词 tab 后才开始拉取歌词。放弃原因是无法满足歌手下方当前行歌词摘要的实时展示，也会让首次进入播放页的感知过慢。
- 在歌曲进入播放队列时批量预拉歌词。放弃原因是过早，会浪费请求和本地 100 首歌词保留额度。

### 4. “下载”与“查看”共用同一份歌词文件存储，采用 100 首 LRU 式保留策略

仓库中没有适合承载大段歌词文本的现成缓存层，因此新增一个歌词本地存储组件，使用 app 私有目录下的独立文件夹按 `songId` 存储解析后的歌词文本或结构化 JSON。设计上把“歌词下载”和“播放页读取后落盘”统一到同一个存储中：

- 首次从远端成功获取歌词后写入本地存储。
- 之后再次播放相同歌曲时优先命中本地存储，减少重复请求。
- 每次写入成功后更新该首歌词的最近访问时间。
- 当本地保留数量超过 100 首时，按最近访问时间从旧到新淘汰，直到回到上限。

这样做避免额外区分“缓存”和“下载库”两套文件源，也能满足“最多保留 100 首，避免资源太多”的目标。若后续需要显式“仅手动下载才长期保留”的产品策略，再在同一存储模型上加 pinned 字段扩展。

备选方案：

- 把歌词全文写进 `SharedPreferences`。放弃原因是文本体积不稳定，不适合长期保留和淘汰管理。
- 完全不做本地持久化，每次进入播放页都重拉。放弃原因是会增加请求次数，也无法满足“歌词下载功能完善”的要求。

### 5. 播放页采用“歌曲页 + 完整歌词页”的双页面结构

播放页会同时保留两种歌词呈现，但它们分别属于两个并列页面，而不是把完整歌词塞进歌曲页的局部区域：

- 歌曲页：保留现有完整播放页结构，包含封面、歌名、歌手、当前一句歌词摘要、工具区、进度区和播控区。歌手名下方展示当前时间对应的一行歌词，替代“歌词待补充”，让用户停留在歌曲页时也能感知当前唱到哪里。
- 歌词页：在顶栏“歌曲”旁边提供“歌词”tab，切换后进入与歌曲页同级的完整歌词页面；从顶栏下方开始就是歌词主内容，不再显示歌名、歌手、进度条、播控区，也不再使用圆角卡片、边框或方框容器包裹歌词。
- 主内容横滑切页：歌曲页封面区域与歌词页主内容区域支持左右横滑，在歌曲页左滑进入歌词页，在歌词页右滑返回歌曲页；横滑手势只挂在主内容区域，不挂在顶栏，也不与歌曲页底部的进度拖动共享同一手势层。

这样做是因为用户明确要求歌词页必须是“和播放页同样式、通过顶栏切换的完整界面”，而不是底部仍保留进度条和播控的半页结构；同时又要求歌手下面只显示当前一句歌词，因此摘要和完整歌词需要分别由歌曲页与歌词页承载。视觉上，歌词页需要去卡片化并移除重复的歌曲信息，才能保持沉浸感而不显得累赘。
版式上，歌词页直接复用歌曲页已经验证过的几何节奏：歌词页主内容顶部起始位置与歌曲页封面顶部对齐，左右内边距与歌曲页主内容一致，底部留白则与歌曲页播控距底部的视觉基线保持一致，避免歌词页看起来悬空或贴底。
为避免不同尺寸设备上出现“封面过大、标题顶到封面、播控过重”的问题，歌曲页与歌词页的关键尺寸不再只靠固定 `dp` 档位，而是统一改成“按视口比例计算，再加最小/最大边界”的响应式 token。封面、标题字号、顶部切换、工具行、进度区、播控尺寸和底部留白都要复用同一套比例化 metrics，这样切到不同屏幕时整体节奏能一起收缩，而不是局部放大或局部塌陷。
在用户确认的参考稿基础上，这轮前台层视觉继续收敛为“重排布局、不重做按钮皮肤”：

- 顶部 `歌曲 / 歌词` tab 采用更细的导航式指示线，而不是按钮感很强的胶囊或大块背景。
- 歌曲页保留现有按钮组件样式，但重排为更接近参考稿的层级：封面使用外壳包裹内封面的层次，信息区采用左侧标题/歌手加右侧收藏按钮，当前歌词摘要居中作为一句引用式文案承接。
- 歌曲百科与更多操作保留现有按钮样式，但下沉为辅助操作行，不再与标题区争抢视觉主导。
- 歌词页不再依赖明显放大来突出当前行，而是使用克制的字号差、白色主字和轻微紫色辉光，其他行按距离进一步弱化，整体更接近沉浸式歌词阅读。

备选方案：

- 只在歌手下方放一行歌词，不做 tab。放弃原因是无法满足完整歌词浏览需求。
- 把完整歌词塞进歌曲页底部信息区与进度区之间。放弃原因是会破坏页面层次，也与“和播放页同级、通过顶栏切换”的交互预期不符。
- 只做完整歌词页，歌曲页不显示当前行。放弃原因是与用户要求不符，也会让主播放区缺少即时反馈。

### 6. 首页 minibar 复用当前歌词摘要，但不改变现有交互语义

`MainShellScreen` 中的 minibar 已经承担主体点击进入播放页、独立播放/暂停按钮、播放列表入口和中间歌曲区横滑切歌。此次除了继续收紧高度、封面尺寸、文字样式和右侧控件区占比之外，还要让中间双行信息区复用歌词展示投影：

- 上行不再固定显示歌名，而是显示当前播放时间对应的一句歌词；当歌词不可用时回退到原歌名或稳定占位。
- 下行统一显示“歌名 - 歌手”，并使用跑马灯承载长文本，避免在紧凑布局下把歌手信息截得支离破碎。
- 右侧控件、主体点击和中间区域横滑切歌语义保持不变。

- 主体点击仍进入播放展开态。
- 播放按钮与列表按钮仍独立响应。
- 中间歌曲区的左右横滑切歌保持不变。

这样可以把首页变更收敛为“展示投影 + 密度优化”，避免把歌词需求连带扩大成首页交互重构。

备选方案：

- 顺带重做 minibar 交互结构。放弃原因是会把 scope 从歌词能力扩散到 Home 壳层重构，超出本次提案范围。

### 7. 系统 `MediaSession` 复用 app 侧 display metadata 投影，而不是在服务内重新计算歌词

歌词解析和当前行命中仍然只在 app 侧发生，但系统媒体入口需要看到与首页 minibar 一致的标题语义。因此新增一条轻量的 display metadata 投影链路：

- app 侧基于 `PlayerUiState.currentTrackTitle`、`currentTrackArtist`、`lyricUiState` 和 `displayedSeekMs` 生成统一的展示文案：`displayTitle`（当前歌词行或歌曲标题回退）与 `displaySubtitle`（`歌名 - 歌手`）。
- `PlayerViewModel` 监听该投影变化，只在文案实际变化时通过 `PlayerServiceBridge` 把新的 display metadata 发给 `MediaSessionService`。
- `playback-service` 仅保存当前投影值，并把它写回当前 `MediaSession` item 的 `MediaMetadata` / 通知标题位；不在服务内发起歌词请求，也不在服务内独立维护歌词时间计算。
- 为了避免 app 自己从远端快照反向读回“歌词标题”后污染播放器原始歌名，跨进程共享 metadata 需要保留原始歌曲标题/歌手字段，供 app 继续重建业务态时优先使用。

这样做的原因是用户已经明确要求首页 minibar 与系统媒体卡片展示保持一致，而服务端并不拥有歌词解析上下文。让 app 负责生成 display metadata，再把结果轻量投影到服务，是最小也最稳定的边界改动。

备选方案：

- 直接在 `playback-service` 内根据播放进度重新计算歌词当前行。放弃原因是会把歌词仓库、歌词解析与计时责任拉进服务，和本轮“歌词逻辑留在 app 侧”的边界相冲突。
- 只更新通知标题，不更新 `MediaSession` 当前 item metadata。放弃原因是系统锁屏卡片、蓝牙设备和其他 session 消费方看到的文案会与通知不一致。

## Risks / Trade-offs

- [歌词接口格式可能包含多种元数据或缺失时间标签] → 在仓库层把解析与 UI 映射分开，先支持标准时间标签歌词；非标准结构统一回退到“暂无歌词”或纯文本兜底。
- [歌曲切换或快速切歌时旧歌词结果串到新歌曲] → 以 `currentSongId` 作为请求归属键，只在返回时仍匹配当前歌曲时更新 `PlayerUiState`。
- [切歌后过早请求远端歌词会造成无效请求堆积] → 采用“本地优先 + 短稳定窗口”的两段式触发；仅在播放页已显式可见时跳过等待立即拉取。
- [歌词高亮依赖 UI 进度刷新，极端情况下会有轻微跳字] → 复用现有 `displayedSeekMs` 保持一致性，避免双计时源；歌词主界面只在命中行变化时触发自动滚动，减少无意义抖动。
- [歌词页横滑与进度条拖动手势互相抢占] → 仅在歌曲页封面区与歌词页主内容区启用横向切页手势；顶栏不参与横滑，歌曲页底部进度区继续保留原有 seek 拖动职责，歌词页自身不再承载进度条。
- [文件存储引入 IO 成本] → 只在后台协程读写，文件数量限制为 100 首，并在解析成功后写入，避免对主线程造成影响。
- [minibar 收紧后长标题和按钮可读性下降] → 保留现有跑马灯与独立控件区，优先缩间距和容器高度，不牺牲主要点击目标；下行统一收敛到“歌名 - 歌手”，避免两段信息各自截断。
- [系统 `MediaSession` 文案跟随歌词频繁刷新，可能反向污染 app 内当前歌曲标题] → 在共享 metadata 中额外保留原始标题/歌手字段，app 重建业务态时优先使用原始字段，display metadata 仅服务系统展示。
- [当前行高亮过强导致歌词页跳眼] → 把强调方式从大幅字号跳变收敛为较小的字号差异、适度字重与透明度层次，高亮仍清晰可识别，但不破坏整页节奏。

## Migration Plan

1. 先补 `playback-lyrics` 新 capability spec，以及 `player-expanded-page`、`user-center-tab-shell`、`media-session-integration` 的 delta spec，固定 requirement 边界。
2. 实现歌词仓库、远端数据源、解析器与本地存储，并在 `AppContainer` 中完成注入。
3. 扩展 `PlayerUiState` / `PlayerViewModel` / `PlayerRuntime` 的播放器 UI 状态，使歌曲切换、加载、重试、下载落盘和高亮推导可闭环。
4. 更新 `PlayerScreen` 与 `MainShellScreen`，分别落歌词 tab/摘要显示、完整沉浸歌词页、当前行高亮收敛和 minibar 的歌词化双行展示。
5. 为 app -> service display metadata 投影补上 `playback-contract`、`playback-client`、`playback-service` 的共享字段与命令接线，让系统 `MediaSession` 标题位按歌词时间刷新。
6. 用 Robolectric / 单元测试补充播放页、minibar 与 `MediaSession` metadata 回归用例，再跑 `:app:testDebugUnitTest`、`:playback-service:testDebugUnitTest`、`:app:assembleDebug` 做收口验证。

本次变更只新增 app 侧歌词能力，没有线上数据迁移要求；若实现后发现歌词文件策略不合适，可直接停用新仓库读取并保留远端兜底，不影响既有播放链路。

## Open Questions

- `/lyric` 返回体里是否需要同时兼容翻译歌词、逐字歌词等扩展字段；若接口已有这些字段，首版是否一律忽略，只保留主歌词文本。
- “歌词下载功能完善”在首版是否需要显式下载按钮；当前设计默认把成功拉取后的本地持久化视为下载完成，若用户期望独立下载入口，需要在 specs 阶段进一步钉清。
