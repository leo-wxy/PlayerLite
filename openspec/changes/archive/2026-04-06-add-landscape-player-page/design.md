## Context

当前独立播放器由 `app` 模块中的 `PlayerActivity` 承载状态接线，由 `feature-player` 模块中的 `PlayerScreen` 承载 UI。现状只有一套竖屏布局，也没有页面内方向模式模型；如果直接在 `Activity` 里硬编码横屏切换，会让 UI 状态、方向状态和横屏视觉层混在一起，后续很难继续迭代参考 UI。

本次实现同时跨越 `app` 模块和 `feature-player` 模块，并涉及方向控制、Compose 布局分支和低风险视觉特效，因此需要先固定技术方案。

## Goals / Non-Goals

**Goals:**
- 为播放器建立可复用的方向模式模型，支持 `自动`、`锁定横屏`、`锁定竖屏`
- 让 `PlayerActivity` 成为方向模式的唯一宿主，保证其他页面不受影响
- 为 `PlayerScreen` 增加横屏布局分支，并保持横竖屏切换时的播放上下文连续
- 在横屏页中为主视觉卡片和底部控件实现局部水波倒影的第一版效果

**Non-Goals:**
- 不在本次引入新的图片/图形依赖
- 不把整个播放器页重写成完全新的设计系统
- 不在第一版就依赖 Android 13+ 的 `RuntimeShader` 作为主路径
- 不做整页真实物理水面模拟或全屏复杂粒子效果

## Decisions

### 决策 1：方向模式放在播放器 UI 状态中，由 PlayerActivity 执行

- 选择：新增 `PlayerOrientationMode` 状态，进入 `PlayerUiState`，由 `PlayerViewModel` / `PlayerRuntime` 负责更新，由 `PlayerActivity` 观察并映射到 `requestedOrientation`
- 原因：方向模式属于播放器页的用户意图，不应只存在于 `Activity` 局部变量；同时真正改变页面方向的宿主仍然必须是 `Activity`
- 备选方案：
  - 只在 `Activity` 里保存方向模式：实现快，但 UI 无法稳定表达当前模式，测试也更差
  - 完全依赖系统方向：无法满足页面内手动切换横竖屏的需求

### 决策 2：PlayerScreen 保持单一入口，内部根据当前方向切换 portrait / landscape 分支

- 选择：继续以 `PlayerScreen` 作为统一入口，在 `feature-player` 模块内部拆出横屏布局分支与横屏专用前景组件
- 原因：能复用现有播放器状态、回调和浮层逻辑，不需要重新搭一套独立横屏页面
- 备选方案：
  - 新建独立 LandscapePlayerScreen：边界清晰，但第一版会重复大量已有参数与浮层 wiring
  - 在现有竖屏布局上硬压成双栏：实现快，但效果会像“横着摆的竖屏页”，无法承接后续视觉稿

### 决策 3：倒影效果采用镜像层 + 条带扰动 + 渐隐遮罩，不把 RuntimeShader 作为基础实现

- 选择：只对参与倒影的前景元素生成局部镜像层，使用 `graphicsLayer`、`Canvas` 或 `drawWithCache` 做轻微条带位移扰动和渐隐遮罩
- 原因：仓库当前最低 `minSdk = 24`，也没有现成 shader 基础；用 Compose 自身能力更稳，第一版就能覆盖全部设备
- 备选方案：
  - 直接用 `RuntimeShader`/AGSL：高版本效果更强，但低版本需要完整 fallback，第一版投入过高
  - 只做静态镜像不做扰动：实现最省事，但水波氛围不够明显，离参考效果差一截

### 决策 4：倒影只反射指定前景元素，不反射整页

- 选择：仅让主视觉卡片和底部控件参与倒影，文本倒影若需要也保持弱化
- 原因：整页镜像容易脏，且会干扰可读性；局部倒影更像海报式播放器
- 备选方案：
  - 整块前景统一倒影：实现简单，但页面容易显得廉价和噪声过多

## Risks / Trade-offs

- [横竖屏切换重组过多] → 通过把方向模式放进 `PlayerUiState`，减少 `Activity` 与 Compose 的隐式状态分叉
- [横屏布局与现有浮层冲突] → 保持现有浮层入口和状态模型不变，只替换主页面内容布局分支
- [倒影特效性能抖动] → 第一版只对少量前景元素做局部倒影，并限制倒影高度和扰动振幅
- [视觉与最终参考稿存在偏差] → 先交付稳定骨架和可调参数，后续根据参考 UI 再调布局比例、文案层和动效强度
