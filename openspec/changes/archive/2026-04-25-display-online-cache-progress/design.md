## Context

当前在线播放已经具备磁盘缓存与缓存读取链路，但这些状态主要留在服务和缓存层内部。前台 UI 只知道“当前播到哪里”，不知道“当前缓存到哪里”。对于播放器型产品，这会让用户在弱网、暂停恢复、重新进入播放页时难以判断是否已经有足够缓存，也无法验证后续预热是否真正生效。

本次设计只覆盖“当前播放项”的缓存进度展示，不扩展到“下一首预热”“整条队列缓存”或“下载中心”。

## Goals / Non-Goals

**Goals:**
- 让 playback service 作为当前播放项缓存状态的唯一权威来源。
- 让 app 通过远端快照直接拿到当前项缓存进度，而不是本地探测缓存文件。
- 让 minibar 与播放页用同一条进度条表达三段状态：已播放、已缓存未播放、未缓存。
- 让完整缓存时缓存条直接拉满；总量未知时先给出稳定估算比例，后续再纠正。

**Non-Goals:**
- 不实现下一首预热或预热状态展示。
- 不实现整首下载、离线下载中心或批量缓存管理。
- 不新增独立的缓存状态卡片、额外颜色体系或复杂字节统计面板。
- 不为整条队列、最近播放或通知栏同步扩展缓存进度展示。

## Decisions

### 决策一：缓存进度由 playback service 权威上报

当前播放项的缓存状态需要和当前实际播放 source 绑定，最稳定的采集点在 playback service 一侧，而不是 UI 或 app runtime 自己扫磁盘。服务侧可基于当前 active resource key 和 `CacheCore.lookup(...)` 的结果得出缓存进度，再通过 media session extras / remote snapshot 向 app 投影。

备选方案是让 app 直接依赖 cache-core 自行查缓存，但这样会让前台拥有第二套缓存状态来源，容易在跨进程、切歌和 source 切换时出现当前项错配。

### 决策二：先只展示当前播放项，且只复用现有进度条

这次要解决的是“当前正在听的歌已经缓存了多少”，不是做缓存中心。UI 上不新增独立控件，而是复用现有进度条：已播放部分继续使用当前进度色，已缓存未播放部分使用同色低透明度覆盖，未缓存部分保留轨道底色。

备选方案是为缓存增加单独进度条或状态卡片，但这样会明显放大 UI 改动范围，也不符合“先简单做”的要求。

### 决策三：完整缓存时缓存条直接拉满

如果当前歌曲已经判定为完整缓存，缓存条必须直接画满整条轨道，不再跟随播放位置停留在已播附近。用户需要一眼看出“这首歌已经全缓存完了”。

备选方案是仍只在播放位置后面显示一小段缓存区域，但这会弱化“完整缓存”这个最关键状态。

### 决策四：总量未知时允许估算比例，后续再纠正

部分在线资源在早期阶段可能还拿不到稳定 `contentLength`。为了避免前台长期停在“0 缓存”的误导状态，本次允许服务侧在总量未知时根据当前可得信息给出估算比例；当真实总量可用后，再把展示比例纠正为真实值。

备选方案是总量未知时完全不画缓存条，但这样会让缓存实际已经推进时前台仍无反馈。

## Data Flow

1. playback service 在当前播放项有效时，基于当前 resource key 查询缓存快照。
2. 服务侧生成当前项缓存状态：
   - `cacheMediaId`
   - `cachedBytes`
   - `totalBytes`
   - `displayRatio`
   - `isFullyCached`
   - `isEstimated`
3. 服务侧把这些字段写入远端 metadata extras。
4. `RemotePlaybackSnapshotMapper` 读取 extras，组装进 `RemotePlaybackSnapshot`。
5. `PlayerRuntime` 只做权威投影，生成 `PlayerUiState` 的缓存展示字段。
6. minibar 与播放页基于同一组比例字段绘制同一套进度条语义。

## UI Rules

- 只对当前播放项展示缓存进度；没有当前项时不显示。
- 只对具备缓存语义的在线播放项展示；本地文件或不走缓存链路的项不伪造缓存条。
- 已播放段：沿用当前进度颜色与位置。
- 已缓存未播放段：使用同色、较低透明度的覆盖段。
- 已完整缓存：缓存段直接拉满整条轨道。
- 总量未知但已有缓存推进：允许先画估算比例。
- 实际缓存比例必须被约束在 `0f..1f`，并且不得短于已播放比例。

## Risks / Trade-offs

- [风险] 缓存快照更新过频会造成远端快照抖动。 → 服务侧按“比例明显变化或状态变化”节流发布。
- [风险] 总量未知时的估算值可能和最终值存在偏差。 → UI 只把它作为临时展示比例，不输出误导性的精确字节承诺。
- [风险] 当前项切歌后，旧缓存快照可能短暂残留。 → 以当前 mediaId / resource key 绑定，切歌时立即重置旧项展示。
- [风险] UI 复用同一条进度条后，颜色层次不够明显。 → 保持同色系，仅通过透明度和层级表达，避免扩大设计面。
