## Context

当前 `MainActivity` 直接承载完整 `PlayerScreen`，页面顶部和主体区域同时混合了播放、文件选择、状态信息与账户入口。随着登录、用户头像、后续用户资料和账户能力逐步增加，首页已经不适合作为唯一主页面；同时，当前播放器主界面也缺少更清晰的“首页 -> 播放页”层级结构，以及首页与播放页之间的联动视觉空间。

这次变更的目标不是重写播放逻辑或用户会话逻辑，而是在 `app` 层建立新的前台页面外壳：底部双 Tab 承载首页和用户中心，播放页不再作为整个 `MainActivity` 的唯一主内容，而是调整为首页域中的可展开播放层/二级页面；用户相关能力收口到右侧用户中心页。

约束与现状：

- 播放链路、`PlayerViewModel`、`PlayerRuntime`、`playback-service` 与登录态恢复逻辑已经存在，优先复用。
- 当前 `LoginActivity`、用户会话状态和头像投影已经可用，用户中心页应消费现有状态，而不是重新发明用户状态来源。
- 用户要求首页左侧 Tab 对应页面底部提供一个较小的播放入口按钮，并希望播放页仍然作为首页的一部分存在，以便后续做歌曲封面慢慢放大这类联动效果。
- 用户进一步强调首版要先把“效果”做出来，次级能力后续再逐步补；同时希望登录页顶部主视觉可以产出为当前应用图标。

## Goals / Non-Goals

**Goals:**

- 建立 `MainActivity` 的底部双 Tab 主外壳：左侧首页、右侧用户中心。
- 将当前播放页调整为首页域内的二级页面/展开层，并让首页通过底部小按钮触发展开。
- 首版优先保证可见的页面效果、层级关系与联动锚点成立，而不是一次性塞满用户中心的全部功能。
- 让用户资料展示、登录入口和后续账户能力优先落在用户中心页，而不是继续放在首页播放器头部。
- 将登录页顶部主视觉沉淀为一份可复用品牌资产，并用于当前 app icon 产出。
- 保持现有播放能力、登录恢复、未登录启动门控和本地播放逻辑可继续工作。
- 为后续账户二级页和更多首页内容扩展提供稳定的页面骨架。

**Non-Goals:**

- 不重构 `playback-service`、播放协议或用户会话底层实现。
- 不在本次设计中引入新的多 Activity 架构；主应用仍以 `MainActivity + LoginActivity` 为主。
- 不一次性做完整的用户中心二级页面体系，只先定义首版个人中心页承载位。
- 不改变登录接口、会话存储格式或在线播放登录卡口规则。
- 不在首版要求把用户中心的所有二级能力一次做完，优先保证主结构和主要视觉效果成立。
- 不在首版补齐收藏歌单等内容型能力，这些能力留待后续继续扩展。

## Decisions

### 1. 使用轻量级 Compose 页面外壳，并把播放页建模为 Home Tab 内部子路由

`MainActivity` 将升级为一个轻量级页面容器，内部维护两个层级状态：

- `selectedTab`：`Home` / `UserCenter`
- `homeSurfaceMode`：`HomeOverview` / `PlayerExpanded`

这次优先使用本地可保存的路由状态（例如 `rememberSaveable` 级别的页面模式），而不是立即引入 `Navigation Compose`。原因是当前应用前台主结构仍然很简单：一个主壳、两个底部 Tab、一个 Home 内部展开的播放层、一个已存在的登录 Activity。直接引入完整导航框架会增加额外心智负担和状态同步复杂度，但不会在首版提供同等价值。

**备选方案：**

- 引入 `Navigation Compose`：扩展性更强，但当前页面层级还不足以证明引入成本合理。
- 播放页改成独立 Activity：会让 `MainActivity`、登录门控和共享 `PlayerViewModel` 的职责变复杂，也不利于首页与播放页之间做联动动画。

### 2. 将当前 `PlayerScreen` 拆成“首页概览态”与“首页域内展开播放态”

现有 `PlayerScreen` 主要承担完整播放控制界面职责。新结构下，它不再直接作为首页唯一内容，而是被拆成两个同域 Surface：

- `HomeOverview`：首页概览态，承载首页内容与底部播放入口按钮
- `PlayerExpanded`：首页域内的完整播放态，承载当前播放器主体

两者共享同一套 Home 域状态和视觉锚点，便于后续做歌曲封面、背景或卡片从首页概览态逐步放大到完整播放态的过渡效果。

首版实现里，Home 域外层切换动画应成为概览态与播放展开态之间的主要入场运动来源；播放器内部不应再叠加一层同类型的整页 reveal，以免产生“先切页、再二次展开”的僵硬感。

具体上会做两层拆分：

- 保留当前播放器 UI 主体作为 `PlayerExpandedSurface`（名称可再调整），由 Home Tab 内部展开态承载
- 新增 `HomeOverviewSurface`，承载首页概览内容，并在页面底部安全区附近放置一个较小的播放入口按钮

这样可以尽量复用已有播放 UI、列表控制和状态投影逻辑，不需要重新发明播放器页；同时首页不再被迫长期显示完整播放器内容，又保留首页与播放页同域联动的基础。

**备选方案：**

- 播放页完全独立成全局路由：层级清晰，但不利于做首页与播放页之间的共享动效。
- 首页只显示一个极简播放器摘要卡片：可行，但首版需要额外定义过多新视觉结构，不如先复用现有播放页更稳。

### 3. 用户中心页接管账户相关入口与展示

用户中心页将成为用户会话相关 UI 的主承载位。已登录时首版先展示头像、昵称和基础用户信息；未登录时展示登录引导与当前账号能力说明。收藏歌单等次级内容不要求在首版完成，首页右上角不再保留头像主入口，避免首页继续承担主要的账户信息展示职责。

这样做的原因是：

- 用户资料和登录入口已经不是“播放器附属功能”，而是明确的第二主域
- 后续新增退出登录、用户详情、收藏、设置或更多个人能力时，用户中心页更自然
- 与“左首页、右用户中心”的底部 Tab 结构语义一致

**备选方案：**

- 继续保留首页右上角头像为主入口：会增加首页视觉噪音，也无法替代稳定的个人中心承载位。
- 用户中心做成弹窗或 Bottom Sheet：短期快，但后续扩展差，也和底部 Tab 结构不一致。

### 4. 首页底部播放入口采用“小按钮触发展开”，而不是常驻 Mini Player

根据当前需求，首页左侧 Tab 对应页面最底部提供一个较小的按钮，用于触发首页概览态到播放展开态的切换。首版设计将其定义为固定位置、易触达但不抢主视觉的播放入口控件；它的职责是“触发展开”，而不是完整展示播放控制。

首版不直接做常驻 Mini Player，原因是：

- Mini Player 会额外引入播放状态摘要、进度、播放/暂停等信息布局要求
- 与底部 Tab 叠加时更容易造成底部空间紧张
- 用户当前要求的是“小按钮用于跳转”，先满足这一明确约束更稳

后续如果需要，可在同一位置自然演进为 Mini Player 入口。

### 5. 播放展开态仍属于 Home Tab，上下文不脱离首页

当用户从首页底部入口进入播放展开态时，本质上仍然停留在 Home Tab 域内；这意味着首页概览态和播放展开态共享同一套首页上下文，可以围绕封面、背景或卡片做联动动画，而不是跨域跳转。

这样处理的理由是：

- 更适合后续做歌曲封面慢慢放大、背景渐变延展等共享动效
- Home 域状态不需要跨全局路由搬运，状态管理更轻
- 用户心理模型更接近“首页里的播放器被展开”，而不是“离开首页去另一个完全无关的页面”
- 播放展开态应隐藏底部 Tab，并提供显式返回按钮，避免用户只能依赖系统/硬件返回
- 首页与播放展开态之间的过渡应尽量连续，优先通过 Home 域外层容器统一驱动，而不是把同一种页面 reveal 分散到多层 UI 中重复执行
- Home 域内容区不应直接跟随底部 Tab 退场时的 Scaffold 底部 inset 一起重排，避免点击进入播放页时出现明显抖动

**备选方案：**

- 播放页完全独立成全局二级页：结构明确，但会削弱共享动效与联动空间。
- 直接把完整播放 UI 常驻在首页：联动空间存在，但首页信息密度过高，用户中心也难以建立清晰主域。

### 6. 登录门控与用户会话规则保持不变，只调整前台承载位置

未登录启动时优先进入 `LoginActivity`、允许跳过、已登录恢复用户会话、在线播放继续受登录卡口约束，这些规则不在本次重写。新的双 Tab 外壳、用户中心页和 Home 内部播放展开态只消费现有 `userSessionUiState`，并复用已有的 `LoginActivity` 打开方式。

这样可以把这次变更聚焦在页面结构上，避免把导航重构和账号底层重构耦合在一起。

### 7. 登录页顶部主视觉与 app icon 复用同一套品牌来源

登录页顶部已经存在一块较强的主视觉区域。首版将其收敛为一份可复用的品牌资产来源，并用于产出当前 app icon，而不是额外再设计一套完全无关的图标语言。

这样处理的理由是：

- 登录页、启动入口与应用图标可以保持统一视觉记忆点
- 在“先出效果”的前提下，可复用现有视觉方向，减少额外设计分叉
- 后续如果继续升级品牌风格，也可以同步更新登录页主视觉和 app icon

**备选方案：**

- app icon 独立设计：灵活，但容易与登录页主视觉脱节
- 暂不调整 app icon：实现更快，但用户已经明确提出希望同步产出

## Risks / Trade-offs

- **[首页信息密度重新分配]** → 首页在完整播放器退出常驻后可能显得内容偏空；首版通过保留轻量摘要信息、视觉锚点和底部播放入口缓解，后续再逐步补更多首页内容。
- **[轻量路由状态可扩展性一般]** → 未来如果用户中心再拆更多二级页面，当前本地路由模型可能需要升级；首版通过明确壳层状态对象和独立页面枚举，给后续迁移到更完整导航预留空间。
- **[用户入口迁移带来的习惯变化]** → 用户可能习惯从首页右上角进入账户；首版通过保留可理解的用户中心 Tab 图标与命名降低迁移成本。
- **[底部小按钮发现性有限]** → 如果按钮太弱，用户可能不知道如何展开播放页；实现阶段需要通过固定底部位置、明确图标和稳定文案/视觉层级解决。
- **[共享动效会反推页面结构]** → 如果后续确定要做封面放大等联动，首页概览态和播放展开态需要共享更多视觉锚点；实现阶段应优先保持两态在同一 Home 容器下。
- **[视觉优先会压缩功能面]** → 首版如果过度追求效果，用户中心可能显得功能偏少；通过把首版目标明确为“主结构 + 核心账户信息 + 可继续扩展”来控制预期。
- **[品牌资产复用质量]** → 登录页主视觉未必天然适合缩成 app icon；实现阶段需要在“同源”前提下允许做适配裁剪与简化。

## Migration Plan

1. 先抽离当前 `MainActivity` 中直接依赖 `PlayerScreen` 的单页结构，建立新的主壳与底部 Tab scaffold。
2. 拆分 Home Tab 的概览态与播放展开态，以及独立的用户中心页，保证它们共享现有 `PlayerViewModel` 与用户会话状态。
3. 把首页底部播放入口按钮接上 Home 域内播放展开态切换。
4. 将账户信息主展示迁移到用户中心页，并让现有登录入口逻辑在那里继续可用。
5. 收敛登录页顶部主视觉并同步产出 app icon 资源。
6. 运行 UI / 单测回归，确认登录门控、本地播放和首页到播放展开态的切换链路不受破坏。

## Open Questions

- 首页在完整播放器退出常驻后，首版除了底部播放入口之外还需要保留多少播放摘要信息？
- 用户中心页首版是否仅承载用户信息和登录/退出入口，还是要同时放入缓存清理、测试入口等“设置型”操作？
- 首页概览态与播放展开态之间，首版是否只做结构预留，还是要同步预留封面放大这类共享动效锚点？
- 登录页主视觉用于 app icon 时，首版是直接裁剪现有图形，还是在同一视觉语义下单独导出一版更适合小尺寸的图标？
