## Context

当前仓库的在线音乐能力以网易云链路为核心：`AppContainer` 只组装了网易云相关的搜索、歌单详情、专辑详情、歌手详情和歌曲详情 repository；`feature-search` 通过 `NeteaseSearchRemoteDataSource` 调用 `/cloudsearch`；播放侧通过 `PlaylistItem` -> `MusicInfo` 的既有映射进入 `PlayerRuntime` 和播放服务。与此同时，`PlaylistController` 和 `playlist-persistence` 已经能够可靠地替换、持久化和恢复“当前播放队列”，但仓库内还没有“本地永久歌单库”这类独立数据模型或存储层。

这意味着“导入网易云 / QQ 音乐网页歌单”不能直接落成一个新的长期歌单系统，也不能指望客户端直接拿 QQ 音乐原始链接进入现有播放链路。要在现有架构里最小代价落地，这个 change 需要分成两个问题：先把网页链接解析成统一的歌单线索，再把这些线索映射成应用当前可播放的在线歌曲队列。

额外约束：
- 当前在线搜索与在线播放都依赖现有登录态，`SearchRepository` 使用的搜索接口为 `requiresAuth = true`。
- 外部网页链接存在多种分享形态、重定向和平台差异，客户端直接抓取 HTML 的维护成本和失败率都很高。
- QQ 音乐来源无法直接复用现有网易云 songId，必须经过“名称/歌手关键字 -> 当前可播放曲目”的匹配过程。

## Goals / Non-Goals

**Goals:**
- 支持用户粘贴网易云歌单和 QQ 音乐歌单网页链接。
- 系统识别链接来源并解析出统一的歌单元数据与曲目线索。
- 基于现有在线搜索能力，把解析出的曲目线索尽可能映射为应用内可播放的歌曲，并向用户展示“成功匹配 / 部分匹配 / 未匹配”的结果。
- 用户确认后，将匹配成功的歌曲生成应用内播放队列并接入现有 `playlist-management` 和 `playlist-persistence` 链路。
- 让导入链路对未来新增来源平台保持可扩展，而不是把平台判断散落在 UI 和播放代码中。

**Non-Goals:**
- 不在本次 change 内实现“独立永久本地歌单库”或新的本地歌单数据库。
- 不直接播放 QQ 音乐原始外链，也不处理跨平台账号授权。
- 不在客户端实现大规模网页抓取或 source-specific HTML 解析。
- 不承诺第一版就做到全量精确匹配；允许存在部分曲目无法匹配或匹配歧义。
- 不扩展现有播放服务协议；导入结果优先复用现有 `PlaylistItem` / `MusicInfo` 契约。

## Decisions

### Decision 1: 使用“客户端来源适配器 + 来源专用详情获取”的两阶段方案

导入流程拆为两个阶段：
1. 客户端对用户输入的链接做来源识别、URL 规范化和歌单 ID 提取。
2. 客户端按来源调用专用详情获取链路，把歌单标题、封面、描述和曲目列表归一化为统一结构。

建议新增一个客户端来源适配层，例如：
- `WebPlaylistImportSource`
  - `parseInput(raw: String): ParsedPlaylistRef`
  - `fetchPlaylist(ref: ParsedPlaylistRef): ImportedPlaylistSnapshot`

来源策略按仓库现状拆分：
- 网易云：直接从分享链接中提取 `playlistId`，随后复用当前已有的歌单详情与曲目接口链路，而不是去抓取网页 DOM。
- QQ 音乐：从分享链接或重定向后的 URL 中提取歌单 ID，随后通过来源专用 remote data source 请求其歌单详情接口并提取曲目信息。

选择这个方案的原因：
- `lx-music-desktop` 的已验证做法不是“通用抓网页正文”，而是按来源做 `songList` 适配器，先从链接中提取歌单 ID，再调用来源专用详情接口拿歌单和曲目数据。
- 当前仓库已经具备网易云歌单详情能力，因此对网易云来源再加一层网页抓取只会提高复杂度，不会提高正确性。
- 对 QQ 音乐这类当前仓库没有现成 repository 的来源，适配器边界可以先接客户端 remote data source；如果后续要下沉到服务端，也不需要推翻上层设计。

考虑过但不采用的方案：
- 对所有来源统一做网页 HTML 抓取：实现脆弱、对移动端维护不友好，而且网易云来源在当前仓库里完全可以绕过这一步。
- 服务端统一解析全部链接：不是不能做，但当前用户明确强调核心是“读取网页信息并提取歌名”，并给出了客户端参考实现；在现阶段先落客户端来源适配器更贴近目标，也更容易复用现有网易云数据链路。

### Decision 2: 先提取准确曲目信息，再仅对缺少当前可播放主键的来源做关键词匹配

客户端新增 `WebPlaylistImportRepository` 或同层 use case，内部组合：
- 来源适配器：按来源读取歌单详情和原始曲目信息。
- 现有 `SearchRepository`：只在某条曲目没有当前可播放稳定主键时执行歌曲搜索。
- `ImportTrackMatcher`：根据标题、歌手、专辑和时长等信息对候选搜索结果打分。

统一结果模型拆成四类：
- `direct`：来源适配器已经拿到当前可直接导入播放链路所需的稳定主键；网易云来源优先落入此类。
- `matched`：原始曲目没有当前可播放主键，但通过关键词匹配找到了高置信候选；QQ 音乐来源优先走这条路径。
- `ambiguous`：存在多个接近候选，但没有足够置信度。
- `unmatched`：未找到可接受候选。

关键词匹配策略第一版仍采用确定性的轻量规则，而不是复杂模型：
- 查询词优先使用 `title + primaryArtist`，必要时退化为 `title + primaryArtist + albumTitle`。
- 标题规范化只做轻量处理，统一空白、常见标点和大小写，不主动抹平 `Live`、`Remix`、`Acoustic`、`Explicit` 等版本差异。
- 仅搜索歌曲类型结果。
- 对候选结果按以下信号打分：
  - 标题规范化后是否精确相等
  - 主歌手及歌手集合是否有高重叠
  - 专辑名是否接近
  - 已知时长是否足够接近，并把时长差作为强约束而不是普通加分项
- 当第一名候选和第二名候选分差不足、或版本信息/时长冲突时，优先落为 `ambiguous`，而不是勉强归入 `matched`。

这样做的原因：
- 与用户强调的“读取网页信息，然后提取歌名信息”一致，先保住准确曲目提取，再决定后续是否需要匹配。
- 网易云来源不再做多余的搜索回填，能直接复用当前现成歌单详情链路。
- 对 QQ 音乐等非当前 catalog 来源，仍保留关键词匹配能力，不会堵死后续导入为可播放队列的路径。
- 明确建模 `direct / matched / ambiguous / unmatched`，可以避免“看起来导入成功，实际有一部分只是猜出来的”。
- 这种做法与 `lx-music-desktop` 的核心思想一致：来源详情读取和跨 catalog 找歌是两段不同职责，而不是“QQ 曲目直接交给网易云处理”。

考虑过但不采用的方案：
- 只取搜索结果第一条：实现最简单，但误匹配率高，不适合作为跨平台导入能力。
- 为第一版提供逐首手工选歌：体验更完整，但会显著放大 UI 和状态管理复杂度；可在后续迭代基于 `ambiguous` 状态扩展。

### Decision 3: 导入先进入“渐进式预览会话”，用户确认后再替换当前播放队列

导入不是用户粘贴 URL 后立刻覆盖当前播放列表，而是先生成一个 `ImportSession` 供预览。该会话包含：
- 来源信息：平台、原始链接、歌单标题、封面、描述
- 原始曲目线索列表
- 每首曲目的解析/匹配状态和候选结果
- 汇总计数：总数、`direct` 数、`matched` 数、歧义数、失败数

UI 流程采用单独的导入 `Activity` / 全屏流程承载，而不是在个人页中内嵌复杂状态机。第一版的页面状态建议为：
1. 输入或粘贴链接
2. 展示来源解析 / 歌单读取进度
3. 初始化预览会话并先展示歌单头信息
4. 后台继续执行受控匹配，预览页逐步更新统计和逐曲状态
5. 用户点击“导入到当前播放列表”后，才执行队列替换

这样做的原因：
- 外部歌单到现有 catalog 的映射天然有损，不适合静默覆盖现有播放队列。
- 当前播放列表替换已经是仓库中的成熟模式，确认动作应显式发生在“用户看过结果之后”。
- 渐进式预览允许在大歌单场景下先显示部分结果，再继续后台匹配，能兼顾风控和体验。
- 预览会话未来可以自然扩展为“手工修正歧义项”而不推翻整体架构。

考虑过但不采用的方案：
- 一步式导入并直接播放：对误匹配和导入失败不透明，也容易让用户丢失当前队列上下文。
- 为输入页、预览页、完成页分别拆成多个 `Activity`：对当前 MVP 来说状态传递和返回栈成本偏高，不如单一导入 `Activity` 内部维护状态机。

### Decision 4: 所有可导入项直接复用现有 `PlaylistItem` / `playlist-management` 契约

用户确认导入后，把所有可导入条目，也就是 `direct + matched` 的条目映射为现有 `PlaylistItem` 并替换当前队列，不引入新的播放协议。映射规则：
- `id`: 采用稳定前缀，例如 `import:<source>:<playlistId>:<index>:<resolvedSongId>`
- `songId`: 使用匹配得到的当前 catalog songId
- `title` / `displayName` / `artistText` / `albumTitle` / `coverUrl` / `durationMs`: 来自匹配后的当前歌曲结果
- `itemType`: `ONLINE`
- `contextType`: `imported_playlist`
- `contextId`: `import:<source>:<sourcePlaylistId 或 sourceUrlHash>`
- `contextTitle`: 导入歌单标题

替换队列后复用现有行为：
- `PlaylistSessionCoordinator.replaceAll(...)`
- `PlayerRuntime` 队列同步和激活项处理
- `playlist-persistence` 的当前队列持久化与恢复

选择这个方案的原因：
- 当前仓库已经有稳定的“整队列替换 -> 指定激活项 -> 进入播放”的能力。
- 不需要为这次 change 新建独立播放服务协议或额外跨进程序列化模型。
- 即使未来引入真正的“已导入歌单详情页”，底层播放契约也不需要重写。

考虑过但不采用的方案：
- 新建独立的 imported-playlist 数据模型并强制所有播放入口感知它：对当前 change 来说范围过大，而且会和现有 `playlist-management` 职责重叠。

### Decision 5: 第一版导入能力以“在线登录态已满足”为前提

虽然外部网页链接本身通常是公开可访问的，但当前仓库中的搜索和在线播放都依赖登录态，导入结果若不能转成可播放队列，用户价值会明显下降。因此第一版采用以下策略：
- 用户进入导入流程时可以先粘贴链接。
- 在真正发起“解析并生成可导入歌单”前，检查当前在线登录态。
- 若未登录，直接走现有登录引导，而不是生成一个最终无法播放的半成品结果。

选择这个方案的原因：
- 保持与现有 `authenticated-online-playback-access` 规格一致。
- 避免出现“导入成功但不能搜索、不能播放”的分裂体验。
- 当前 `SearchRepository` 已经是 `requiresAuth = true`，这条约束并不是新加的。

考虑过但不采用的方案：
- 允许未登录解析并展示预览，但在导入或播放时再失败：实现上可行，但状态会更复杂，且第一版收益有限。

### Decision 6: 首个入口放在个人页快捷入口，后续设置页复用同一导入流程

由于仓库当前没有本地永久歌单库，本 change 的 UI 范围控制为：
- 提供一个可稳定到达的导入入口
- 提供 URL 输入与导入预览页面
- 允许用户把导入结果转成当前播放队列

第一版入口放在个人页快捷入口，与“喜欢 / 最近 / 本地”同层级，文案为“导入”；后续若新增设置页，则设置页只复用同一导入 `Activity`，不再新做一套导入流程或独立状态机。设计上只要求存在一个清晰入口，不把本 change 扩展成新的歌单管理系统。

选择这个方案的原因：
- 这能交付真正可用的“导入歌单”能力，同时不引入尚未存在的本地歌单产品形态。
- 个人页快捷入口比挂在“自建歌单”区块头部更直接，用户更容易理解这是一个工具入口而不是歌单管理动作。
- 后续如果产品要做“保存到我的导入歌单”，可以在 `ImportSession` 和解析结果模型之上继续扩展，而不是推翻本次实现。

### Decision 7: 匹配流程默认做查询去重、受控并发与渐进式更新

QQ 详情读取只负责拿歌单和原始曲目信息，后续匹配流程必须显式受控，避免大歌单触发搜索接口风控。第一版建议：
- 对相同的规范化查询做去重，同一导入会话内不重复搜索。
- 对搜索结果做短期复用，至少保证会话内缓存，后续可扩展到短 TTL 的内存缓存。
- 控制匹配并发在低位，例如 2 到 3 个 worker。
- 允许预览页在所有匹配未完成时先展示部分结果和当前进度。
- 当出现频率异常、连续搜索失败或来源接口受限时，暂停后续匹配并保留已完成结果，而不是清空整个会话。

选择这个方案的原因：
- 风控风险主要来自“QQ 一首歌对应一次搜索请求”的放大效应，而不是单次歌单详情读取。
- 渐进式更新允许用户更早看到结果，也不需要为了等待完整结果一次性压出大量请求。
- 去重、缓存和低并发能够显著降低重复导入、页面重建和大歌单场景下的请求总量与瞬时峰值。

## Risks / Trade-offs

- [跨平台曲目命名不一致导致误匹配] → 通过显式 `direct / matched / ambiguous / unmatched` 分类、展示原始来源信息和失败计数降低误导，不把低置信候选自动入队。
- [大歌单导入时搜索请求数过多] → 采用查询去重、会话内结果缓存、受控并发和渐进式预览；必要时后续把批量匹配下沉到服务端，但不影响当前接口分层。
- [外部平台分享链接或重定向规则变化] → 通过来源适配器集中处理链接归一化；网易云尽量只依赖 ID 提取和既有接口，QQ 音乐优先依赖其公开 JSON 接口而不是网页 DOM。
- [用户误以为“导入歌单”会永久保存到应用歌单库] → 在第一版 UI 文案里明确这是“导入到当前播放列表”，不是“创建长期本地歌单”。
- [登录前置会让入口看起来多一步] → 在输入页或提交前明确说明“导入后需要在线匹配和播放，因此需要登录”，减少突兀跳转。
- [QQ 音乐来源理论上可能无法完整映射到当前 catalog] → 允许部分成功；导入完成页必须能解释哪些歌曲被跳过以及为什么。
- [搜索接口触发限频后导致整单失败] → 对匹配流程做暂停和错误分类，保留已完成结果并允许用户稍后继续，而不是把当前会话整体作废。

## Migration Plan

- 先实现客户端来源适配层，再接入导入 UI。
- 客户端发布时，若某个来源解析失败或来源接口不可用，入口仍可保留，但要明确展示“当前链接暂不支持”而不是失败崩溃。
- 本次不涉及数据库 schema 迁移，也不修改现有播放服务跨进程协议。
- 回滚时可以直接隐藏导入入口或让解析接口返回关闭状态；已导入并进入当前队列的歌曲仍按普通当前播放队列处理，不会产生额外清理成本。

## Open Questions

- QQ 音乐来源是直接在客户端请求其歌单详情接口，还是先保留 adapter 边界、后续再视稳定性下沉到服务端？
- 第一版是否需要把“匹配已暂停，请稍后继续”做成显式恢复动作，还是允许用户重新进入导入页后自动复用缓存结果继续匹配？
