## ADDED Requirements

### Requirement: 歌手详情页专辑列表支持分页加载
系统 SHALL 在歌手详情页提供歌手专辑列表区块，并使用“获取歌手专辑”接口按 `limit/offset` 语义分页加载该歌手的专辑内容，而不是只展示一次性的静态专辑摘要。

#### Scenario: 首次进入歌手详情页后展示首批专辑
- **WHEN** 歌手详情页已经成功展示歌手头部信息
- **AND** 专辑列表首批请求成功返回
- **THEN** 系统在歌手详情页展示首批歌手专辑内容
- **AND** 每个专辑条目至少展示封面、标题与可用的辅助说明信息

#### Scenario: 滚动接近专辑列表尾部时继续加载下一页
- **WHEN** 用户滚动接近歌手详情页专辑列表尾部
- **AND** 当前仍存在下一页可加载数据
- **THEN** 系统自动按下一页 `offset` 请求更多专辑内容
- **AND** 已经展示的专辑条目保持可见且不被回退

#### Scenario: 专辑列表续页失败时保留已加载内容并允许重试
- **WHEN** 歌手详情页已经成功展示至少一页专辑内容
- **AND** 后续专辑分页请求失败
- **THEN** 系统继续保留已展示的专辑条目
- **AND** 在专辑列表尾部展示明确的失败提示与重试入口

#### Scenario: 专辑列表没有更多内容时停止继续请求
- **WHEN** 歌手详情页已经加载完该歌手全部可用专辑内容
- **THEN** 系统停止继续触发新的专辑分页请求
- **AND** 专辑列表尾部不再重复展示加载中的状态

## MODIFIED Requirements

### Requirement: 系统提供歌手详情页承载歌手基础信息、简要百科信息与热门 50 首歌曲
系统 SHALL 提供可通过歌手实体标识直接进入的歌手详情页，并使用“获取歌手详情”接口加载该歌手的基础信息与可展示摘要，同时使用需登录的“歌手简要百科信息”接口 `/ugc/artist/get` 补充百科摘要/段落信息，使用“歌手热门 50 首歌曲”接口加载热门歌曲列表，并在同一详情上下文中承载该歌手的专辑列表；系统 SHOULD 额外接入 `/artist/detail/dynamic` 与 `/artist/follow/count` 作为头部增强软依赖，用于补充是否关注、视频数、粉丝数等信息，其中头部增强字段以“可用则展示、缺失则降级”为原则，不得改变基础详情页的可进入性与主体承载。

#### Scenario: 从搜索结果进入歌手详情页
- **WHEN** 用户点击任一歌手搜索结果项
- **THEN** 系统打开对应歌手详情页
- **AND** 详情页使用该结果项携带的歌手实体标识请求“获取歌手详情”接口

#### Scenario: 歌手详情接口成功返回后展示基础信息
- **WHEN** 歌手详情页成功获得可解析的歌手详情数据
- **THEN** 系统展示歌手名称、头像或封面与可用的辅助说明信息
- **AND** 页面不再以纯占位或临时提示替代真实详情承载

#### Scenario: 头部增强接口成功返回后补充关注与统计信息
- **WHEN** 歌手详情页已经成功展示基础头部信息
- **AND** `/artist/detail/dynamic` 或 `/artist/follow/count` 返回可用增强字段
- **THEN** 系统在头部补充展示是否关注、视频数、粉丝数等可用增强信息
- **AND** 这些增强字段以补充方式并入现有头部，而不是替代基础信息

#### Scenario: 粉丝数达到一万及以上时按 w 单位展示
- **WHEN** 歌手详情页获得可用的 `fansCount`
- **AND** `fansCount >= 10000`
- **THEN** 系统展示按 `10000` 换算后的 `w` 单位结果
- **AND** 页面中不再展示原始大整数形式的粉丝数

#### Scenario: 粉丝数低于一万时展示具体数值
- **WHEN** 歌手详情页获得可用的 `fansCount`
- **AND** `fansCount < 10000`
- **THEN** 系统展示具体粉丝数值
- **AND** 页面不使用 `w` 单位近似该数值

#### Scenario: 登录态百科接口成功返回后展示百科摘要
- **WHEN** 歌手详情页在可用登录态下成功获得该歌手的百科信息
- **THEN** 系统在歌手详情页展示百科摘要或分段信息的紧凑承载
- **AND** 用户可以在不离开歌手详情页的情况下继续查看完整内容

#### Scenario: 点击歌手简介后以浮层展示完整内容
- **WHEN** 用户点击歌手详情页中的简介摘要区域
- **AND** 该摘要区域位于顶部头部信息区并最多展示两行预览
- **THEN** 系统以浮层形式展示该歌手简介的完整内容
- **AND** 浮层中的长文本支持纵向滑动查看全文
- **AND** 用户关闭浮层后仍回到原有歌手详情页上下文

#### Scenario: 点击歌手头像后放大展示主视觉
- **WHEN** 用户点击歌手详情页中的歌手头像
- **THEN** 系统以浮层或前景预览形式展示更大的歌手主视觉
- **AND** 用户关闭预览后仍回到原有歌手详情页上下文

#### Scenario: 歌手热门 50 首歌曲接口成功返回后展示歌曲列表
- **WHEN** 歌手详情页成功获得该歌手的热门 50 首歌曲数据
- **THEN** 系统在歌手详情页展示该歌手的热门歌曲列表
- **AND** 列表中的每首歌曲至少展示标题、歌手/专辑辅助信息与紧凑行承载

### Requirement: 歌手详情页提供可恢复的数据状态
系统 SHALL 为歌手详情页提供首屏加载态、失败态与重试入口；当歌手详情接口、登录态百科接口、热门歌曲接口或专辑接口返回的信息不足以填满页面时，系统 MUST 继续保持详情页框架可用，而不是直接退回空白页。

#### Scenario: 首次进入歌手详情页时展示带详情壳层的加载态
- **WHEN** 用户首次打开歌手详情页且歌手详情请求尚未完成
- **THEN** 系统展示可识别的详情页加载态
- **AND** 页面仍保留返回能力与基本详情页结构

#### Scenario: 歌手详情请求失败时允许重试
- **WHEN** 歌手详情页请求失败
- **THEN** 系统在页内展示明确的失败提示与重试入口
- **AND** 用户无需返回搜索页即可重新发起该歌手详情请求

#### Scenario: 热门歌曲请求失败时保留已加载的歌手头部信息
- **WHEN** 歌手详情页已经成功展示歌手头部信息但热门歌曲请求失败
- **THEN** 系统继续保留已展示的歌手头部信息
- **AND** 热门歌曲区域展示明确的失败提示与重试入口

#### Scenario: 登录态百科请求失败或不可用时保留已加载的歌手主体信息
- **WHEN** 歌手详情页已经成功展示歌手头部信息
- **AND** 登录态百科请求失败、无登录态或返回空内容
- **THEN** 系统继续保留已展示的歌手头部信息与其他已成功内容
- **AND** 百科信息区域展示稳定兜底文案、提示态或隐藏为非阻塞缺省状态

#### Scenario: 专辑首批请求失败时保留已加载的歌手主体信息
- **WHEN** 歌手详情页已经成功展示歌手头部信息
- **AND** 专辑列表首批请求失败
- **THEN** 系统继续保留已展示的歌手头部信息与其他已成功内容
- **AND** 专辑区域展示明确的失败提示与重试入口

#### Scenario: 头部增强软依赖失败时保留基础头部
- **WHEN** 歌手详情页已经成功展示基础头部信息
- **AND** `/artist/detail/dynamic` 或 `/artist/follow/count` 请求失败、超时或返回空值
- **THEN** 系统继续保留已展示的基础头部信息
- **AND** 页面不会因为增强字段缺失而回退为页级失败态

### Requirement: 歌手详情页视觉层次偏向云音乐风格
系统 SHALL 让歌手详情页采用更偏云音乐风格的音乐内容详情视觉层次，包括 edge-to-edge 顶部区域、带氛围感的头部背景、全宽正方形且无圆角的头图主视觉、基于歌手主视觉主色提取的渐变氛围层、固定在外层的返回入口，以及与百科、热门歌曲、专辑区块相匹配的紧凑信息排版，而不是通用白底列表页样式；页面 MUST 保持自然纵向滚动，不应为了首屏可见性强行压缩内容；HTML 参考只作为视觉参考而非 1:1 布局或逐像素复刻约束；页面中的背景、卡片、文本、按钮与图标颜色 SHOULD 优先使用 theme color roles（如 `primary`、`secondary`、`surface`、`background` 及对应 `on*` roles），而不是直接写死颜色；顶部 compact header 与 sticky tab MUST 分离职责并共用一致的安全区高度来源，不得通过双重状态栏占位把标题、tab 或内容彼此挤压；百科 tab 或其等价承载区域 SHOULD 在视觉上尽量填满所在 section 以避免明显回弹观感；切换 tab 时，系统 MUST 遵循“未吸顶前不改外层滚动、已吸顶后重置到新 tab 起始内容”的语义，而不是无条件继承上一 tab 的滚动偏移。

#### Scenario: 歌手详情页展示带氛围感的头部区域
- **WHEN** 用户查看歌手详情页
- **THEN** 系统在页面顶部展示带氛围层次的详情头部
- **AND** 返回入口固定在详情页外层且不随正文滚动
- **AND** 头图主视觉以全宽正方形且无圆角区域承载
- **AND** 头部氛围色优先依据歌手主视觉主色生成渐变效果
- **AND** 当主视觉缺失或取色失败时退回稳定的默认渐变
- **AND** 标题、辅助信息、百科、热门歌曲与专辑区块之间保持更接近音乐 App 的紧凑层级

#### Scenario: 滚动时顶部渐变与标题连续过渡
- **WHEN** 用户在歌手详情页上下滚动
- **THEN** 顶部渐变、居中标题淡入与头部大标题/按钮淡出按照连续滚动进度变化
- **AND** 页面不会在吸顶阈值附近出现明显的 0/1 硬切
- **AND** 过渡过程中标题、tab 与内容不会因为额外占位而互相覆盖

#### Scenario: 长详情内容通过滚动承载而不是压缩进首屏
- **WHEN** 歌手详情页同时承载头部、简介、热门歌曲与专辑等多个区块
- **THEN** 系统允许用户通过纵向滚动浏览完整内容
- **AND** 不为了让更多区块首屏可见而强行压缩头像尺寸、简介预览、区块间距或正文承载

#### Scenario: 简介摘要与全文承载分离
- **WHEN** 系统在歌手详情页展示简介摘要
- **THEN** 简介摘要区域最多展示两行预览
- **AND** 完整正文继续通过浮层或等价前景承载展示
- **AND** 全文承载支持纵向滚动查看

#### Scenario: 视觉对齐优先复用主题颜色角色
- **WHEN** 系统渲染歌手详情页的背景、卡片、文本、按钮、占位和图标
- **THEN** 系统优先复用 theme color roles 组织颜色层次
- **AND** 只有头部氛围层等需要主视觉联动的场景才允许基于 theme roles 做动态渐变或透明度变化
- **AND** 页面不以直接写死颜色值作为主要视觉实现手段

#### Scenario: 列表区块使用小间距卡片式排版
- **WHEN** 系统渲染热门歌曲、专辑或等价内容列表 item
- **THEN** 系统使用小间距卡片式承载这些列表项
- **AND** 列表项之间保持紧凑但可读的间距

#### Scenario: 百科承载区域尽量填满所在 section
- **WHEN** 歌手详情页展示百科 tab 或其等价承载区域
- **THEN** 系统让该区域在视觉上尽量填满所在 section
- **AND** 页面避免出现明显的上浮留白或回弹观感

#### Scenario: 固定头部正确避让状态栏安全区
- **WHEN** 系统渲染固定头部、返回入口与顶部内容
- **THEN** 系统正确处理状态栏安全区
- **AND** 头部内容与交互元素不会卡进状态栏

#### Scenario: Sticky tab 与固定头部共同避让状态栏安全区
- **WHEN** 系统渲染 sticky tab、固定头部与顶部内容区
- **THEN** 固定头部负责状态栏渐变、顶部操作与居中标题，sticky tab 负责内容入口
- **AND** 二者不会通过双重状态栏占位产生额外留白或互相覆盖
- **AND** tab 内容、返回入口与其他顶部交互元素不会互相覆盖或卡进状态栏

#### Scenario: Sticky tab 未吸顶前切换 tab 不改变外层滚动位置
- **WHEN** 用户已经把 tab 区域滚动到可见
- **AND** sticky tab 尚未真正吸顶
- **AND** 用户切换到另一个 tab
- **THEN** 系统只切换对应 panel
- **AND** 外层滚动位置保持稳定
- **AND** tab 上方已可见的简介或其他内容不会被强制拉回顶部

#### Scenario: 切换 tab 后从新 tab 顶部开始浏览
- **WHEN** 用户已经在某个歌手详情 tab 中向下滚动浏览内容
- **AND** 用户切换到另一个 tab
- **THEN** 系统将内容位置重置到 sticky tab 下方的新 tab 起始内容
- **AND** 新 tab 不继承上一 tab 的滚动偏移

#### Scenario: HTML 参考不强制约束为 1:1 布局
- **WHEN** 设计或实现需要参考外部 HTML 样式文件
- **THEN** 系统将其视为视觉方向与信息层次参考
- **AND** 不要求在 Compose 中逐像素复刻相同布局
- **AND** 详情页仍以当前可滚动壳层和可读性优先的内容承载为准
